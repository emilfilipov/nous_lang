What the numbers say

Headline. Across 417 real-world functions in six languages, Lullaby is the tersest of the compiled/systems languages - 7% fewer tokens than C, 11% fewer than C++, and slightly ahead of Rust - while its native code runs within ~1.0-1.2x of C (the prime benchmark is now 1.04x C after this benchmark drove inlining + loop register promotion) and 17-27x faster than CPython. The two dynamically-typed languages are terser still: Python by 21%, JavaScript by 9% (no type annotations to spend tokens on). It is not "half the tokens of C" - that isn't borne out. The real, defensible claim is the tersest statically-typed systems language here, at near-C speed.

Tokens

Python is the tersest overall (dynamic typing, no annotations) - 21% fewer tokens - with JavaScript second (9% fewer); Lullaby cannot match the untyped languages on scripting-shaped code. But among the languages Lullaby actually competes with on performance - C, C++, Rust - it is the tersest: 7% ahead of C, 11% ahead of C++, and marginally ahead of Rust. Static typing plus indentation-only blocks buys back most of what the annotations cost.

The advantage is uneven and instructive. Lullaby is at its best on structured, branch-heavy logic: enum + match state machines, record/struct math, numeric business logic, sorting. It is at its worst on text/string-heavy code (parsing, string algorithms, validation), where concrete language gaps inflate its token count (see Language gaps): no sized-array allocation and limited string ergonomics. The % operator was the top gap this benchmark surfaced and has now shipped across all five backends - the corpus dropped its hand-written rem(a,b) helpers for native a % b. The measured token delta is small only because the strict metric counts a function solely when it exists in Lullaby and another language, so the Lullaby-only helper it replaced was never in the total; the real ergonomic win (no helper to write at all) sits just outside the comparison.

Performance

Lullaby's native backend clusters with C/C++/Rust and leaves CPython far behind. This is optimizer maturity, not language design - and this benchmark drove a big step of it: leaf-helper inlining plus register promotion for for-loops moved the prime-counting workload from 1.57x to 1.04x C (native now at parity with C and ahead of C++), on top of the earlier fib 1.72x->1.2x and loop 5.6x->2.1x wins. The one remaining lever is SIMD auto-vectorization for reduction/map loops. Interpreted tiers (ast/ir/bytecode) are ~200-1200x C and are dev-time tools, not the performance story.

Honest caveats

  • The corpus is deterministic integer/string logic - it deliberately excludes GC-pressure, heavy generics, and async, where the picture would differ.
  • Every implementation is written idiomatic-but-minimal in each language (not code-golfed), and every Lullaby file passes lullaby check. Tokens count definitions only.
  • Integer-division semantics are aligned across languages: JavaScript uses Math.trunc, and Python's ///% coincide with C-style truncation only because inputs are non-negative.
  • JavaScript runs on Node (no compiled/native tier here), so it appears in the token comparison but not the performance one.
