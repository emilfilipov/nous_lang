What the numbers say

Headline. Across the corpus in six languages, Lullaby is now the tersest of every language except Python - ~22% fewer tokens than C, ~24% fewer than C++, ~18% fewer than Rust, and ~8% fewer than JavaScript - while its native code runs at parity with C on the prime benchmark (1.01x, at/below C++) and 17-27x faster than CPython (deep recursion is the one workload still above C, at 1.26x). Python is the only language still ahead on tokens, and only by +6.3% (down from +16.9%), after return-type inference and the four inline-syntax features shipped. It is not "half the tokens of C" - that isn't borne out. The real, defensible claim: the tersest statically-typed language measured here, at near-C speed, second only to Python and closing.

Tokens

Python is the only language still ahead (dynamic typing, no annotations) - ~6.3% fewer tokens than Lullaby, down from +16.9%. Lullaby now beats every other language: ~22% ahead of C, ~24% ahead of C++, ~18% ahead of Rust, ~8% ahead of JavaScript. Static typing plus indentation-only blocks, terse inline syntax (ternary, in, slices), and return-type inference buy back almost everything the annotations cost.

The advantage is uneven and instructive. Lullaby is at its best on structured, branch-heavy logic: enum + match state machines, record/struct math, numeric business logic, sorting. The text/string weak spot this benchmark first surfaced has been closed (inline conditional, string + char concat, the in operator, string slicing, array_fill for DP buffers; text_processing 703 → 606, string_algos 1996 → 1628). The last gap to Python is a single structural cost: mandatory parameter type annotations (return types now infer). The corpus is mostly array/numeric code, so string interpolation and dropping length parameters were measured and do not help the counted metric (see Open gaps); only parameter-type inference does, and that is a deliberate design decision.

Performance

Lullaby's native backend clusters with C/C++/Rust and leaves CPython far behind. This is optimizer maturity, not language design - and this benchmark drove it step by step: leaf-helper inlining + for-loop register promotion moved prime-counting 1.57x → 1.04x C, and switching the remainder to the native % operator took it to 1.01x C - at parity with C and C++ (only Rust's 0.92x is ahead). The only compute workload still trailing C is deep recursion (fib 1.26x), a per-call-overhead gap. Two levers guessed earlier were checked against the emitted machine code and ruled out: overflow-check elision (native emits plain add/imul - there is no check to remove) and leaf-frame omission (the prologue already self-omits when the frame is empty). The remaining real levers are recursive-call codegen and broader SIMD. Interpreted tiers (ast/ir/bytecode) are ~200-1200x C and are dev-time tools, not the performance story.

Honest caveats

  • The corpus is deterministic integer/string logic - it deliberately excludes GC-pressure, heavy generics, and async, where the picture would differ.
  • Every implementation is written idiomatic-but-minimal in each language (not code-golfed), and every Lullaby file passes lullaby check. Tokens count definitions only.
  • Integer-division semantics are aligned across languages: JavaScript uses Math.trunc, and Python's ///% coincide with C-style truncation only because inputs are non-negative.
  • JavaScript runs on Node (no compiled/native tier here), so it appears in the token comparison but not the performance one.

Open gaps -> the path to #1

what's left to be tersest AND fastest

Goal: fewer tokens than every language (beat Python) and native speed at or below C. Lullaby now beats every language except Python on tokens and sits at +6.3% over Python (down from +16.9%). Only two things remain: on tokens, a single design lever - parameter-type inference; on speed, the prime workload is now at parity with C (1.01x, at/below C++) after switching to the native % operator, leaving only deep recursion (fib 1.26x C). This tab lists only what is still open - completed gaps are removed once they ship.

Investigated and ruled out (measured, not worth building): string interpolation - the counting metric strips the main driver, and there are zero to_string calls in the counted functions, so it saves ~0 counted tokens. Implicit array length - swapping a short length parameter n for len(a) is net-neutral to slightly worse (len(a) is more tokens than n) unless the loop is also restructured, which the corpus already does where possible.

Already shipped by this benchmark and no longer listed: return-type inference (dropped 449 -> T clauses, 21,535 → 20,321), the % operator, function inlining, array_fill sized arrays, string ergonomics, for-loop register promotion, float unary minus, SIMD phase 1, and the four second-wave token features (inline conditional, string+char, in, slicing).
