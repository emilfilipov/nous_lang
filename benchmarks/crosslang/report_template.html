What the numbers say

Headline. Across the corpus in six languages, Lullaby is now the tersest of every language except Python - ~23% fewer tokens than C, ~26% fewer than C++, ~20% fewer than Rust, and ~10% fewer than JavaScript - while its native code runs at parity with C on the prime benchmark (0.99–1.01x, at/below C++) and 17-27x faster than CPython (deep recursion is the one workload still above C, at 1.26x). Python is the only language still ahead on tokens, and only by +4.7% (down from +16.9%), after return-type inference, the inline-syntax features, and an idiomatic-stdlib pass (words/count, inline conditionals). It is not "half the tokens of C" - that isn't borne out. The real, defensible claim: the tersest statically-typed language measured here, at near-C speed, second only to Python and closing - while keeping every type explicit.

Tokens

Python is the only language still ahead (dynamic typing, no annotations) - ~4.4% fewer tokens than Lullaby, down from +16.9%. Lullaby now beats every other language: ~23% ahead of C, ~26% ahead of C++, ~20% ahead of Rust, ~10% ahead of JavaScript. Static typing plus indentation-only blocks, terse inline syntax (ternary, in, slices), return-type inference, and an idiomatic string stdlib (words/count) buy back almost everything the annotations cost. The remaining gap is concentrated: Lullaby already beats Python outright on structured/numeric categories (statistics, state machines, geometry, bitwise, arrays) and trails only on the most string/parse-heavy ones.

The advantage is uneven and instructive. Lullaby is at its best on structured, branch-heavy logic: enum + match state machines, record/struct math, numeric business logic, sorting. The text/string weak spot this benchmark first surfaced has been closed (inline conditional, string + char concat, the in operator, string slicing, array_fill for DP buffers; text_processing 703 → 606, string_algos 1996 → 1628). The last gap to Python is a single structural cost: mandatory parameter type annotations (return types now infer). The corpus is mostly array/numeric code, so string interpolation and dropping length parameters were measured and do not help the counted metric (see Open gaps); only parameter-type inference does, and that is a deliberate design decision.

Performance

Lullaby's native backend clusters with C/C++/Rust and leaves CPython far behind. This is optimizer maturity, not language design - and this benchmark drove it step by step: leaf-helper inlining + for-loop register promotion moved prime-counting 1.57x → 1.04x C, and switching the remainder to the native % operator took it to 1.01x C - at parity with C and C++ (only Rust's 0.92x is ahead). The only compute workload still trailing C is deep recursion (fib 1.26x), a per-call-overhead gap. Two levers guessed earlier were checked against the emitted machine code and ruled out: overflow-check elision (native emits plain add/imul - there is no check to remove) and leaf-frame omission (the prologue already self-omits when the frame is empty). On the SIMD front, auto-vectorization now spans three bit-for-bit-identical phases: i64 sum reduction (3.3x), element-wise map c[i]=a[i]±b[i] (3.36x), and bitwise reductions/maps ^ | & (2.89x) via pand/por/pxor — and native scalar i64 bitwise operators shipped alongside (previously interpreter-only). The rest of the requested SIMD family is blocked by the x86-64 baseline ISA, not effort: dot/product need a 64-bit packed multiply SSE2 lacks (emulation measures slower than scalar imul), min/max need SSE4.2’s pcmpgtq, and f64 accumulation would break bit-exact parity (float add isn’t associative) — each waits on runtime CPU-feature dispatch or a fast-math opt-in (see Open gaps). The remaining in-reach lever is recursive-call codegen (fib 1.26x). Interpreted tiers (ast/ir/bytecode) are ~200-1200x C and are dev-time tools, not the performance story - though this pass still doubled their for-loop throughput (all three tiers, ~2x) by binding the loop variable once and updating it in place instead of allocating a scope and cloning its name every iteration; the earlier attempt to move the while-loop benchmark this way measured as noise (it never hit that path), a reminder to profile before claiming.

The interpreter perf picture is no longer three micro-benchmarks: a whole-corpus compute harness now wraps every category's functions in a hot loop and times all three tiers on real work across the full 434-function / 26-category set. It confirms the diagnosis at scale — on numeric categories the AST, IR, and bytecode tiers cluster within ~1–2%, and bytecode is only ~2% faster than the tree-walker overall, so the bytecode VM is not yet earning its tier: the shared Value-model and string-keyed Env cost dominates, drowning its structural advantage. On string-heavy categories (statistics, text_processing, validation) the IR and bytecode tiers actually run slower than the AST walker. The clear next lever is slot-based variable resolution (index instead of the per-access name scan), which hits every variable access on every tier; it is a large, correctness-critical change to the shared representation, scoped rather than rushed, with this harness now in place to measure it honestly.

Honest caveats

  • The corpus is deterministic integer/string logic - it deliberately excludes GC-pressure, heavy generics, and async, where the picture would differ.
  • Every implementation is written idiomatic-but-minimal in each language (not code-golfed), and every Lullaby file passes lullaby check. Tokens count definitions only.
  • Integer-division semantics are aligned across languages: JavaScript uses Math.trunc, and Python's ///% coincide with C-style truncation only because inputs are non-negative.
  • JavaScript runs on Node (no compiled/native tier here), so it appears in the token comparison but not the performance one.

Open gaps -> the path to #1

what's left to be tersest AND fastest

Goal: fewer tokens than every language (beat Python) and native speed at or below C. Lullaby now beats every language except Python on tokens and sits at +4.7% over Python (down from +16.9%). On tokens, the remaining deficit is a deliberate choice: parameter type annotations are kept explicit - a systems language should state its types - so Lullaby stays the tersest statically-typed language rather than chasing a dynamically-typed one. On speed, the prime workload is at parity with C (0.99–1.01x, at/below C++), leaving only deep recursion (fib 1.26x C). This tab lists only what is still open - completed gaps are removed once they ship.

Measured and set aside (do not move the counted-token metric): string interpolation was built anyway as a real ergonomic feature ("n=${x}"), but it saves ~0 counted tokens - the metric strips the main driver and there are zero to_string calls in the counted functions. Implicit array length - swapping a short length parameter n for len(a) is net-neutral to worse (len(a) is more tokens than n) unless the loop is also restructured, which the corpus already does where possible.

Already shipped by this benchmark and no longer listed: return-type inference (dropped 449 -> T clauses), an idiomatic string stdlib + corpus pass (words/count builtins, inline conditionals, min over hand-rolled helpers), grouped same-type parameter annotations (fn dist x, y, z f64 — every type still explicit), and completing the graph_algos category for all six languages (shrunk with array_fill). Across the now-complete 26-category / 434-function corpus Lullaby sits at +4.7% over Python (from +16.9%), beating every other language. Also shipped: string interpolation "a=${expr}", native % in the prime workload (perf 1.04x → 1.01x C), function inlining, array_fill sized arrays, string ergonomics, for-loop register promotion, float unary minus, SIMD phases 1–3 (i64 sum reduction + element-wise map + bitwise reductions/maps, 2.89–3.36x), native scalar i64 bitwise operators & | ^ << >>, a ~2x interpreter for-loop speedup (all three tiers), and the four second-wave token features (inline conditional, string+char, in, slicing).
