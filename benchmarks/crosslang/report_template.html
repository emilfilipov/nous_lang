What the numbers say

Headline. Across the corpus in six languages, Lullaby is the tersest of the compiled/systems languages - ~17% fewer tokens than C, ~20% fewer than C++, ~13% fewer than Rust - and after four token-ergonomics features shipped it now also edges out JavaScript (~3% fewer). Its native code runs within ~1.0-1.2x of C (the prime benchmark is 1.04x C after this benchmark drove inlining + loop register promotion) and 17-27x faster than CPython. Python remains the tersest (no type annotations), but the gap has closed from +16.9% to +12.6%. It is not "half the tokens of C" - that isn't borne out. The real, defensible claim: the tersest statically-typed systems language here, at near-C speed, and now under JavaScript too.

Tokens

Python is still the tersest overall (dynamic typing, no annotations) - ~12.6% fewer tokens than Lullaby, down from +16.9% before the second-wave features. Lullaby now beats JavaScript (~3% fewer) and leads the languages it actually competes with on performance - ~17% ahead of C, ~20% ahead of C++, ~13% ahead of Rust. Static typing plus indentation-only blocks, plus terse inline syntax (ternary, in, slices), buys back most of what the annotations cost.

The advantage is uneven and instructive. Lullaby is at its best on structured, branch-heavy logic: enum + match state machines, record/struct math, numeric business logic, sorting. Its old weak spot was text/string-heavy code - but the string-ergonomics gaps this benchmark surfaced have since shipped (the inline conditional, string + char concat, the in operator, string slicing, plus array_fill(n,0) for DP buffers), and those categories tightened sharply (text_processing 703 → 606, string_algos 1996 → 1628, validation 1035 → 938). The remaining gap to Python is now structural, not ergonomic: the corpus is mostly array/numeric code, so what still separates the two is mandatory type annotations on every parameter/return and explicit array-length parameters carried for cross-language parity - addressable by optional return-type inference and dropping redundant length params.

Performance

Lullaby's native backend clusters with C/C++/Rust and leaves CPython far behind. This is optimizer maturity, not language design - and this benchmark drove a big step of it: leaf-helper inlining plus register promotion for for-loops moved the prime-counting workload from 1.57x to 1.04x C (native now at parity with C and ahead of C++), on top of the earlier fib 1.72x->1.2x and loop 5.6x->2.1x wins. The one remaining lever is SIMD auto-vectorization for reduction/map loops. Interpreted tiers (ast/ir/bytecode) are ~200-1200x C and are dev-time tools, not the performance story.

Honest caveats

  • The corpus is deterministic integer/string logic - it deliberately excludes GC-pressure, heavy generics, and async, where the picture would differ.
  • Every implementation is written idiomatic-but-minimal in each language (not code-golfed), and every Lullaby file passes lullaby check. Tokens count definitions only.
  • Integer-division semantics are aligned across languages: JavaScript uses Math.trunc, and Python's ///% coincide with C-style truncation only because inputs are non-negative.
  • JavaScript runs on Node (no compiled/native tier here), so it appears in the token comparison but not the performance one.

Open gaps -> the path to #1

what's left to be tersest AND fastest

Goal: fewer tokens than every language (beat Python) and native speed at or below C. This tab lists only what is still open, each with its measured impact - completed gaps are removed once they ship. The remaining token gap to Python is 2,415 tokens (12.6%) and is now structural (type annotations + redundant length params), not string-ergonomics; the remaining speed gap is 1.04x C on primes, 1.20x on fib, gated by checked-arithmetic and frame overhead.

Already shipped by this benchmark and no longer listed: the % operator, function inlining, array_fill sized arrays, string ergonomics (+=/indexing/ordering), for-loop register promotion, float unary minus, SIMD phase 1, and the four second-wave token features (inline conditional, string+char, in, slicing) that cut the corpus 22,356 → 21,535.
