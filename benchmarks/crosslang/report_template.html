What the numbers say

Headline. Across the corpus in six languages, Lullaby is now the tersest of every language except Python - ~23% fewer tokens than C, ~26% fewer than C++, ~20% fewer than Rust, and ~10% fewer than JavaScript - while its native code runs at or below C on every measured workload — the prime benchmark at parity (0.99–1.01x, at/below C++), deep recursion now matching C (fib ~1.00x), the counting-sum loop beating C (0.52x, ILP-unrolled) — and 17-27x faster than CPython. Python is the only language still ahead on tokens, and only by +4.7% (down from +16.9%), after return-type inference, the inline-syntax features, and an idiomatic-stdlib pass (words/count, inline conditionals). It is not "half the tokens of C" - that isn't borne out. The real, defensible claim: the tersest statically-typed language measured here, at near-C speed, second only to Python and closing - while keeping every type explicit.

Tokens

Python is the only language still ahead (dynamic typing, no annotations) - ~4.4% fewer tokens than Lullaby, down from +16.9%. Lullaby now beats every other language: ~23% ahead of C, ~26% ahead of C++, ~20% ahead of Rust, ~10% ahead of JavaScript. Static typing plus indentation-only blocks, terse inline syntax (ternary, in, slices), return-type inference, and an idiomatic string stdlib (words/count) buy back almost everything the annotations cost. The remaining gap is concentrated: Lullaby already beats Python outright on structured/numeric categories (statistics, state machines, geometry, bitwise, arrays) and trails only on the most string/parse-heavy ones.

The advantage is uneven and instructive. Lullaby is at its best on structured, branch-heavy logic: enum + match state machines, record/struct math, numeric business logic, sorting. The text/string weak spot this benchmark first surfaced has been closed (inline conditional, string + char concat, the in operator, string slicing, array_fill for DP buffers; text_processing 703 → 606, string_algos 1996 → 1628). The last gap to Python is a single structural cost: mandatory parameter type annotations (return types now infer). The corpus is mostly array/numeric code, so string interpolation and dropping length parameters were measured and do not help the counted metric (see Open gaps); only parameter-type inference does, and that is a deliberate design decision.

Performance

Lullaby's native backend clusters with C/C++/Rust and leaves CPython far behind. This is optimizer maturity, not language design - and this benchmark drove it step by step: leaf-helper inlining + for-loop register promotion moved prime-counting 1.57x → 1.04x C, and switching the remainder to the native % operator took it to 1.01x C - at parity with C and C++ (only Rust's 0.92x is ahead). Deep recursion (fib) is now at parity with C too (~1.00x, from 1.26x): folding promoted-register operands into the per-call cmp (no mov rax reload) took it to 1.09x, then forming the recursive arguments fib(n-1)/fib(n-2) with a single lea rcx,[rbx-k] (exactly as C does, instead of mov;sub;mov) closed the rest. Two levers guessed earlier were checked against the emitted machine code and ruled out: overflow-check elision (native emits plain add/imul - there is no check to remove) and leaf-frame omission (the prologue already self-omits when the frame is empty). On the SIMD front, auto-vectorization now spans three bit-for-bit-identical phases: i64 sum reduction (3.3x), element-wise map c[i]=a[i]±b[i] (3.36x), and bitwise reductions/maps ^ | & (2.89x) via pand/por/pxor — and native scalar i64 bitwise operators shipped alongside (previously interpreter-only). The rest of the requested SIMD family is blocked by the x86-64 baseline ISA, not effort: dot/product need a 64-bit packed multiply SSE2 lacks (emulation measures slower than scalar imul), min/max need SSE4.2’s pcmpgtq, and f64 accumulation would break bit-exact parity (float add isn’t associative) — each waits on runtime CPU-feature dispatch or a fast-math opt-in (see Open gaps). With fib now at C parity, native has no workload left above C, and the sum-loop beats C: recognizing while i < N: acc = acc + i; i = i + 1 and folding four iterations per step into one dependent add (lea rax,[i*4+6]; add acc,rax, exact under wrapping) broke the serial acc chain and took native from 1.96x to 0.52x C (native ~2x faster than C). Interpreted tiers (ast/ir/bytecode) are ~200-1200x C and are dev-time tools, not the performance story - though this pass still doubled their for-loop throughput (all three tiers, ~2x) by binding the loop variable once and updating it in place instead of allocating a scope and cloning its name every iteration; the earlier attempt to move the while-loop benchmark this way measured as noise (it never hit that path), a reminder to profile before claiming.

The interpreter perf picture is no longer three micro-benchmarks: a whole-corpus compute harness now wraps every category's functions in a hot loop and times all three tiers on real work across the full 434-function / 26-category set. It immediately surfaced a real bug: a[i] and s.field cloned the whole array/struct on every access, and a[i] = v cloned-mutated-and-wrote-back — O(len) per operation, O(len²) in a loop. Fixing them to borrow the container and touch only the element (a read/write "borrow, don't clone" pass across both interpreter crates) sped up the array- and struct-heavy categories on all three tiers: graph_algos ~2x, sorting ~1.5x, data_records ~1.47x, matrix ~1.38x, hashing ~1.17x; the in-place write fix takes a large-array write loop from a >6-minute timeout to ~0.5s. After that the tiers clustered within ~4%, with bytecode slower than the tree-walker — so the next lever, slot-based variable resolution, shipped: a resolver runs once at interpretation entry and rewrites each name-resolvable local read into a (depth, slot)-indexed access, mirroring the evaluator's scope push/pop exactly (including the two-scope for). It is correct-by-construction — the indexed lookup re-validates the binding name at that position and falls back to the scan on any mismatch, so a wrong slot can only be slower, never wrong — and because the bytecode tier executes by converting to IR and running the same evaluator, both tiers benefit with no format change. Result across the 434-function corpus: IR 106%→102% of AST, bytecode 105%→101%, and on numeric loops the resolved IR/bytecode tiers now run faster than the AST tree-walker (152 vs 162 ns/iter) — the "bytecode earns its tier" goal, met. (The AST tier keeps the name scan: it works on the shared parser AST, which can't carry a per-node slot without touching every crate.) A tighter Value cell then shrank the shared per-op footprint from 32 to 24 bytes: the four largest variants were boxed to a 16-byte-or-smaller payload (String→Box<str>, Array→Box<[Value]>, Func→Box<str>, Closure→Box<Closure>) — the same single indirection as before, just without the unused growth-capacity word, so no access penalty. Every scalar clone/move copies 24 bytes instead of 32; the AST tier (which clones a Value on every variable read, being unresolved) dropped from 1.30x to 1.20x C on the sum loop (~7%), while the already-borrowing IR/bytecode tiers were neutral. Finally the bytecode tier earned its tier for real: it used to convert to IR and run the same recursive tree-walker (so it only ever tied the IR tier), and now compiles each eligible function once into a flat op stream executed by a single loop { match } dispatch loop with slot-indexed locals — no recursion, no scope stack, no name scan. Every operation reuses the exact same Value helpers, so only control-flow lowering is new and results stay identical tier-for-tier (any match/try/closure/indexed-assign falls back to the tree-walker). The bytecode VM is now 2.06x faster than the tree-walkers on the sum loop (72 vs 149 ns/iter) and 77% of AST across the whole 434-function corpus — distinctly the fastest interpreter, not just marginally. The harness also grew a native column: it folds each i64-scalar category's function results directly (no strings) and compiles them to a native .exe, showing native runs ~98x faster than the AST interpreter across the 12 purely-i64-scalar categories; heap/string/array categories correctly defer (a real language boundary, shown as n/a).

Honest caveats

  • The corpus is deterministic integer/string logic - it deliberately excludes GC-pressure, heavy generics, and async, where the picture would differ.
  • Every implementation is written idiomatic-but-minimal in each language (not code-golfed), and every Lullaby file passes lullaby check. Tokens count definitions only.
  • Integer-division semantics are aligned across languages: JavaScript uses Math.trunc, and Python's ///% coincide with C-style truncation only because inputs are non-negative.
  • JavaScript runs on Node (no compiled/native tier here), so it appears in the token comparison but not the performance one.

Open gaps -> the path to #1

what's left to be tersest AND fastest

Goal: fewer tokens than every language (beat Python) and native speed at or below C. Lullaby now beats every language except Python on tokens and sits at +4.7% over Python (down from +16.9%). On tokens, the remaining deficit is a deliberate choice: parameter type annotations are kept explicit - a systems language should state its types - so Lullaby stays the tersest statically-typed language rather than chasing a dynamically-typed one. On speed, every measured native workload is now at or below C: primes at parity (0.99–1.01x, at/below C++), deep recursion matching C (fib ~1.00x), and the counting-sum loop beating C (0.52x, ILP-unrolled). This tab lists only what is still open - completed gaps are removed once they ship.

Measured and set aside (do not move the counted-token metric): string interpolation was built anyway as a real ergonomic feature ("n=${x}"), but it saves ~0 counted tokens - the metric strips the main driver and there are zero to_string calls in the counted functions. Implicit array length - swapping a short length parameter n for len(a) is net-neutral to worse (len(a) is more tokens than n) unless the loop is also restructured, which the corpus already does where possible.

Already shipped by this benchmark and no longer listed: return-type inference (dropped 449 -> T clauses), an idiomatic string stdlib + corpus pass (words/count builtins, inline conditionals, min over hand-rolled helpers), grouped same-type parameter annotations (fn dist x, y, z f64 — every type still explicit), and completing the graph_algos category for all six languages (shrunk with array_fill). Across the now-complete 26-category / 434-function corpus Lullaby sits at +4.7% over Python (from +16.9%), beating every other language. Also shipped: string interpolation "a=${expr}", native % in the prime workload (perf 1.04x → 1.01x C), function inlining, array_fill sized arrays, string ergonomics, for-loop register promotion, float unary minus, SIMD phases 1–3 (i64 sum reduction + element-wise map + bitwise reductions/maps, 2.89–3.36x), native scalar i64 bitwise operators & | ^ << >>, a ~2x interpreter for-loop speedup (all three tiers), and the four second-wave token features (inline conditional, string+char, in, slicing).
