Lullaby

Offline documentation for the current state of Lullaby: a compiled systems language in active development toward 1.0, with concise indentation-only syntax, strong typing, and an executable Rust-based frontend, runtime, and backends.

Overview

Lullaby is implemented in Rust and in active development toward 1.0. It runs today through a clear, testable compiler frontend plus AST, typed IR, and instruction-bytecode execution paths, a scalar-subset WebAssembly backend that also compiles the fixed-width integer operations, f32/f64 floating-point arithmetic, comparison, and to_f32/to_f64 conversions within those functions, and enum values and match for enums with scalar payloads (option/result/user enums), and a native x86-64 backend that compiles i64-scalar functions including their fixed-width integer operations, their f64/f32 floating-point arithmetic, comparison, and to_f32/to_f64 conversions, and enum values with scalar payloads (option/result/user enums) with match over them.

The currently implemented language surface is recorded in documents/alpha1_language_surface.md. Anything documented on this page is accepted by the compiler; features only in the broader Markdown design docs are planned material unless they appear here.

First-class functions

A function type is spelled fn(T1, T2) -> R. A bare top-level function name is a value of that type, so a function can be stored in a let, passed to another function, returned, and called through a local. Inline capturing closures are also function values of the same type. Calling a non-function local, or passing a function value with the wrong signature, reports L0390.

fn inc x i64 -> i64
    x + 1

fn dbl x i64 -> i64
    x * 2

fn apply f fn(i64) -> i64 v i64 -> i64
    f(v)

fn picker -> fn(i64) -> i64
    dbl

fn main -> i64
    let g fn(i64) -> i64 = inc
    let h fn(i64) -> i64 = picker()
    apply(inc, 10) + g(5) + h(20) + apply(dbl, 3)

Closures

An inline closure literal fn PARAMS -> EXPR is an anonymous function value that captures the enclosing scope's locals. It reuses the fn keyword and appears only in expression position (the right side of a let/assignment, a call argument, or a returned value). PARAMS is zero or more name type pairs (the top-level fn shape — explicit types, no parentheses), and the body is a single expression after ->. Its value type is fn(param types) -> typeof(EXPR), so a closure interoperates with everything that accepts a function value: a function-typed let, a function-typed parameter (like apply), parallel_map, and returning a function.

Capture is by value, taken when the literal is evaluated: each captured local is cloned into the closure (an eager snapshot). Value-semantic collections snapshot independently; reference-semantic handles (Chan/Mutex/rc) share their underlying resource. Mutating an enclosing local after the closure is built does not change the captured copy, and captured names are read-only in the body. Closures run on the AST, IR, and bytecode interpreters at parity; the native and WASM backends skip a function that constructs or calls a closure and run it on the interpreters instead.

fn apply f fn(i64) -> i64 v i64 -> i64
    f(v)

fn main -> i64
    let n i64 = 10
    let add_n fn(i64) -> i64 = fn x i64 -> x + n
    apply(add_n, 5) + add_n(2)

Generic functions

A function can declare type parameters in angle brackets after its name and use each name as an ordinary type in its parameters and return type, including inside built-in generics such as list<T> and option<T>. Call sites infer the type parameters from the argument types and substitute them into the return type — identity(41) yields i64, identity("hi") yields string, and wrap(1) yields option<i64>. Generics are erased at runtime, so they run identically on every backend. A bare T value supports only universal operations (binding, passing, returning, equality between two same-T values, and use as a payload to a built-in generic like some(x)); arithmetic and ordering on a bare T wait on trait bounds.

fn identity<T> x T -> T
    x

fn wrap<T> x T -> option<T>
    some(x)

fn choose<T> pick bool a T b T -> T
    if pick
        return a
    b

fn unwrap<T> o option<T> fallback T -> T
    match o
        some(v) -> v
        none -> fallback

fn first<T> xs list<T> -> option<T>
    if len(xs) == 0
        return none
    some(get(xs, 0))

fn main -> i64
    let n i64 = identity(41)
    let s string = identity("hi")
    let picked i64 = choose(true, 10, 20)
    let maybe option<i64> = wrap(1)
    let nums list<i64> = list_new()
    nums = push(nums, 5)
    nums = push(nums, 9)
    let head i64 = unwrap(first(nums), 0)
    let empty list<i64> = list_new()
    let none_head i64 = unwrap(first(empty), 100)
    n + picked + len(s) + unwrap(maybe, 0) + head + none_head

Existing locals can be updated with regular assignment or numeric compound assignment.

fn count -> i64
    let total i64 = 0
    total += 1
    total *= 42
    total

Executable fixture: tests/fixtures/valid/docs_count_assignment.lby.

Numeric literals may use _ as a digit separator between digits for readability — 1_000_000 and 3.141_592 — which does not change the value. A leading, trailing, doubled, or .-adjacent underscore is a malformed-literal parse error, and lullaby fmt renders literals without separators. Integers may also be written in hexadecimal (0xFF), binary (0b1010), or octal (0o17), with the same digit-separator rules; a base-prefixed literal is integer-only and an empty, out-of-radix, or fractional form is a parse error.

Arrays

Lullaby supports homogeneous arrays with array<T> type spelling, non-empty array literals, and bounds-checked indexing.

fn main -> i64
    let values array<i64> = [2, 4, 6]
    values[2]

Executable fixture: tests/fixtures/valid/run_array.lby.

  • Array literal values must all have the same static type.
  • Empty array literals are not accepted yet because empty-array inference is not implemented (use list<T> for growable, initially empty collections).
  • Index expressions require an i64 index.
  • Out-of-bounds indexes are runtime errors.

Array elements are mutable through assignment, including compound operators, and len(x) returns the i64 length of an array or string.

fn main -> i64
    let xs array<i64> = [1, 2, 3]
    xs[0] = 10
    xs[len(xs) - 1] += 4
    xs[0] + xs[2]

Char And Byte

Lullaby adds two scalar types beyond the numeric ones. A char is a single Unicode scalar written with single quotes ('a'); a char literal must hold exactly one character (an empty, multi-character, or unterminated literal is lexer error L0105). A byte is an 8-bit unsigned integer (0–255) with no literal form. Both compare with ==/!= and order with </<=/>/>= — chars by code point, bytes numerically.

  • char_code(c char) returns the Unicode scalar value as i64; char_from(i i64) returns the char for a scalar value (runtime error on an invalid scalar).
  • Character classification predicates return bool: is_digit (ASCII 0–9), is_alpha, is_alnum, is_whitespace, is_upper, and is_lower — useful for writing tokenizers and parsers in Lullaby.
  • byte(i i64) returns a byte (runtime error outside 0–255); byte_val(b byte) returns its i64 value.
  • to_string(x) renders any scalar — the whole numeric lattice (i64/f64/i8…usize/f32), bool, string, char, or byte. Wrong argument types or arities to these builtins report L0389.

Fixed-width integers

Beyond the default i64, Lullaby has the fixed-width integer types i8, i16, i32, u8, u16, u32, u64, and the pointer-sized isize/usize (64-bit on the current targets). u8 is the 8-bit unsigned arithmetic type — distinct from byte, the raw-I/O octet (constructed with byte(), which errors outside 0–255 rather than wrapping). There is no implicit numeric coercion: a fixed-width integer never mixes with an i64 or a different width in one arithmetic or comparison expression (mixing widths is an L0307 type error). Convert explicitly:

  • to_i8/to_i16/to_i32/to_u8/to_u16/to_u32/to_u64/to_isize/to_usize reinterpret an i64 into that width, truncating to the width's low bits (the wrapping conversion).
  • to_i64(x) widens any fixed-width integer back to i64; the bit pattern is reinterpreted, so a u64 above i64::MAX reads as a negative i64.

Arithmetic (+ - * /) on a fixed-width integer wraps modulo the type width — total and deterministic, never a trap — and division and comparison respect signedness. The same bit pattern reads as a large magnitude for an unsigned type but a negative value for a signed one: to_u32(0 - 1) is 4294967295, while to_i32(0 - 1) is -1, and to_u64(0 - 1) / to_u64(2) divides on the unsigned magnitude. Every backend normalizes at the same points, so results agree bit-for-bit.

Typed numeric-literal suffixes

A numeric literal can pin its type directly with a suffix instead of calling a conversion: 100i32, 4000000000u32, 0xFFu16, 120i8, 2.5f32. The i64/f64 suffixes are the defaults. The literal is range-checked at parse time — 256i8 and 100000u16 are rejected, and a decimal point with an integer suffix (1.5i32) is an error. A u64/usize literal is writable up to i64::MAX; larger (still valid) values are built with to_u64. A suffix never applies to a base-prefixed float form: 0xABF32 is the hex integer 0xABF32, not 0xAB with an f32 suffix.

# Typed numeric-literal suffixes: a suffix pins a literal to a fixed-width type
# directly (range-checked at parse time), desugaring to the matching to_
# conversion. Deterministic i64 (4000000519) on the AST, IR, and bytecode
# backends.
fn main -> i64
    let a i32 = 100i32
    let b u32 = 4000000000u32
    let c i8 = 120i8
    let d u16 = 0xFFu16
    let f i64 = 42i64

    # A large u64 literal (writable up to i64::MAX).
    let g u64 = 9000000000000000000u64
    let g_flag i64 = 0
    if g > to_u64(0)
        g_flag = 1

    # f32 suffix on a decimal literal.
    let e f32 = 2.5f32
    let e_flag i64 = 0
    if e > to_f32(2.0)
        e_flag = 1

    # 100 + 4000000000 + 120 + 255 + 42 + 1 + 1 = 4000000519
    to_i64(a) + to_i64(b) + to_i64(c) + to_i64(d) + f + g_flag + e_flag

Overflow-aware integer arithmetic

The + - * operators on a fixed-width integer wrap by default. When wrapping is wrong, these builtins give explicit control. Each takes two operands of the same fixed-width type T (not i64, whose default arithmetic already traps on overflow):

  • checked_add/checked_sub/checked_mul return option<T> — none on overflow.
  • saturating_add/saturating_sub/saturating_mul return T, clamped to T's bounds.
  • wrapping_add/wrapping_sub/wrapping_mul return T, wrapping modulo the width (the explicit form of the default).
# Overflow-aware fixed-width arithmetic: checked_* (option, none on
# overflow), saturating_* (clamp to bounds), and wrapping_* (wrap modulo width).
# Deterministic i64 (4294967541) on the AST, IR, and bytecode backends.

# checked_add on i8, unwrapping the option to i64 or a fallback.
fn checked_i8_or m i8 n i8 fallback i64 -> i64
    match checked_add(m, n)
        some(v) -> to_i64(v)
        none -> fallback

fn main -> i64
    # 127 + 1 overflows i8 -> none -> -1
    let a i64 = checked_i8_or(to_i8(127), to_i8(1), 0 - 1)
    # 100 + 20 stays in range -> some(120)
    let b i64 = checked_i8_or(to_i8(100), to_i8(20), 0 - 1)

    # saturating_add clamps to i8::MAX (127 + 100 -> 127)
    let c i8 = saturating_add(to_i8(127), to_i8(100))

    # saturating_mul clamps to u32::MAX (100000 * 100000 -> 4294967295)
    let d u32 = saturating_mul(to_u32(100000), to_u32(100000))

    # wrapping_add wraps u32 (u32::MAX + 1 -> 0)
    let e u32 = wrapping_add(to_u32(4294967295), to_u32(1))

    # -1 + 120 + 127 + 4294967295 + 0 = 4294967541
    a + b + to_i64(c) + to_i64(d) + to_i64(e)

32-bit floats

Alongside the default f64, Lullaby has the single-precision float f32. As with the integer widths, an f32 never mixes with an f64 in one arithmetic or comparison expression (mixing is an L0307 type error): to_f32(x f64) rounds an f64 to single precision and to_f64(x f32) widens back exactly. Every f32 operation is rounded to f32 precision, so it loses resolution an f64 keeps — 2^24 + 1 rounds back to 2^24 in f32 but is exact in f64.

# 32-bit floats: f32 arithmetic rounds to single precision, so it loses
# resolution an f64 keeps. 2^24 is the largest integer with a unique f32; adding
# 1 rounds back to 2^24 in f32 but is exact in f64. Deterministic i64 (3) on the
# AST, IR, and bytecode backends.
fn main -> i64
    let total i64 = 0

    # f32 cannot represent 2^24 + 1, so the increment rounds away.
    let big f32 = to_f32(16777216.0)
    let bumped f32 = big + to_f32(1.0)
    if bumped == big
        total = total + 1

    # f64 keeps the extra bit of precision.
    let big64 f64 = 16777216.0
    if big64 + 1.0 > big64
        total = total + 1

    # Basic f32 arithmetic and comparison, plus a round-trip widening to f64.
    let half f32 = to_f32(1.0) / to_f32(2.0)
    if to_f64(half) < 1.0
        total = total + 1

    total
# Fixed-width integer lattice (i32/u32): wrapping arithmetic, signed vs
# unsigned comparison, and conversions to/from i64. Every operation is
# width-normalized identically on the AST, IR, and bytecode backends, so this
# returns a fixed i64.
fn main -> i64
    # u32 subtraction wraps: 0 - 1 => 4294967295
    let a u32 = to_u32(0)
    let b u32 = to_u32(1)
    let wrapped i64 = to_i64(a - b)

    # i32 addition wraps at its max: 2147483647 + 1 => -2147483648
    let big i32 = to_i32(2147483647)
    let one i32 = to_i32(1)
    let overflow i64 = to_i64(big + one)

    # u32 multiplication wraps: 65536 * 65536 = 2^32 => 0
    let m u32 = to_u32(65536)
    let squared i64 = to_i64(m * m)

    # Same bit pattern (-1) reads as a huge u32 but a signed i32.
    let u_neg u32 = to_u32(0 - 1)
    let s_neg i32 = to_i32(0 - 1)
    let unsigned_big i64 = 0
    if u_neg > to_u32(5)
        unsigned_big = 1
    let signed_small i64 = 0
    if s_neg < to_i32(5)
        signed_small = 1

    # 4294967295 + (-2147483648) + 0 + 1 + 1 => 2147483649
    wrapped + overflow + squared + unsigned_big + signed_small
fn main -> i64
    let a char = 'A'
    let b char = char_from(char_code(a) + 1)
    let code i64 = char_code(b)
    let ordered i64 = 0
    if a < b
        ordered = 1
    let small byte = byte(10)
    let big byte = byte(250)
    let val i64 = byte_val(big)
    let byte_ordered i64 = 0
    if small < big
        byte_ordered = 1
    let text string = to_string(a) + to_string(small)
    let text_len i64 = len(text)
    code + ordered + val + byte_ordered + text_len

Enums

An enum NAME declaration, followed by indented Variant type... lines, declares a nominal tagged-union type at the top level. Each variant is a name plus zero or more positional, unnamed payload types. Construction reuses existing spelling with no new syntax: a payload variant reads like a call (Circle(2.0)), and a unit variant reads like a bare name (Empty). Variant names are globally unique across all enums.

enum Shape
    Circle f64
    Rect f64 f64
    Empty

fn sides s Shape -> i64
    4

fn main -> i64
    let c Shape = Circle(2.0)
    let r Shape = Rect(3.0, 4.0)
    let e Shape = Empty
    sides(c) + sides(r) + sides(e)

Executable fixture: tests/fixtures/valid/docs_enum.lby.

  • Invalid declarations (duplicate enum, duplicate variant, or empty enum) report L0380.
  • Construction arity or payload-type mismatches report L0381.
  • A variant name colliding across enums reports L0382.
  • Enums construct and pass identically across the AST, IR, and bytecode backends. Consume them with match; enum methods and generics over enums are still planned.

Pattern Matching

match SCRUTINEE selects on an enum value with an indented arm list. Each arm is a pattern, then ->, then an inline expression or an indented block whose last expression is the arm's value. A Variant(bind1, bind2, ...) pattern binds the variant's positional payloads to arm-scoped locals (typed by the variant's declared payload types); a bare Variant matches a unit variant; _ is a wildcard. Like if/try, a match is an expression: when every arm yields the same type it produces that value, so it can be a function's final expression.

enum Shape
    Circle i64
    Rect i64 i64
    Empty

fn area s Shape -> i64
    match s
        Circle(r) -> r * r
        Rect(w, h) -> w * h
        Empty -> 0

fn main -> i64
    area(Rect(4, 5))

Executable fixture: tests/fixtures/valid/docs_match.lby.

  • Matching must be exhaustive: every variant covered, or a _ wildcard arm present. A non-exhaustive match reports L0384.
  • The scrutinee must be an enum type (L0383); an unknown or duplicate variant arm, or a wrong payload binding count, reports L0385.
  • match runs identically across the AST, IR, and bytecode backends, including under optimization.

option and result

option<T> and result<T, E> are two built-in generic enums for absence and fallible results. option<T> has variants some(value) and none; result<T, E> has variants ok(value) and err(error). The variant names some/none/ok/err are reserved, and option/result cannot be redeclared as user enums. Construction is context-directed: some(v) synthesizes option<typeof v>, while none, ok(v), and err(v) take their type from a let annotation or a function return type.

fn unwrap_or o option<i64> fallback i64 -> i64
    match o
        some(v) -> v
        none -> fallback

fn divide n i64 d i64 -> result<i64, string>
    if d == 0
        return err("divide by zero")
    ok(n / d)

fn main -> i64
    let value option<i64> = some(41)
    let quotient result<i64, string> = divide(84, 2)
    match quotient
        ok(q) -> unwrap_or(value, 0) + q
        err(m) -> len(m)

Executable fixture: tests/fixtures/valid/docs_option_result.lby.

  • Type spelling reuses the generic form with a comma-separated argument list: option<i64>, result<i64, string>, option<array<i64>>.
  • Building none/ok/err takes its type from context — a let annotation, a function return type, a match scrutinee, or a call argument's parameter type (so describe(ok(5)) and f(none) infer without a separate annotated let). Without any such context it reports L0386; a payload whose type disagrees with the annotation reports L0303.
  • match over an option<U> covers some + none (or _); over a result<T, E> it covers ok + err (or _), reusing L0384/L0385.
  • The postfix ? operator propagates errors: expr? evaluates to the payload on ok/some and otherwise returns err(e)/none from the enclosing function. The enclosing function must return a compatible result<U, E> (same E) or option<U>; misuse reports L0427/L0428/L0429. It desugars to a match with an early return, so it behaves identically on every backend.
  • option/result construct and match identically across the AST, IR, and bytecode backends, including under optimization.

Growable lists (list<T>)

fn build -> list<i64>
    let l list<i64> = list_new()
    l = push(l, 10)
    l = push(l, 20)
    l = push(l, 30)
    l

fn main -> i64
    let l list<i64> = build()
    l = set(l, 1, 25)
    let a i64 = get(l, 0)
    let b i64 = get(l, 2)
    let n i64 = len(l)
    l = pop(l)
    let m i64 = len(l)
    let last i64 = get(l, 1)
    a + b + n + m + last

Executable fixture: tests/fixtures/valid/run_list.lby (prints 70).

  • list<T> is a growable, value-semantic list. Its "mutating" builtins are functional and return a new list, so the idiom is l = push(l, x).
  • list_new() builds an empty list and takes its element type from the expected list<...> type (a let annotation, a function return type, or a call argument's parameter type — so push(list_new(), byte(65)) inside a let list<byte> infers list<byte>); missing that context reports L0387.
  • push(l, x) appends, get(l, i) reads (bounds-checked, L0413), set(l, i, x) replaces at an index, pop(l) drops the last element, and len(l) returns the i64 length. Wrong argument types or arities report L0387.
  • reverse(l) returns a new reversed list, concat(a, b) appends two lists of the same element type, and slice(l, start, end) returns the half-open range [start, end) with the bounds clamped into [0, len] (an empty list when start >= end). sort(l) returns a new list<i64> sorted ascending. list_index_of(l, x) returns the index of the first element equal to x (or -1), and list_contains(l, x) returns a bool — the search value must match the element type.

Hash maps (map<K, V>)

fn build -> map<string, i64>
    let m map<string, i64> = map_new()

fn lookup m map<string, i64> k string -> i64
    match map_get(m, k)
        some(v) -> v
        none -> 0

Traits

A trait names a set of methods a type can implement. A trait declaration lists method signatures fn method self [param Type ...] -> Ret with no body; the receiver is self, and Self in a later parameter or return type means the implementing type. An impl Trait for Type block provides a body for every method the trait requires. A trait method is called with the usual UFCS spelling — p.show() desugars to show(p) and dispatches on the receiver's concrete runtime type. A generic function bounds a type parameter with <T: Trait> (or <T: A + B>): inside the function a value of type T may call the bound trait's methods, and at a call site the inferred concrete type must implement every bound. Dispatch uses one shared impl table consulted identically by the AST, IR, and bytecode backends.

# Traits: declaration, impls for a struct and an enum, receiver-type method
# dispatch, and a bounded generic that calls a trait method.

trait Show
    fn show self -> string

struct Point
    x i64
    y i64

enum Light
    Red
    Green

impl Show for Point
    fn show self -> string
        "(" + to_string(self.x) + ", " + to_string(self.y) + ")"

impl Show for Light
    fn show self -> string
        match self
            Red -> "red"
            Green -> "green"

fn describe<T: Show> v T -> string
    "show=" + v.show()

fn main -> i64
    let p Point = Point(3, 4)
    let g Light = Green
    # Direct trait-method calls dispatch by the receiver's concrete type.
    let a i64 = len(p.show())
    let b i64 = len(g.show())
    # A bounded generic resolves `v.show()` on the concrete runtime type.
    let c i64 = len(describe(p))
    let d i64 = len(describe(g))
    a + b + c + d

Concurrency

parallel_map(f fn(i64) -> i64, args list<i64>) -> list<i64> evaluates f(arg) for every element of args concurrently on separate OS threads and returns the results in the same order as args, so the output is deterministic regardless of thread scheduling. Each thread builds a fresh interpreter over the shared program, so heaps are per-thread with no shared mutable state and no locking; f must be an ordinary top-level fn(i64) -> i64. It runs identically on the AST, IR, and bytecode backends. Wrong arity, a non-fn(i64) -> i64 first argument, or a non-list<i64> second argument report L0334.

Atomics. atomic_i64 is a shared, thread-safe 64-bit counter created with atomic_new(v i64) -> atomic_i64. All operations use sequentially-consistent ordering: atomic_load(a) -> i64, atomic_store(a, v), atomic_swap(a, v) -> i64 (returns the previous value), atomic_cas(a, expected, new) -> i64 (strong compare-and-swap, returns the value found), and the fetch-and-op family atomic_add/atomic_sub/atomic_and/atomic_or/atomic_xor (each returns the previous value). The handle is shared across threads, so concurrent updates never lose writes. Those default forms are sequentially consistent; the full memory-ordering set is available via the MemoryOrder prelude enum (relaxed/acquire/release/acq_rel/seq_cst) and the ordered variants atomic_load_ordered(a, order), atomic_store_ordered(a, v, order), atomic_swap_ordered/atomic_{add,sub,and,or,xor}_ordered, and atomic_cas_ordered(a, expected, new, success_order, failure_order), plus standalone fence(order) — each maps to the real std::sync::atomic ordering (an acquire load truly uses acquire), and an ordering invalid for an operation (a release load, an acquire store, a relaxed fence) is rejected with L0432.

    x * x

    base = push(base, 1)
    base = push(base, 2)
    base = push(base, 3)
    base = push(base, 4)
    let out list<i64> = parallel_map(sq, base)

Threads, channels, and a mutex

Message passing with detached threads is delivered on all three backends. chan_new() -> Chan creates an unbounded i64 channel (a shared handle: cloning shares the same queue); send(ch Chan, v i64) enqueues without blocking, recv(ch Chan) -> i64 dequeues blocking, and try_recv(ch Chan) -> option<i64> dequeues non-blocking. spawn(f fn(Chan, i64) -> void, ch Chan, v i64) -> Task runs f(ch, v) on a detached OS thread that owns a share of the program (an Arc clone) and returns a Task; task_join(t Task) -> void waits for it (named task_join, not join, because join is the string-list joiner). mutex_new(v i64) -> Mutex creates a shared mutex over one i64; mutex_get/mutex_set/mutex_add read, write, and atomically add-and-return. Because scheduling is non-deterministic, sum received values instead of asserting order. Wrong arity or a wrong-typed argument reports L0337. Generic channels, select, async/await, and cross-thread socket sharing remain deferred.

fn worker ch Chan v i64 -> void
    send(ch, v * v)

fn main -> i64
    let ch Chan = chan_new()
    let t1 Task = spawn(worker, ch, 1)
    let t2 Task = spawn(worker, ch, 2)
    let t3 Task = spawn(worker, ch, 3)
    let t4 Task = spawn(worker, ch, 4)
    task_join(t1)
    task_join(t2)
    task_join(t3)
    task_join(t4)
    let total i64 = 0
    for i from 0 to 3
        total += recv(ch)
    let m Mutex = mutex_new(0)
    for j from 0 to 3
        mutex_add(m, 1)
    total + mutex_get(m)

Boolean Logic

Lullaby supports boolean logic with and, or, and unary not. Logical operands must be bool, and and/or short-circuit during runtime execution.

fn main -> bool
    not false and true or false

Executable fixture: tests/fixtures/valid/run_logic.lby.

Logical expressions can be used anywhere a bool is required, including branch and loop conditions.

Bitwise operators work on every integer type — i64 and the fixed-width kinds (i8…usize): & (and), | (or), ^ (xor), ~ (not), << (shift left), and >> (shift right). Precedence, tightest to loosest, is ~, then * /, + -, << >>, &, ^, |, then comparisons and and/or. Both operands must be the same integer type (no width mixing), and the result is that type. The shift amount is masked to the type's width (amount & (width-1) — e.g. & 63 for i64, & 7 for a u8), so any amount is total and deterministic. Right shift is logical (zero-filling) for unsigned types and arithmetic (sign-preserving) for signed types; ~ is the one's complement re-normalized to the width.

Testing

assert(cond bool) -> void is the assertion primitive. When cond is false it raises a catchable runtime error with the message assertion failed — the same error path a user throw uses, so try/catch recovers it; when cond is true it returns void. A non-bool argument reports L0342.

lullaby test path/to/tests.lby runs a language-level test suite. It validates the file as a library (no main required), then runs every top-level function whose name starts with test_, takes zero parameters, and returns void/i64/bool through the interpreter. A test passes if it returns without a runtime error and fails if it produces one (an assert(false), a throw, or any other runtime error). The runner prints PASS test_name or FAIL test_name: <message> per test, then a N passed, M failed summary, and exits non-zero when any test fails. A runnable demo suite ships at examples/valid/tests_demo/tests_demo.lby.

Math Builtins

Lullaby includes a small, type-directed math library over i64 and f64. abs(x) returns the magnitude (i64->i64 or f64->f64). min(a, b) and max(a, b) take two matching numeric operands and return the smaller or larger. pow(base, exp) raises to a power; the integer form requires exp >= 0 (a negative integer exponent is runtime L0417). sqrt, floor, ceil, and round take one f64 and return f64. The transcendental builtins also take and return f64: sin, cos, tan, atan, exp, ln, and log10 are unary, and atan2(y, x) takes two f64s and returns the angle in radians. clamp(x, lo, hi) limits a value to [lo, hi] (three matching i64/f64; lo > hi yields lo). sign(x) returns -1/0/1 as i64. gcd(a, b) returns the non-negative greatest common divisor of two i64s. Over a list<i64>/list<f64>, list_sum returns the element-type sum (empty → 0), and list_min/list_max return option<T> (none on empty). The higher-order list builtins take a function value (a named function or a closure literal): list_map(l, f) maps f: fn(T)->U over the list, list_filter(l, pred) keeps elements where pred: fn(T)->bool, and list_reduce(l, init, f) left-folds with f: fn(U,T)->U. sort(l) sorts a list<i64>/list<f64>/list<string> ascending (stable), and sort_by(l, cmp fn(T,T)->i64) sorts by a comparator (negative/zero/positive; stable, and total even for an inconsistent comparator). A wrong or mismatched operand type reports L0374.

Bitwise intrinsics operate on i64: rotate_left(x, n) / rotate_right(x, n) rotate the 64 bits by n & 63 positions; count_ones(x) is the population count; leading_zeros(x) / trailing_zeros(x) count high/low zero bits (0–64); and reverse_bytes(x) swaps byte order. All are total and deterministic.

Modules

Each .lby file is a module. A top-level fn, struct, enum, or alias is private to its file unless you prefix it with pub, which exports it. A file pulls in another module with one or more leading import NAME lines placed before any declaration; import NAME loads NAME.lby from the same directory as the entry file and makes that module's pub items available unqualified.

A frontend-only loader lexes, parses, resolves imports (recursively), enforces visibility, and merges every module into one program before type checking, so the type checker and all three backends run unchanged. The merged namespace is flat with no shadowing.

# geometry.lby
pub struct Point
    x i64
    y i64

pub fn dot a Point b Point -> i64
    a.x * b.x + a.y * b.y

fn priv_helper n i64 -> i64      # private to geometry.lby
    n * n
# main.lby
import geometry

fn main -> i64
    let p Point = Point(3, 4)
    dot(p, p)                    # uses pub Point and dot; returns 25

Running lullaby run main.lby loads both files and prints 25. Using priv_helper from main.lby would report L0392, because it is private to geometry.lby. A name declared in two modules reports L0391, an import cycle reports L0393, and a missing module file reports L0397. Dotted module paths, qualified access, and per-field visibility are not yet implemented.

Projects

A program can grow beyond a single directory into a project: a directory with a lullaby.json manifest that names the project, an optional executable entry (a .lby file), one or more src source directories (default ["."]), and dependencies mapping a name to another local project root that has its own lullaby.json. In a project build, import NAME resolves NAME.lby by searching the importing file's directory, then every src directory of the project, then every src directory of each dependency; the same visibility and no-shadowing rules apply across the whole set. The commands run, check, build, test, compile, wasm, and native accept either a single .lby file (unchanged) or a project directory / lullaby.json path. run/build/compile/wasm/native need an entry; check/test can validate a library project without one. A missing or malformed manifest, a missing src directory, or a missing dependency reports L0343. Dependencies are local paths only; remote/registry fetching is deferred. The worked project lives at examples/valid/project_demo/ (with its mathx dependency); lullaby run examples/valid/project_demo prints 45.

cargo run -p lullaby_cli -- wasm --verbose -o path/to/file.wasm path/to/file.lbyCompile the source's scalar functions (i64/f32/f64/bool/char/byte, plus strings/structs/fixed arrays, growable list<T>, and map<K, V> (scalar or string keys by content-equality; scalar/string values) as linear-memory pointers) to a binary .wasm module, exported by name; --verbose lists which functions compiled and which were skipped. Within those functions the WASM backend also compiles the fixed-width integer types (i8/i16/i32/u8/u16/u32/u64/isize/usize): wrapping arithmetic, signedness-correct comparison and division, bitwise & | ^ ~, width-masked shifts, and the to_<T>/to_i64 conversions, each stored as the same normalized i64 cell the interpreters use. It also compiles f32/f64 floating-point values: single/double-precision arithmetic + - * / (f32.add…/f64.add…, so f32 stays single precision), IEEE-754 NaN-aware comparisons < <= > >= == !=, and the to_f32 (f32.demote_f64, round) / to_f64 (f64.promote_f32, widen) conversions inlined as builtin calls, matching the interpreters bit-for-bit (node execution parity). It also compiles enum values and match for enums with scalar payloads — the built-in option<T> and result<T, E> when T/E are scalar, plus user enums whose every variant payload is scalar — laying each enum out in linear memory as a [tag: i32][payload slots…] record (the tag is the variant’s declaration-order index, matching the interpreters) and dispatching match on the loaded tag (including _ wildcard arms). It also compiles runtime string + concatenation, allocating a fresh [char_len][byte_len][utf8] record and copying both operands’ byte ranges via memory.copy (len of the result is the summed char count, matching the interpreters). It also compiles to_string(x) for the integer widths (in-WASM itoa, incl. i64::MIN/u64::MAX), bool, char (UTF-8), byte, and string (identity); to_string of a float is still deferred (dtoa). The index-based string ops compile too: substring and find (char-indexed, UTF-8 aware) and the byte-wise contains/starts_with/ends_with; replace/upper/lower/split/join still fall back. It also compiles list<string> and map<K, string> (string elements/values are shared immutable pointers) and enums with a string payload (so option<string>/result<i64, string> and string-carrying user enums now compile). Lists/maps of one-level mutable-heap elements (list<struct>, list<list<scalar>>, map<K, struct>) and enums with a one-level mutable payload (option<struct>, result<i64, list<i64>>) now compile with recursive value-semantic deep copies. Deeper nesting, array/map elements, float math builtins (sqrt/sin/…), heap values beyond the pointer types above, and the to_string-of-float builtin is still deferred: functions using it are skipped and still run on the interpreters, and when no function is eligible it reports L0338. The overflow builtins checked_*/saturating_*/wrapping_* now compile.
cargo run -p lullaby_cli -- native --verbose -o path/to/file.exe path/to/file.lbyEmit a native x86-64 COFF object for the source's i64-scalar functions (all params and return i64) and, best-effort, link it into a runnable Windows console .exe; the process exit code is main's i64 result (mod 256). --verbose lists which functions compiled and which were skipped. The native backend compiles i64-scalar functions and, within them, the fixed-width integer types (i8/i16/i32/u8/u16/u32/u64/isize/usize): their wrapping arithmetic, signedness-correct comparison, bitwise and shift operators, and the to_<T>/to_i64 conversions, each width-normalized to match the interpreters bit-for-bit. It also compiles f64/f32 floating-point values within those otherwise-i64 functions using SSE scalar (XMM) registers: IEEE-754 arithmetic + - * /, the ordered comparisons < <= > >= == !=, and the to_f32/to_f64 conversions, with every f32 op rounded to single precision so results match the interpreters bit-for-bit. It also compiles enum values with scalar payloads and match: option<T>/result<T, E> (scalar T/E) and user enums are laid out as a tag word plus a payload region, and match dispatches on the tag (binding scalar payloads, honoring the _ wildcard) with the interpreters’ variant ordering. Native functions accept any number of parameters (5th+ passed on the stack per the Win64 ABI). It also compiles growable list<T> of scalar elements (list_new/push/get/set/len/pop) as a heap-backed [len][cap][slots] record with capacity-doubling growth and value semantics, and lists cross function boundaries — at parity with the WASM backend. It also compiles scalar map<K, V> (map_new/map_set/map_get→option<V>/map_has/map_len) as an insertion-ordered [len][cap][(key,value) entries] heap record with in-place update, capacity-doubling growth, and value semantics — integer/bool/char/byte keys, any scalar value (string/heap/float keys deferred). list<string>, map<i64, string>, and enums with a string payload (option<string>/result<i64, string>) also compile (string elements/values/payloads are shared immutable pointers), at parity with the WASM backend. One-level mutable-heap elements compile too — list<struct>, list<list<scalar>>, map<i64, struct>, and enums with a mutable payload (option<struct>, result<i64, list<i64>>) — with recursive value-semantic deep copies (via a HeapStruct representation), matching the interpreters. It also compiles first-class heap string values: string literals as values, string locals/parameters/returns (an immutable pointer word, no copy), runtime + concatenation, len of any string, to_string for integers/bool/char/byte/string, and the index-based string ops substring/find (char-indexed, UTF-8 aware) and byte-wise contains/starts_with/ends_with — to_string of a float and replace/upper/lower/split/join still fall back. Functions with a float parameter or return type, heap values (including heap-payload enums like result<i64, string>, and enums used as parameters/returns or matched from an enum-returning call), the transcendental/math builtins (sqrt/sin/floor/…), are still skipped and run on the interpreters, and when no eligible function is found it reports L0339. The overflow builtins checked_*/saturating_*/wrapping_* now compile natively. The default target emits a Windows x86-64 COFF object; --target x86_64-unknown-linux-gnu emits an ELF64 object and --target x86_64-apple-darwin a Mach-O object (same x86-64 machine code, a freestanding raw-syscall entry stub, structurally verified) — cross-linking and run of those is deferred to cross-platform CI, and aarch64-apple-darwin is rejected (L0347, no ARM64 codegen yet). The object is always written; linking is attempted only for the Windows target when rust-lld and kernel32.lib (via the MSVC LIB environment) are available. Add --debug (alias -g) to also emit a CodeView .debug$S source-line section mapping each function's entry offset to its .lby declaration line; it is opt-in and leaves the default object bytes unchanged.

Current Limitations

This page documents the currently implemented language. Lullaby is in active development toward 1.0, so some systems features described in the broader Markdown specs are still planned and may not be accepted by the compiler yet.

  • Execution runs on AST, typed IR, and instruction-bytecode interpreters plus versioned .lbc bytecode artifacts; a scalar-subset WebAssembly backend (lullaby wasm) and a native x86-64 backend (lullaby native) exist. The native backend compiles i64-scalar functions — including the full scalar control flow (if/elif/else, while, loop, and range for i from a to b [by s] with break/continue) and inter-function calls among compiled functions (Win64 register arguments, result in rax, including recursion) — and, within them, the fixed-width integer operations (arithmetic, comparison, bitwise, shifts, and to_<T>/to_i64 conversions) and f64/f32 floating-point operations (arithmetic, comparison, and to_f32/to_f64 conversions, via SSE scalar registers with single-precision f32 rounding). It also compiles enum values with scalar payloads and match over them: the built-in option<T>/result<T, E> (for scalar T/E) and user enums are laid out on the stack as a tag word plus a payload region, and match dispatches on the tag (binding scalar payloads, honoring the _ wildcard) with the same variant ordering as the interpreters. Enums whose payload is a heap value (e.g. result<i64, string>), enums used as parameters/returns, and a match on an enum-returning call are deferred and fall back to the interpreters. Float-typed signatures, heap values, math builtins, and option-returning builtins also still fall back to the interpreters. The WASM backend covers the scalar subset (plus strings/structs/fixed arrays, growable list<T>, and map<K, V> (scalar or string keys by content-equality; scalar/string values) as linear-memory pointers) and, within those functions, the same fixed-width integer operations as native (arithmetic, comparison, bitwise, shifts, and to_<T>/to_i64 conversions) and the same f32/f64 floating-point operations (single/double-precision arithmetic, NaN-aware comparison, and the to_f32/to_f64 conversions, with f32 kept in single precision). Unlike native, the WASM backend also compiles enum values and match for enums with scalar payloads (the built-in option/result when their payloads are scalar, and user enums whose every variant payload is scalar), laying each enum out as a tag+payload linear-memory record and dispatching match on the loaded tag. The WASM backend also passes and returns scalar-field aggregates across function boundaries — a struct (of scalars, nested scalar structs, or fixed scalar arrays), a fixed scalar array, or a scalar-payload enum may be a parameter, return value, or call argument — while preserving Lullaby value semantics: each mutable aggregate argument is deep-copied into a fresh linear-memory record at the call site (recursing into nested aggregates) so a callee mutating its parameter cannot change the caller's copy, exactly matching the interpreters; an immutable string argument is shared rather than copied, and a returned aggregate is the callee's own fresh record. Enums with a heap payload (e.g. result<i64, string>), aggregates containing list/map, float math builtins, other heap values, and the option-returning builtins are still deferred there. Non-eligible functions still run on the interpreters, and full native lowering of the whole language is still in progress.
  • The optimizer currently exposes opt-in constant folding, conservative common subexpression elimination, conservative loop-invariant motion, conservative block-local copy propagation, block-local dead-code elimination, and the combined alpha pipeline.
  • Typed IR reports memory operations and bounds-checked indexing with sequence and safety metadata, and version 5 .lbc artifacts preserve that metadata in memory_operations. The full region memory model, ARC/reference counting, cleanup insertion, and lifetime analysis are still pending.
  • The current I/O surface is text/binary file builtins, directory operations, direct program-plus-argv command calls, TCP/UDP sockets, and a minimal HTTP/1.1 client; buffered streams, memory mapping, async, IPC, and TLS/HTTPS are future work.
  • Structs, enums, pattern matching (match), traits with bounded generics, generic functions, option/result/list/map, modules and visibility (import/pub), multi-directory projects (lullaby.json), structured error handling (throw/try/catch), and threads/channels/mutex concurrency are all implemented today. Still planned: default trait methods, trait objects (dyn), associated types and supertraits, capturing closures (lambda literals), enum methods and generics over enums, wider integer types (i32, u32, …), and byte arithmetic.
  • The current distribution path is a Windows portable archive with optional user PATH helper scripts; the cross-platform package driver also prepares Unix PATH helpers for Linux/macOS archives, which still need non-Windows host validation and active CI runs. A full graphical installer is future work.
  • Genuinely reserved keywords that are not yet implemented — module, package, union, interface, class, switch, catch (as a standalone statement), and coroutine — are rejected with L0211 [parser] until their feature areas are implemented.
  • Inline assembly (asm b0, b1, ..., a list of raw 0..=255 machine-code bytes inside an unsafe block) is emitted verbatim into the current function's .text by the native backend only. It is inherently unsafe: the bytes are trusted and not decoded, relocated, or register-modeled, and a trailing asm that leaves a value in rax becomes the function's return value. The AST, IR, and bytecode interpreters cannot execute raw machine code, so any asm statement is rejected with L0425; compile with lullaby native to emit and link it. Out-of-range bytes or use outside unsafe are caught at check time (L0425 / L0330).
  • Raw-memory / layout intrinsics: size_of(x) and align_of(x) return the C-natural byte size / alignment of x's type (scalars 1/2/4/8 by width, pointers 8, structs field-aligned and rounded to their max-field alignment, fixed arrays n·stride); offset_of(x, "field") is a struct field's byte offset. The unsafe-gated ptr_to_int(p)/int_to_ptr(n) round-trip a pointer through an integer, and volatile_load(p)/volatile_store(p, v) are non-eliding raw reads/writes (the volatility guarantee is realized by the native backend; the interpreters behave like load/store). An unsupported layout type or unknown field is L0431.
  • Calling C via extern fn (C-ABI FFI) is native-only. A body-less extern fn NAME params -> Ret declares an imported C symbol; on the native backend a call lowers to a real Win64 call of that external symbol, linked against the C runtime import library ucrt.lib. Marshalling covers the integer-class C scalars — every fixed-width integer (i8…u64, isize/usize) plus bool/char/byte: arguments pass in the low bits of rcx/rdx/r8/r9 and a narrow C return is re-normalized in rax (sign-extend for signed, zero-extend for unsigned). For example, extern fn toupper c i32 -> i32 links against ucrt and toupper('a') returns 'A'. Floating (f32/f64) externs are marshalled through the XMM argument registers (xmm0–xmm3 by Win64 position, mixing with the integer registers per argument type) with the float return read from xmm0 — so e.g. extern fn sqrt x f64 -> f64 links against ucrt and runs natively. The AST, IR, and bytecode interpreters cannot execute C, so any extern fn call is rejected with L0423 — compile with lullaby native to link and run it. lullaby check still validates extern declarations and their call sites. Beyond scalars, extern fn now marshals pointers (a ptr<T> arg/return passes the raw machine address, so a Lullaby-held pointer round-trips through C), cstr (a string passed to a cstr parameter is materialized as a NUL-terminated const char*), and more than four arguments (the 5th+ spill onto the Win64 stack) — verified by linking against ucrt and running (strlen, malloc+strcpy, a six-argument call). Exposing Lullaby to C via export fn is implemented for scalar signatures (i64/f64/f32). Struct-by-value and callbacks (passing a Lullaby fn as a C function pointer) are rejected with L0424 and remain the deferred FFI tail (ABI-fiddly and not verifiable on the Windows host without a matching C toolchain); non-Windows C ABIs are deferred with the cross-platform work.
  • The native backend (lullaby native) compiles the i64-scalar subset to an x86-64 Windows COFF object and, best-effort, links it into a runnable .exe. Adding --freestanding (alias --no-std) builds a no-C-runtime executable: it links kernel32.lib only (zero ucrt/vcruntime/msvcrt) and terminates through the minimal OS import kernel32!ExitProcess, bypassing the C runtime startup. It is still a Windows PE, not a bare-metal binary; true bare-metal (raw-syscall exit with no OS imports, other object formats, a custom entry and linker script) is deferred. A freestanding build that declares an extern fn (which needs the C runtime) is rejected with L0426. Adding --debug (alias -g) emits native source-line debug info: a CodeView .debug$S section maps each compiled function's entry offset to its .lby declaration line (per-function granularity) so a debugger can break at a function and show its source line; --debug is opt-in and without it the object bytes are unchanged. Per-statement line mapping, variable/type info, and a full compiler-emitted PDB are deferred.
  • The checked-in documentation bundle is currently hand-authored HTML. The standard-library generator from canonical docs is available at offline_docs/generate_offline_docs.py, emits package-ready quick start, CLI reference, package layout, diagnostics, limitations, shipped-doc parity sections, and fixture-backed examples, and is now used by the Windows portable package path.
