Lullaby

Offline documentation for the current Lullaby alpha: a compiled systems language prototype with concise indentation-only syntax, strong typing, and an executable Rust-based frontend/runtime.

Overview

Lullaby is implemented in Rust. The current alpha focuses on a clear, testable compiler frontend plus AST, IR, and initial bytecode execution paths before native code generation.

The Alpha 1 language surface is frozen in documents/alpha1_language_surface.md. Broader Markdown design docs are planned material unless the feature appears in that file or on this page.

Extension.lby is the canonical source file extension.
BlocksIndentation defines scope. Curly braces are rejected.
StatementsSemicolons are not statement terminators.
Runtimelullaby run executes the supported subset with AST, IR, or bytecode backends.

Quick Start

Create a file named hello.lby:

fn main -> i64
    42

Executable fixture: tests/fixtures/valid/docs_quick_start.lby.

Check it:

cargo run -p lullaby_cli -- check hello.lby

Run it:

cargo run -p lullaby_cli -- run hello.lby

Run it through the typed IR backend:

cargo run -p lullaby_cli -- run --backend ir hello.lby

Run it through the typed IR backend with the opt-in constant-folding optimizer:

cargo run -p lullaby_cli -- run --backend ir --optimize constant-fold hello.lby

Run it through the current alpha optimizer pipeline:

cargo run -p lullaby_cli -- run --backend ir --optimize alpha hello.lby

Compile it to a bytecode artifact with metadata, a function table, and memory operation metadata; inspect the artifact, then run it:

cargo run -p lullaby_cli -- compile --optimize alpha -o hello.lbc hello.lby
cargo run -p lullaby_cli -- build --optimize alpha -o hello-build.lbc hello.lby
cargo run -p lullaby_cli -- inspect hello.lbc
cargo run -p lullaby_cli -- run hello.lbc

During development, the CLI is available through Cargo. The Alpha 1 portable package builds a direct bin\lullaby.exe command, bundles docs\index.html, includes install.cmd and uninstall.cmd for optional user PATH setup, ships user-facing examples such as examples\valid\calculator.lby, emits lullaby-alpha1-windows-x64.zip.sha256, and supports lullaby docs and lullaby examples to locate local resources. The cross-platform package driver also prepares install.sh and uninstall.sh for Linux/macOS portable archives after those targets are validated.

Source Files

  • Use the .lby extension.
  • Use spaces for indentation. A deeper indent opens a block; dedenting closes it.
  • Do not use { or } as block delimiters.
  • Do not end statements with semicolons.
  • Comments begin with # and run to the end of the line.

Functions

Functions start with fn, followed by the name, typed parameters, an optional return arrow, and an indented body.

fn add x i64 y i64 -> i64
    x + y

When a function body reaches its final expression normally, that expression is the return value.

Void functions

A function that returns nothing declares -> void. It may use an empty return, or simply finish with no value-producing final expression.

fn release ptr ptr_i64 -> void

    return

First-class functions

A function type is spelled fn(T1, T2) -> R. A bare top-level function name is a value of that type, so a function can be stored in a let, passed to another function, returned, and called through a local. Capturing closures (lambda literals) are not yet implemented. Calling a non-function local, or passing a function value with the wrong signature, reports L0390.

fn inc x i64 -> i64
    x + 1

fn dbl x i64 -> i64
    x * 2

fn apply f fn(i64) -> i64 v i64 -> i64
    f(v)

fn picker -> fn(i64) -> i64
    dbl

fn main -> i64
    let g fn(i64) -> i64 = inc
    let h fn(i64) -> i64 = picker()
    apply(inc, 10) + g(5) + h(20) + apply(dbl, 3)

Executable fixture: tests/fixtures/valid/run_first_class_fn.lby. Returns 63.

Generic functions

A function can declare type parameters in angle brackets after its name and use each name as an ordinary type in its parameters and return type, including inside built-in generics such as list<T> and option<T>. Call sites infer the type parameters from the argument types and substitute them into the return type — identity(41) yields i64, identity("hi") yields string, and wrap(1) yields option<i64>. Generics are erased at runtime, so they run identically on every backend. A bare T value supports only universal operations (binding, passing, returning, equality between two same-T values, and use as a payload to a built-in generic like some(x)); arithmetic and ordering on a bare T wait on trait bounds.

fn identity<T> x T -> T
    x

fn wrap<T> x T -> option<T>
    some(x)

fn choose<T> pick bool a T b T -> T
    if pick
        return a
    b

fn unwrap<T> o option<T> fallback T -> T
    match o
        some(v) -> v
        none -> fallback

fn first<T> xs list<T> -> option<T>
    if len(xs) == 0
        return none
    some(get(xs, 0))

fn main -> i64
    let n i64 = identity(41)
    let s string = identity("hi")
    let picked i64 = choose(true, 10, 20)
    let maybe option<i64> = wrap(1)
    let nums list<i64> = list_new()
    nums = push(nums, 5)
    nums = push(nums, 9)
    let head i64 = unwrap(first(nums), 0)
    let empty list<i64> = list_new()
    let none_head i64 = unwrap(first(empty), 100)
    n + picked + len(s) + unwrap(maybe, 0) + head + none_head

Executable fixture: tests/fixtures/valid/run_generics.lby. Returns 159.

Existing locals can be updated with regular assignment or numeric compound assignment.

fn count -> i64
    let total i64 = 0
    total += 1
    total *= 42
    total

Executable fixture: tests/fixtures/valid/docs_count_assignment.lby.

Numeric literals may use _ as a digit separator between digits for readability — 1_000_000 and 3.141_592 — which does not change the value. A leading, trailing, doubled, or .-adjacent underscore is a malformed-literal parse error, and lullaby fmt renders literals without separators.

Arrays

The alpha supports homogeneous arrays with array<T> type spelling, non-empty array literals, and bounds-checked indexing.

fn main -> i64
    let values array<i64> = [2, 4, 6]
    values[2]

Executable fixture: tests/fixtures/valid/run_array.lby.

  • Array literal values must all have the same static type.
  • Empty array literals are not accepted yet because empty-array inference is not implemented.
  • Index expressions require an i64 index.
  • Out-of-bounds indexes are runtime errors.

Array elements are mutable through assignment, including compound operators, and len(x) returns the i64 length of an array or string.

fn main -> i64
    let xs array<i64> = [1, 2, 3]
    xs[0] = 10
    xs[len(xs) - 1] += 4
    xs[0] + xs[2]

Executable fixture: tests/fixtures/valid/docs_array_mutation.lby.

Char And Byte

The alpha adds two scalar types beyond the numeric ones. A char is a single Unicode scalar written with single quotes ('a'); a char literal must hold exactly one character (an empty, multi-character, or unterminated literal is lexer error L0105). A byte is an 8-bit unsigned integer (0–255) with no literal form. Both compare with ==/!= and order with </<=/>/>= — chars by code point, bytes numerically.

  • char_code(c char) returns the Unicode scalar value as i64; char_from(i i64) returns the char for a scalar value (runtime error on an invalid scalar).
  • byte(i i64) returns a byte (runtime error outside 0–255); byte_val(b byte) returns its i64 value.
  • to_string(x) also accepts a char or a byte. Wrong argument types or arities to these builtins report L0389. No byte arithmetic is implemented for Alpha 1, and other sized integer types (i32, u32, …) remain planned.
fn main -> i64
    let a char = 'A'
    let b char = char_from(char_code(a) + 1)
    let code i64 = char_code(b)
    let ordered i64 = 0
    if a < b
        ordered = 1
    let small byte = byte(10)
    let big byte = byte(250)
    let val i64 = byte_val(big)
    let byte_ordered i64 = 0
    if small < big
        byte_ordered = 1
    let text string = to_string(a) + to_string(small)
    let text_len i64 = len(text)
    code + ordered + val + byte_ordered + text_len

Executable fixture: tests/fixtures/valid/run_char_byte.lby (returns 321).

Enums

An enum NAME declaration, followed by indented Variant type... lines, declares a nominal tagged-union type at the top level. Each variant is a name plus zero or more positional, unnamed payload types. Construction reuses existing spelling with no new syntax: a payload variant reads like a call (Circle(2.0)), and a unit variant reads like a bare name (Empty). Variant names are globally unique across all enums.

enum Shape
    Circle f64
    Rect f64 f64
    Empty

fn sides s Shape -> i64
    4

fn main -> i64
    let c Shape = Circle(2.0)
    let r Shape = Rect(3.0, 4.0)
    let e Shape = Empty
    sides(c) + sides(r) + sides(e)

Executable fixture: tests/fixtures/valid/docs_enum.lby.

  • Invalid declarations (duplicate enum, duplicate variant, or empty enum) report L0380.
  • Construction arity or payload-type mismatches report L0381.
  • A variant name colliding across enums reports L0382.
  • Enums construct and pass identically across the AST, IR, and bytecode backends. Consume them with match; enum methods and generics over enums are still planned.

Pattern Matching

match SCRUTINEE selects on an enum value with an indented arm list. Each arm is a pattern, then ->, then an inline expression or an indented block whose last expression is the arm's value. A Variant(bind1, bind2, ...) pattern binds the variant's positional payloads to arm-scoped locals (typed by the variant's declared payload types); a bare Variant matches a unit variant; _ is a wildcard. Like if/try, a match is an expression: when every arm yields the same type it produces that value, so it can be a function's final expression.

enum Shape
    Circle i64
    Rect i64 i64
    Empty

fn area s Shape -> i64
    match s
        Circle(r) -> r * r
        Rect(w, h) -> w * h
        Empty -> 0

fn main -> i64
    area(Rect(4, 5))

Executable fixture: tests/fixtures/valid/docs_match.lby.

  • Matching must be exhaustive: every variant covered, or a _ wildcard arm present. A non-exhaustive match reports L0384.
  • The scrutinee must be an enum type (L0383); an unknown or duplicate variant arm, or a wrong payload binding count, reports L0385.
  • match runs identically across the AST, IR, and bytecode backends, including under optimization.

option and result

option<T> and result<T, E> are two built-in generic enums for absence and fallible results. option<T> has variants some(value) and none; result<T, E> has variants ok(value) and err(error). The variant names some/none/ok/err are reserved, and option/result cannot be redeclared as user enums. Construction is context-directed: some(v) synthesizes option<typeof v>, while none, ok(v), and err(v) take their type from a let annotation or a function return type.

fn unwrap_or o option<i64> fallback i64 -> i64
    match o
        some(v) -> v
        none -> fallback

fn divide n i64 d i64 -> result<i64, string>
    if d == 0
        return err("divide by zero")
    ok(n / d)

fn main -> i64
    let value option<i64> = some(41)
    let quotient result<i64, string> = divide(84, 2)
    match quotient
        ok(q) -> unwrap_or(value, 0) + q
        err(m) -> len(m)

Executable fixture: tests/fixtures/valid/docs_option_result.lby.

  • Type spelling reuses the generic form with a comma-separated argument list: option<i64>, result<i64, string>, option<array<i64>>.
  • Building none/ok/err takes its type from context — a let annotation, a function return type, a match scrutinee, or a call argument's parameter type (so describe(ok(5)) and f(none) infer without a separate annotated let). Without any such context it reports L0386; a payload whose type disagrees with the annotation reports L0303.
  • match over an option<U> covers some + none (or _); over a result<T, E> it covers ok + err (or _), reusing L0384/L0385.
  • option/result construct and match identically across the AST, IR, and bytecode backends, including under optimization.

Growable lists (list<T>)

fn build -> list<i64>
    let l list<i64> = list_new()
    l = push(l, 10)
    l = push(l, 20)
    l = push(l, 30)
    l

fn main -> i64
    let l list<i64> = build()
    l = set(l, 1, 25)
    let a i64 = get(l, 0)
    let b i64 = get(l, 2)
    let n i64 = len(l)
    l = pop(l)
    let m i64 = len(l)
    let last i64 = get(l, 1)
    a + b + n + m + last

Executable fixture: tests/fixtures/valid/run_list.lby (prints 70).

  • list<T> is a growable, value-semantic list. Its "mutating" builtins are functional and return a new list, so the idiom is l = push(l, x).
  • list_new() builds an empty list and takes its element type from the expected list<...> type (a let annotation, a function return type, or a call argument's parameter type — so push(list_new(), byte(65)) inside a let list<byte> infers list<byte>); missing that context reports L0387.
  • push(l, x) appends, get(l, i) reads (bounds-checked, L0413), set(l, i, x) replaces at an index, pop(l) drops the last element, and len(l) returns the i64 length. Wrong argument types or arities report L0387.

Hash maps (map<K, V>)

fn build -> map<string, i64>
    let m map<string, i64> = map_new()
    m = map_set(m, "a", 10)
    m = map_set(m, "b", 20)
    m = map_set(m, "a", 15)
    m

fn lookup m map<string, i64> k string -> i64
    match map_get(m, k)
        some(v) -> v
        none -> 0

fn flag_score m map<string, i64> k string -> i64
    if map_has(m, k)
        return 100
    0

fn main -> i64
    let m map<string, i64> = build()
    let n0 i64 = map_len(m)
    let a i64 = lookup(m, "a")
    let b i64 = lookup(m, "b")
    let missing i64 = lookup(m, "z")
    let has_a i64 = flag_score(m, "a")
    let m2 map<string, i64> = map_del(m, "b")
    let n1 i64 = map_len(m2)
    let b2 i64 = lookup(m2, "b")
    n0 + a + b + missing + has_a + n1 + b2

Executable fixture: tests/fixtures/valid/run_map.lby (prints 138).

  • map<K, V> is an insertion-ordered, value-semantic hash map with keys restricted to i64 or string. Its "mutating" builtins are functional and return a new map, so the idiom is m = map_set(m, k, v).
  • map_new() builds an empty map and takes its key/value types from the expected map<...> type (a let annotation, a function return type, or a call argument's parameter type — so map_set(map_new(), "x", 7) inside a let map<string, i64> infers map<string, i64>); missing that context, an unsupported key type, or a wrong argument reports L0388.
  • map_get(m, k) returns option<V> (some(v) or none), map_has(m, k) returns a bool, map_len(m) returns the i64 entry count, and map_del(m, k) removes a key. Reads are unwrapped with match.

Traits

A trait names a set of methods a type can implement. A trait declaration lists method signatures fn method self [param Type ...] -> Ret with no body; the receiver is self, and Self in a later parameter or return type means the implementing type. An impl Trait for Type block provides a body for every method the trait requires. A trait method is called with the usual UFCS spelling — p.show() desugars to show(p) and dispatches on the receiver's concrete runtime type. A generic function bounds a type parameter with <T: Trait> (or <T: A + B>): inside the function a value of type T may call the bound trait's methods, and at a call site the inferred concrete type must implement every bound. Dispatch uses one shared impl table consulted identically by the AST, IR, and bytecode backends.

# Traits: declaration, impls for a struct and an enum, receiver-type method
# dispatch, and a bounded generic that calls a trait method.

trait Show
    fn show self -> string

struct Point
    x i64
    y i64

enum Light
    Red
    Green

impl Show for Point
    fn show self -> string
        "(" + to_string(self.x) + ", " + to_string(self.y) + ")"

impl Show for Light
    fn show self -> string
        match self
            Red -> "red"
            Green -> "green"

fn describe<T: Show> v T -> string
    "show=" + v.show()

fn main -> i64
    let p Point = Point(3, 4)
    let g Light = Green
    # Direct trait-method calls dispatch by the receiver's concrete type.
    let a i64 = len(p.show())
    let b i64 = len(g.show())
    # A bounded generic resolves `v.show()` on the concrete runtime type.
    let c i64 = len(describe(p))
    let d i64 = len(describe(g))
    a + b + c + d

Executable fixture: tests/fixtures/valid/run_traits.lby. Returns 32.

  • An impl that does not match its trait (missing, extra, or mismatched method, or a trait-method/free-function name clash) reports L0398; a duplicate impl Trait for Type reports L0399; a type used where a bound requires a trait it does not implement reports L0400.
  • Default method bodies, trait objects (dyn), associated types, and supertraits are still planned.

Concurrency

parallel_map(f fn(i64) -> i64, args list<i64>) -> list<i64> evaluates f(arg) for every element of args concurrently on separate OS threads and returns the results in the same order as args, so the output is deterministic regardless of thread scheduling. Each thread builds a fresh interpreter over the shared program, so heaps are per-thread with no shared mutable state and no locking; f must be an ordinary top-level fn(i64) -> i64. It runs identically on the AST, IR, and bytecode backends. Wrong arity, a non-fn(i64) -> i64 first argument, or a non-list<i64> second argument report L0334.

fn sq x i64 -> i64
    x * x

fn main -> i64
    let base list<i64> = list_new()
    base = push(base, 1)
    base = push(base, 2)
    base = push(base, 3)
    base = push(base, 4)
    let out list<i64> = parallel_map(sq, base)
    let total i64 = 0
    for i from 0 to 3
        total += get(out, i)
    total

Executable fixture: tests/fixtures/valid/run_parallel.lby. Returns 30.

Threads, channels, and a mutex

Message passing with detached threads is delivered on all three backends. chan_new() -> Chan creates an unbounded i64 channel (a shared handle: cloning shares the same queue); send(ch Chan, v i64) enqueues without blocking, recv(ch Chan) -> i64 dequeues blocking, and try_recv(ch Chan) -> option<i64> dequeues non-blocking. spawn(f fn(Chan, i64) -> void, ch Chan, v i64) -> Task runs f(ch, v) on a detached OS thread that owns a share of the program (an Arc clone) and returns a Task; task_join(t Task) -> void waits for it (named task_join, not join, because join is the string-list joiner). mutex_new(v i64) -> Mutex creates a shared mutex over one i64; mutex_get/mutex_set/mutex_add read, write, and atomically add-and-return. Because scheduling is non-deterministic, sum received values instead of asserting order. Wrong arity or a wrong-typed argument reports L0337. Generic channels, select, async/await, and cross-thread socket sharing remain deferred.

fn worker ch Chan v i64 -> void
    send(ch, v * v)

fn main -> i64
    let ch Chan = chan_new()
    let t1 Task = spawn(worker, ch, 1)
    let t2 Task = spawn(worker, ch, 2)
    let t3 Task = spawn(worker, ch, 3)
    let t4 Task = spawn(worker, ch, 4)
    task_join(t1)
    task_join(t2)
    task_join(t3)
    task_join(t4)
    let total i64 = 0
    for i from 0 to 3
        total += recv(ch)
    let m Mutex = mutex_new(0)
    for j from 0 to 3
        mutex_add(m, 1)
    total + mutex_get(m)

Executable fixture: tests/fixtures/valid/run_spawn.lby. Returns 34 (1+4+9+16 received, plus a mutex count of 4).

Networking

A Socket is an opaque handle to an OS network resource (a TCP listener, TCP stream, or UDP socket), surfaced as an integer handle backed by a per-runtime table (like a heap pointer). Every fallible socket builtin returns a result<T, string>, so network failures are ordinary runtime values matched with match (an err(message) on failure), never panics. Sockets run identically on the AST, IR, and bytecode backends.

BuiltinResult
tcp_connect(host string, port i64)result<Socket, string> — open a client stream.
tcp_listen(host string, port i64)result<Socket, string> — bind and listen.
tcp_accept(listener Socket)result<Socket, string> — block for a connection, return a stream.
tcp_read(conn Socket)result<string, string> — read up to 4096 bytes (empty on clean EOF).
tcp_write(conn Socket, data string)result<i64, string> — write the full byte buffer, flush, return the count.
tcp_shutdown(conn Socket)void — gracefully shut down the write half (signal EOF) so a buffered response is delivered before the socket is dropped.
tcp_close(conn Socket)void — free the handle.
udp_bind(host string, port i64)result<Socket, string> — bind a datagram socket.
udp_send_to(sock Socket, data string, host string, port i64)result<i64, string> — send one datagram, return the count.
udp_recv(sock Socket)result<string, string> — receive one datagram (sender addr dropped).

Wrong argument types or arities report L0335.

fn main -> i64
    let outcome result<Socket, string> = tcp_connect("127.0.0.1", 1)
    match outcome
        ok(sock) -> 0
        err(message) -> 1

HTTP/1.1 client

A minimal HTTP/1.1 client built over TcpStream (std only, no TLS). Each builtin takes a URL (with the http scheme) and returns the response body on a 2xx/3xx response, or an err(message) on a connection/parse/HTTP error. Only the http scheme is supported — an HTTPS URL returns err("https not supported"); chunked transfer decoding is not implemented (responses are read to EOF via Connection: close); a 4xx/5xx status returns err("http {code}: {first-body-line}"); a 10-second read timeout surfaces a hung server as err. The executable parity fixture tests/fixtures/valid/run_http.lby calls http_get against a closed loopback port, so it deterministically takes the err arm and returns 1 on the AST, IR, and bytecode backends without any external server.

BuiltinResult
http_get(url string)result<string, string> — HTTP/1.1 GET, returns the response body.
http_post(url string, body string)result<string, string> — HTTP/1.1 POST (text/plain body), returns the response body.

Wrong argument types or arities report L0336.

Writing an HTTP server in Lullaby

A full HTTP/1.1 server can be written in pure Lullaby on the TCP socket builtins — request-line parsing (split), path routing, and response building are ordinary pub functions. Each connection is served with the graceful teardown order tcp_write then tcp_shutdown then tcp_close, so the buffered response is delivered (EOF signalled on the write half) before the socket is dropped and an ordinary client never sees an empty reply. The worked example lives at examples/valid/http_server/ (server.lby plus a reusable http.lby module); run it with lullaby run server.lby 8080 5 to serve five requests on port 8080.

A full-stack web demo sharing one module

The same domain module can drive both the browser and the server. examples/valid/fullstack/ keeps a shared.lby module inside the WASM-eligible surface (scalars and strings only), then compiles it two ways: frontend.lby lowers to WebAssembly with lullaby wasm frontend.lby and renders shared task-priority labels through the console_log/dom_set_text host imports (a self-contained index.html supplies env.console_log/env.dom_set_text and loads frontend.wasm relatively), while backend.lby is a pure-Lullaby HTTP server that answers /classify with the identical shared label and score. Both sides run the same classify/priority_score logic with no duplication.

Boolean Logic

The alpha supports boolean logic with and, or, and unary not. Logical operands must be bool, and and/or short-circuit during runtime execution.

fn main -> bool
    not false and true or false

Testing

assert(cond bool) -> void is the assertion primitive. When cond is false it raises a catchable runtime error with the message assertion failed — the same error path a user throw uses, so try/catch recovers it; when cond is true it returns void. A non-bool argument reports L0342.

lullaby test path/to/tests.lby runs a language-level test suite. It validates the file as a library (no main required), then runs every top-level function whose name starts with test_, takes zero parameters, and returns void/i64/bool through the interpreter. A test passes if it returns without a runtime error and fails if it produces one (an assert(false), a throw, or any other runtime error). The runner prints PASS test_name or FAIL test_name: <message> per test, then a N passed, M failed summary, and exits non-zero when any test fails. A runnable demo suite ships at examples/valid/tests_demo/tests_demo.lby.

Math Builtins

The alpha includes a small, type-directed math library over i64 and f64. abs(x) returns the magnitude (i64->i64 or f64->f64). min(a, b) and max(a, b) take two matching numeric operands and return the smaller or larger. pow(base, exp) raises to a power; the integer form requires exp >= 0 (a negative integer exponent is runtime L0417). sqrt, floor, ceil, and round take one f64 and return f64. The transcendental builtins also take and return f64: sin, cos, tan, atan, exp, ln, and log10 are unary, and atan2(y, x) takes two f64s and returns the angle in radians. A wrong or mismatched operand type reports L0374.

fn main -> i64
    let magnitude i64 = abs(0 - 7)
    let smaller i64 = min(magnitude, 4)
    let raised i64 = pow(smaller, 3)
    let root f64 = sqrt(64.0)
    if root == 8.0
        raised
    else
        0

Modules

Each .lby file is a module. A top-level fn, struct, enum, or alias is private to its file unless you prefix it with pub, which exports it. A file pulls in another module with one or more leading import NAME lines placed before any declaration; import NAME loads NAME.lby from the same directory as the entry file and makes that module's pub items available unqualified.

A frontend-only loader lexes, parses, resolves imports (recursively), enforces visibility, and merges every module into one program before type checking, so the type checker and all three backends run unchanged. The merged namespace is flat with no shadowing.

# geometry.lby
pub struct Point
    x i64
    y i64

pub fn dot a Point b Point -> i64
    a.x * b.x + a.y * b.y

fn priv_helper n i64 -> i64      # private to geometry.lby
    n * n
# main.lby
import geometry

fn main -> i64
    let p Point = Point(3, 4)
    dot(p, p)                    # uses pub Point and dot; returns 25

Running lullaby run main.lby loads both files and prints 25. Using priv_helper from main.lby would report L0392, because it is private to geometry.lby. A name declared in two modules reports L0391, an import cycle reports L0393, and a missing module file reports L0397. Dotted module paths, qualified access, and per-field visibility are not yet implemented.

Projects

A program can grow beyond a single directory into a project: a directory with a lullaby.json manifest that names the project, an optional executable entry (a .lby file), one or more src source directories (default ["."]), and dependencies mapping a name to another local project root that has its own lullaby.json. In a project build, import NAME resolves NAME.lby by searching the importing file's directory, then every src directory of the project, then every src directory of each dependency; the same visibility and no-shadowing rules apply across the whole set. The commands run, check, build, test, compile, wasm, and native accept either a single .lby file (unchanged) or a project directory / lullaby.json path. run/build/compile/wasm/native need an entry; check/test can validate a library project without one. A missing or malformed manifest, a missing src directory, or a missing dependency reports L0343. Dependencies are local paths only; remote/registry fetching is deferred. The worked project lives at examples/valid/project_demo/ (with its mathx dependency); lullaby run examples/valid/project_demo prints 45.

Current Limitations

This page documents the current executable alpha. Planned syntax and systems features in the broader Markdown specs may not be accepted by the compiler yet.

  • No native code generation yet. Execution currently supports AST, typed IR, an instruction-bytecode backend, and versioned .lbc bytecode artifacts. The optimizer currently exposes opt-in constant folding, conservative common subexpression elimination, conservative loop-invariant motion, conservative block-local copy propagation, block-local dead-code elimination, and the combined alpha pipeline.
  • Typed IR now reports Alpha 1 memory operations and bounds-checked indexing with sequence and safety metadata for backend work, and version 5 .lbc artifacts preserve that metadata in memory_operations. The full region memory model, ARC/reference counting, cleanup insertion, and lifetime analysis are pending.
  • The type system currently covers the alpha scalar, array, call, local binding, assignment, control-flow, and interim pointer subset.
  • The current I/O surface is limited to text/binary file builtins, direct program-plus-argv command calls, and basic TCP/UDP sockets; buffered streams, memory mapping, async, and IPC are future work.
  • Modules, generics beyond array element spelling, structs, packages, and a full installer are future work. The current Alpha 1 distribution path is a Windows portable archive with optional user PATH helper scripts; the cross-platform package driver also prepares Unix PATH helpers for future Linux/macOS archives.
  • Planned syntax keywords such as import, module, struct, match, and try are rejected with L0211 [parser] until their feature areas are implemented.
  • Inline assembly (asm b0, b1, ..., a list of raw 0..=255 machine-code bytes inside an unsafe block) is emitted verbatim into the current function's .text by the native backend only. It is inherently unsafe: the bytes are trusted and not decoded, relocated, or register-modeled, and a trailing asm that leaves a value in rax becomes the function's return value. The AST, IR, and bytecode interpreters cannot execute raw machine code, so any asm statement is rejected with L0425; compile with lullaby native to emit and link it. Out-of-range bytes or use outside unsafe are caught at check time (L0425 / L0330).
  • The native backend (lullaby native) compiles the i64-scalar subset to an x86-64 Windows COFF object and, best-effort, links it into a runnable .exe. Adding --freestanding (alias --no-std) builds a no-C-runtime executable: it links kernel32.lib only (zero ucrt/vcruntime/msvcrt) and terminates through the minimal OS import kernel32!ExitProcess, bypassing the C runtime startup. It is still a Windows PE, not a bare-metal binary; true bare-metal (raw-syscall exit with no OS imports, other object formats, a custom entry and linker script) is deferred. A freestanding build that declares an extern fn (which needs the C runtime) is rejected with L0426.
  • The checked-in documentation bundle is currently hand-authored HTML. The standard-library generator from canonical docs is available at offline_docs/generate_offline_docs.py, emits package-ready quick start, CLI reference, package layout, diagnostics, limitations, shipped-doc parity sections, and fixture-backed examples, and is now used by the Windows portable package path.
