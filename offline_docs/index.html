Lullaby

Offline documentation for the current Lullaby alpha: a compiled systems language prototype with concise indentation-only syntax, strong typing, and an executable Rust-based frontend/runtime.

Overview

Lullaby is implemented in Rust. The current alpha focuses on a clear, testable compiler frontend plus AST, IR, and initial bytecode execution paths before native code generation.

The Alpha 1 language surface is frozen in documents/alpha1_language_surface.md. Broader Markdown design docs are planned material unless the feature appears in that file or on this page.

Extension.lby is the canonical source file extension.
BlocksIndentation defines scope. Curly braces are rejected.
StatementsSemicolons are not statement terminators.
Runtimelullaby run executes the supported subset with AST, IR, or bytecode backends.

Quick Start

Create a file named hello.lby:

fn main -> i64
    42

Executable fixture: tests/fixtures/valid/docs_quick_start.lby.

Check it:

cargo run -p lullaby_cli -- check hello.lby

Run it:

cargo run -p lullaby_cli -- run hello.lby

Run it through the typed IR backend:

cargo run -p lullaby_cli -- run --backend ir hello.lby

Run it through the typed IR backend with the opt-in constant-folding optimizer:

cargo run -p lullaby_cli -- run --backend ir --optimize constant-fold hello.lby

Run it through the current alpha optimizer pipeline:

cargo run -p lullaby_cli -- run --backend ir --optimize alpha hello.lby

Compile it to a bytecode artifact with metadata, a function table, and memory operation metadata; inspect the artifact, then run it:

cargo run -p lullaby_cli -- compile --optimize alpha -o hello.lbc hello.lby
cargo run -p lullaby_cli -- build --optimize alpha -o hello-build.lbc hello.lby
cargo run -p lullaby_cli -- inspect hello.lbc
cargo run -p lullaby_cli -- run hello.lbc

During development, the CLI is available through Cargo. The Alpha 1 portable package builds a direct bin\lullaby.exe command, bundles docs\index.html, includes install.cmd and uninstall.cmd for optional user PATH setup, ships user-facing examples such as examples\valid\calculator.lby, emits lullaby-alpha1-windows-x64.zip.sha256, and supports lullaby docs and lullaby examples to locate local resources. The cross-platform package driver also prepares install.sh and uninstall.sh for Linux/macOS portable archives after those targets are validated.

Source Files

  • Use the .lby extension.
  • Use spaces for indentation. A deeper indent opens a block; dedenting closes it.
  • Do not use { or } as block delimiters.
  • Do not end statements with semicolons.
  • Comments begin with # and run to the end of the line.

Functions

Functions start with fn, followed by the name, typed parameters, an optional return arrow, and an indented body.

fn add x i64 y i64 -> i64
    x + y

Executable fixture: tests/fixtures/valid/add.lby.

Last-expression return

When a function body reaches its final expression normally, that expression is the return value.

fn choose flag bool -> i64
    if flag
        10
    else
        20

Explicit return

fn early value i64 -> i64
    if value > 10
        return value
    10

Executable fixture: tests/fixtures/valid/docs_early_return.lby.

Void functions

A function that returns nothing declares -> void. It may use an empty return, or simply finish with no value-producing final expression.

fn release ptr ptr_i64 -> void
    dealloc(ptr)
    return

Variables And Assignment

Use let to bind a local name with an explicit type and initializer, or omit the type when the initializer has a concrete inferred local type.

fn main -> i64
    let value i64 = 40
    value + 2

Executable fixture: tests/fixtures/valid/docs_let_binding.lby.

fn add x i64 y i64 -> i64
    x + y

fn main -> i64
    let value = add(40, 2)
    let values = [value, 1]
    values[0]

Executable fixture: tests/fixtures/valid/run_inferred_let.lby.

Existing locals can be updated with regular assignment or numeric compound assignment.

fn count -> i64
    let total i64 = 0
    total += 1
    total *= 42
    total

Arrays

The alpha supports homogeneous arrays with array<T> type spelling, non-empty array literals, and bounds-checked indexing.

fn main -> i64
    let values array<i64> = [2, 4, 6]
    values[2]

Executable fixture: tests/fixtures/valid/run_array.lby.

  • Array literal values must all have the same static type.
  • Empty array literals are not accepted yet because empty-array inference is not implemented.
  • Index expressions require an i64 index.
  • Out-of-bounds indexes are runtime errors.

Array elements are mutable through assignment, including compound operators, and len(x) returns the i64 length of an array or string.

fn main -> i64
    let xs array<i64> = [1, 2, 3]
    xs[0] = 10
    xs[len(xs) - 1] += 4
    xs[0] + xs[2]

Executable fixture: tests/fixtures/valid/docs_array_mutation.lby.

While loops

fn sum_to limit i64 -> i64
    let total i64 = 0
    let index i64 = 0
    while index < limit
        total += index
        index += 1
    total

Range for loops

for loops iterate over inclusive i64 ranges. The default step is 1; by supplies an explicit step, and runtime execution rejects step 0.

fn main -> i64
    let total i64 = 0
    for i from 1 to 3
        total += i
    total

    let total i64 = 0
    for i from 5 to 1 by -2

    total

Loop, break, continue

fn first_even_after start i64 -> i64
    let value i64 = start
    loop
        value += 1
        if value == 3
            continue
        if value == 4
            break
    value

Boolean Logic

The alpha supports boolean logic with and, or, and unary not. Logical operands must be bool, and and/or short-circuit during runtime execution.

fn main -> bool
    not false and true or false

Executable fixture: tests/fixtures/valid/run_logic.lby.

Logical expressions can be used anywhere a bool is required, including branch and loop conditions.

fn ready value i64 enabled bool -> bool
    enabled and value > 0

Executable fixture: tests/fixtures/valid/docs_ready_logic.lby.

Math Builtins

The alpha includes a small, type-directed math library over i64 and f64. abs(x) returns the magnitude (i64->i64 or f64->f64). min(a, b) and max(a, b) take two matching numeric operands and return the smaller or larger. pow(base, exp) raises to a power; the integer form requires exp >= 0 (a negative integer exponent is runtime L0417). sqrt, floor, ceil, and round take one f64 and return f64. A wrong or mismatched operand type reports L0374.

fn main -> i64
    let magnitude i64 = abs(0 - 7)
    let smaller i64 = min(magnitude, 4)
    let raised i64 = pow(smaller, 3)
    let root f64 = sqrt(64.0)
    if root == 8.0
        raised
    else
        0

Executable fixture: tests/fixtures/valid/docs_math.lby.

Memory Builtins

The alpha runtime includes heap-slot builtins. These are an interim executable model while the full region/ARC memory design is still being implemented.

BuiltinCurrent behavior
alloc(value)Stores a value in a runtime heap slot and returns an interim pointer such as ptr_i64.
load(ptr)Loads the value from a valid pointer.
store(ptr, value)Replaces the value in a valid pointer slot. The stored value must match the pointer element type.
dealloc(ptr)Clears the heap slot. Invalid or double deallocation is a runtime error.
 
fn main -> i64
    let ptr ptr_i64 = alloc(0)
    store(ptr, 41)
    let value i64 = load(ptr)
    dealloc(ptr)
    value + 1

Executable fixture: tests/fixtures/valid/run_store.lby.

I/O And System Builtins

The alpha runtime includes flat text file I/O builtins and a conservative system command abstraction. Dotted io.* APIs, streams, binary I/O, memory mapping, async, sockets, and IPC are planned rather than current syntax.

BuiltinCurrent behavior
read_file(path)Reads a UTF-8 text file into a string.
write_file(path, content)Writes text to a file, replacing existing contents.
append_file(path, content)Appends text to a file, creating it if needed.
file_exists(path)Returns whether host metadata for the path can be read.
sys_status(program, args)Runs a program directly with an array<string> argv and returns its exit status.
sys_output(program, args)Runs a program directly with an array<string> argv and returns stdout.

String Builtins

The alpha runtime includes a small character-indexed string library. All indices are character indices (like len on a string), not byte offsets. Wrong argument types report L0375.

BuiltinCurrent behavior
substring(s, start, end)Returns the half-open [start, end) character slice; out-of-range or negative indices are a runtime error.
find(s, needle)Returns the first character index of needle as an i64, or -1 when absent.
contains(s, needle)Returns a bool for whether needle occurs in s.
split(s, sep)Splits on a non-empty separator into array<string>; an empty separator is a runtime error.
join(parts, sep)Joins an array<string> with sep into a string.
trim(s)Removes leading and trailing ASCII whitespace.
replace(s, from, to)Replaces every occurrence of from; an empty from is a runtime error.
upper(s) / lower(s)Returns an upper- or lower-cased copy of s.
 
fn main -> string
    let parts array<string> = split("a,b,c", ",")
    upper(join(parts, "-"))

Diagnostics

Diagnostics use stable short codes and are available in concise, verbose, and JSON output modes. The default concise mode is compact terminal feedback. The --verbose mode adds source excerpts, caret markers, root cause, suggested fix text, related notes, and runtime traceback frames when available. The --format json mode emits deterministic machine-readable diagnostics.

L0102 [lexer error] at tests/fixtures/invalid/brace.lby:2:5: curly braces are not block delimiters in Lullaby

Source:
   2 |     {
     |     ^

Root cause:
  The source contains a curly brace, which is not a block delimiter.

Suggested fix:
  Remove the brace and express the block with indentation.
{"status":"error","diagnostics":[{"code":"L0313","phase":"semantic","severity":"error","span":{"line":2,"column":24},"root_cause":"The argument expression type does not match the parameter type.","suggested_fix":"Pass a value of the expected type or change the called function signature.","traceback":[]}]}

The diagnostic registry in documents/diagnostic_registry.md defines every stable L#### code, phase, meaning, likely cause, and suggested fix.

Current Limitations

This page documents the current executable alpha. Planned syntax and systems features in the broader Markdown specs may not be accepted by the compiler yet.

  • No native code generation yet. Execution currently supports AST, typed IR, an instruction-bytecode backend, and versioned .lbc bytecode artifacts. The optimizer currently exposes opt-in constant folding, conservative common subexpression elimination, conservative loop-invariant motion, conservative block-local copy propagation, block-local dead-code elimination, and the combined alpha pipeline.
  • Typed IR now reports Alpha 1 memory operations and bounds-checked indexing with sequence and safety metadata for backend work, and version 5 .lbc artifacts preserve that metadata in memory_operations. The full region memory model, ARC/reference counting, cleanup insertion, and lifetime analysis are pending.
  • The type system currently covers the alpha scalar, array, call, local binding, assignment, control-flow, and interim pointer subset.
  • The current I/O surface is limited to text file builtins and direct program-plus-argv command calls; streams, binary I/O, memory mapping, async, sockets, and IPC are future work.
  • Modules, generics beyond array element spelling, structs, packages, and a full installer are future work. The current Alpha 1 distribution path is a Windows portable archive with optional user PATH helper scripts; the cross-platform package driver also prepares Unix PATH helpers for future Linux/macOS archives.
  • Planned syntax keywords such as import, module, struct, match, and try are rejected with L0211 [parser] until their feature areas are implemented.
  • The checked-in documentation bundle is currently hand-authored HTML. The standard-library generator from canonical docs is available at offline_docs/generate_offline_docs.py, emits package-ready quick start, CLI reference, package layout, diagnostics, limitations, shipped-doc parity sections, and fixture-backed examples, and is now used by the Windows portable package path.
