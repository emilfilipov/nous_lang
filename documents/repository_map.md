# Repository Map

This file maps the repository layout and explains where to find core information. Keep it up to date whenever source files, directories, commands, tests, docs, or responsibilities change.

## Root

- [CLAUDE.md](../CLAUDE.md): operating guide for agents and contributors. Defines workflow, documentation rules, MCP/ClickUp/GitHub usage, testing expectations, Git rules, and where to find core language information.
- [README.md](../README.md): user-facing project overview, portable package installation, checksum verification, local build commands, CLI summary, and documentation links.
- `Cargo.toml`: Rust workspace manifest for the Lullaby compiler/toolchain crates. The lockfile records the pinned dependency graph, including serde/serde_json for bytecode artifact serialization.
- `.gitignore`: ignores local build outputs, caches, editor state, and generated artifacts once implementation begins.
- `.gitattributes`: normalizes repository text files to LF line endings.
- `crates/`: Rust implementation crates.
- `examples/`: user-facing `.lby` examples packaged with the toolchain.
- `scripts/`: release packaging and verification scripts.
- `tests/`: shared `.lby` fixtures used by crate and CLI tests.
- `documents/`: core language documents and planning material.
- `offline_docs/`: self-contained browser documentation bundle that can be opened directly from disk.

- `documents/modules_design.md`: modules/imports/visibility design — file-as-module, `pub` to export, flat unqualified `import NAME`, a frontend-only loader that merges all files into one `Program` (backends unchanged), no-shadowing rule, and `L0391`/`L0392`/`L0393`; plus the `lullaby.json` project manifest (multiple `src` directories and local-path dependencies, resolved across the project and its dependencies, `L0343`). Dotted paths, qualified access, per-field visibility, and remote/registry dependency fetching deferred.

- `documents/generics_design.md`: user-defined generics design — generic functions `fn name<T> ...` with call-site type inference, erased at runtime (backends unchanged), inference mirrored in the IR lowerer, and `L0394`/`L0395`/`L0396`; generic user types and explicit type arguments deferred (trait bounds are now implemented — see [traits_design.md](traits_design.md)).
- `documents/traits_design.md`: traits/interfaces design — `trait Name` method signatures with a `self` receiver, `impl Trait for Type`, receiver-type method dispatch via a shared impl table on all backends, and trait bounds on generics (`<T: Ord>`); records `L0398`/`L0399`/`L0400` and defers default bodies, trait objects, associated types, and supertraits.
- `documents/wasm_backend_design.md`: WebAssembly backend design — compile the typed IR's scalar subset (`i64`/`f64`/`bool`/`char`/`byte`, arithmetic/comparison/logic, `if`/`while`/`loop`/`for`, calls, locals) to a binary `.wasm` module with std-only encoding, a `lullaby wasm` CLI command, and node/wasmtime-gated execution parity against the interpreter. The first linear-memory step is landed: the module exports a `"memory"` (min 1 page), declares a mutable `i32` bump-pointer global, seeds a Data section, emits an internal `__alloc(size i32) -> i32` bump helper, and imports the host function `env.log_i64` surfaced to Lullaby as the builtin `wasm_log(x i64) -> void` (imports occupy the low function indices, so internal function indices/call targets/exports are shifted by the import count). **Heap types are now lowered into linear memory:** `string` values are `i32` pointers to `[len i32 char-count][utf8 bytes]` with literals interned once into the Data section (and `len(s)`); `struct` values are pointers to one 8-byte slot per field with positional construction/field read/field mutation; fixed `array` literals are pointers to `[len i32][8-byte elem slots]` with construction, indexed read/write, and `len(a)`; field/element slots may be scalars or nested pointers, and `__alloc` builds structs/arrays at runtime. Enum/tagged-union + `match` lowering, `option`/`result`, `list`/`map`, runtime string construction, a free-list allocator, and JS/DOM interop remain deferred.
- `documents/lsp_design.md`: Language Server Protocol server design — the `lullaby_lsp` crate and `lullaby lsp` command, JSON-RPC-over-stdio `Content-Length` framing, the pure `handle_message` request core, advertised capabilities (`textDocumentSync=1` full, `documentFormattingProvider`, `hoverProvider`, `definitionProvider`), lifecycle (`initialize`/`initialized`/`shutdown`/`exit`), full-text document sync, diagnostics from the reused lex/parse/semantic pipeline mapped to 0-based LSP ranges, canonical-formatter formatting edits, `analysis.rs` hover (function signature / local type / struct-enum declaration / builtin description from the checked metadata) and go-to-definition (declaration or `let`/parameter binding `Location`), testing strategy, and deferred features (completion/references, incremental sync, project-aware analysis).
- `documents/lullaby_control_structures.md`: control flow and operators, including the current alpha if/while/loop rules plus planned switches, try/catch, coroutines, arithmetic/logical/bitwise operators, collections, conversions, and utility operations.
- `documents/lullaby_input_output.md`: I/O and concurrency model, including files, streams, memory-mapped files, threads, processes, async, multiplexing, IPC, sockets, and performance strategies.
- `documents/lullaby_error_handling.md`: error model, compact error tokens, compile-time and runtime categories, throw/catch/recovery behavior, diagnostics, and compiler integration.
- `documents/lullaby_compilation_architecture.md`: compiler architecture from tokenization through semantic analysis, IR, optimization, code generation, linking, and binary verification.

## Source Layout

The implementation is a Rust workspace. Unless changed by an explicit architecture decision, keep this layout:

- `crates/lullaby_lexer/`: source extension validation, tokenization, indentation scanning, string- and char-literal lexing (single-quoted `'c'` char literals, with malformed-literal diagnostic `L0105`), numeric literals with `_` digit separators (scanned into the number token; the parser validates placement), forbidden brace/semicolon diagnostics, current and planned keyword recognition (including the `import` and `pub` module keywords), and lexical tests.
- `crates/lullaby_diagnostics/`: shared diagnostic data structures, serializable span/traceback metadata, registry metadata, concise/verbose renderers, and deterministic JSON rendering.
- `crates/lullaby_lsp/`: additive Language Server Protocol (LSP) server for `.lby` files, standard-library + `serde_json` only (no third-party LSP/async framework). `src/transport.rs` implements JSON-RPC 2.0 over stdio with `Content-Length: N\r\n\r\n<json>` framing and the read/write loop (`serve`/`run_stdio`); `src/lib.rs` holds the pure `handle_message(state, method, id, params) -> Vec<Message>` request core (so the protocol is testable without stdio), the `ServerState` in-memory document map, `initialize` (advertising `textDocumentSync=1` full, `documentFormattingProvider=true`, `hoverProvider=true`, `definitionProvider=true`), `initialized`/`shutdown`/`exit`, and full-text `didOpen`/`didChange`/`didClose`; `src/diagnostics.rs` runs the existing lex -> parse -> `validate` pipeline over a document and converts each Lullaby diagnostic (code/message/1-based span) into an LSP `Diagnostic` with a 0-based `Range` (widened to the token at the span, else one char) for `textDocument/publishDiagnostics`, and formats via `format_program` for `textDocument/formatting` (one full-document `TextEdit`, or no edits when the document does not parse); `src/analysis.rs` resolves the identifier under the cursor for `textDocument/hover` (function signature, local/parameter type from the checker's `expression_types`, struct/enum declaration, or builtin description) and `textDocument/definition` (a `Location` for the declaration or `let`/parameter binding), reusing the parser plus `validate` metadata with no new semantics accessor. Unit tests drive `handle_message` directly for the initialize/diagnostics/formatting/hover/definition cases and the framed stdio loop end to end. Run over stdio with `lullaby lsp`. Design: [lsp_design.md](lsp_design.md).
- `crates/lullaby_parser/`: AST model and parser for function declarations, typed parameters, return types, indentation blocks, explicit and inferred `let`, assignment, `return`, `break`, `continue`, if/elif/else, while/loop/range-for, calls, integer and `f64` float literals (with optional `_` digit separators validated and stripped via `normalize_number_literal`), `'c'` char literals, array literals/indexing, struct declarations/construction/field access, enum (tagged-union) declarations with positional payload types, `match` expressions over enums with variant/wildcard patterns and payload bindings, variables, arithmetic, comparison, logical expressions, leading `import NAME` module lines and an optional `pub` export modifier on top-level `fn`/`struct`/`enum`/`alias` declarations, an optional `<T, U>` or bounded `<T: Trait + Other>` generic type-parameter list after a function name with duplicate/built-in-shadowing rejection (`L0394`), `trait NAME` declarations (indented `fn method self ... -> Ret` signatures) and `impl Trait for Type` blocks (indented `fn method self ... -> Ret` bodies, with the untyped `self` receiver injected as the implementing type), reported as `L0216` when malformed (`imports`/`is_public`/`type_params`/`traits`/`impls` are serde-defaulted, and a `TypeParam` serializes as the bare name string when unbounded, so single-file artifacts and AST snapshots stay valid), and `L0211` rejection for planned syntax keywords that are not in Alpha 1. The parser also accepts an `async` modifier on `fn` declarations, the `Future<T>` type, and `await EXPR` expressions. Also hosts `format.rs`, the canonical source formatter (`format_program`) that renders an AST back to idempotent, source-ordered `.lby` source (including `trait`/`impl` blocks and `<T: Bound>` lists) and backs `lullaby fmt`, with per-construct idempotency/round-trip regression tests over representative valid fixtures.
- `crates/lullaby_parser/tests/`: parser integration tests, including checked-in AST golden snapshots under `crates/lullaby_parser/tests/snapshots/`. Refresh snapshots intentionally with `$env:LULLABY_UPDATE_PARSER_SNAPSHOTS='1'; cargo test -p lullaby_parser --test ast_snapshots; Remove-Item Env:LULLABY_UPDATE_PARSER_SNAPSHOTS`.
- `crates/lullaby_semantics/`: static validation for duplicate declarations, explicit and inferred local binding types, assignment targets/types, function call arity/types, return behavior, bool conditions, loop-control placement, arithmetic/comparison/logical expression operand types, homogeneous non-empty arrays, array indexing, nominal struct types with positional construction and field access, nominal enum (tagged-union) types with globally unique variant names and unit/payload construction, interim pointer-style memory builtins, reference-counted `rc<T>`/`ref<T>`/`ptr<T>` types with `rc_new`/`rc_clone`/`rc_release`/`rc_get`/`rc_borrow`/`ref_get` builtins and `unsafe`-gated raw `ptr_read`/`ptr_write`, text file I/O builtins, standard stream builtins (`print`/`println`/`warn`/`flush`), the `wasm_log(x i64) -> void` host-log builtin (prints the value on the interpreters; lowers to the imported `env.log_i64` on the WASM backend), `i64`/`f64` numeric arithmetic, `to_string` conversion and `+` string concatenation, safe system command builtins, user-defined generic functions with call-site type inference (structural unification of arguments against type-variable-containing parameter types, substitution into the return type, conflicting inference `L0395`, uninferable-return-only type variable `L0396`; a bare `T` supports only universal operations, so arithmetic/ordering on it is rejected), traits and impls (a trait table plus an impl table built from `trait`/`impl` declarations, `impl` conformance checking with `Self` substitution `L0398`, duplicate-impl `L0399`, trait-method resolution that dispatches a UFCS call on the receiver type, and trait-bound checking at bounded-generic call sites and on bounded receivers `L0400`, with free-function and trait-method namespaces kept disjoint), `async fn`/`await` typing (an `async fn ... -> T` produces `Future<T>`; `await e` requires `e: Future<T>` and yields `T`; misuse is `L0344`), and executable zero-argument `main` validation for compile/run. Successful validation returns `CheckedProgram` with function signatures (including generic `type_params` and per-parameter trait bounds) and inferred expression-type metadata; the generic unifier (`unify_param`/`substitute_type`/`infer_generic_return`) is public so the IR lowerer reuses it.
- `crates/lullaby_ir/`: typed semantic IR schema, lowering from `CheckedProgram`, Alpha 1 memory-operation analysis with sequence and safety metadata for allocation/load/store/deallocation, bounds-checked indexing, reference-counted `rc`/`ptr` operations, and region creation, deterministic stack-frame layout analysis (`frame_layout`) assigning parameters/locals/loop-variables stable word-aligned slots with per-scope reverse-order cleanup plans, reserved safety semantics for future region/copy/cleanup operations, serializable native backend contract data for Alpha 1 target/layout/calling/cleanup policy, first `x86_64-pc-windows-msvc` COFF object-emission prototype for literal-return, stack-backed `i64` local arithmetic, and straight-line `i64` assignment entry functions, deterministic optimization pass configuration with opt-in constant folding, conservative block-local CSE, conservative loop-invariant motion, conservative block-local copy propagation, block-local dead-code elimination, executable IR interpreter with `throw`/`try`/`catch` structured error handling lowered and executed on the IR and bytecode backends at parity with the AST runtime, `async fn`/`await`/`Future` lowering and execution on the IR and bytecode backends at parity with the AST runtime (`L0344` on misuse), trait-method dispatch via a shared impl table (`impls`/`trait_methods` on `IrModule`/`BytecodeModule`, serde-defaulted and threaded through lowering, optimization, and the bytecode round-trip; a trait-method call selects the impl function by the receiver's runtime type), inline and fixture-driven AST/IR/bytecode parity tests including optimized backend variants, explicit instruction-bytecode lowering, versioned `.lbc` artifact encoding/decoding with metadata/function-table/memory-operation/instruction-contract compatibility checks, checked-in backend memory metadata snapshots, and bytecode VM entry point for the current alpha subset. `src/wasm.rs` is the WebAssembly backend (scalar subset, linear-memory step, and heap types in linear memory): `emit_wasm_module(&IrModule) -> Result<WasmArtifact, WasmError>` lowers every eligible top-level function to a binary `.wasm` module, implementing the WASM binary encoder from scratch with the Rust standard library only (module header, Type/Import/Function/Memory/Global/Export/Code/Data sections in canonical order, LEB128, and the stack-machine opcodes for literals, locals, arithmetic/comparison/logic, structured `block`/`loop`/`if`/`br`/`br_if`, calls, `return`, and typed `i32`/`i64`/`f64` `load`/`store`). Scalars map as before (`i64`/`f64`/`bool`/`char`/`byte`); **heap types are `i32` pointers into linear memory**: a `string` points to `[len i32 char-count][utf8 bytes]` with literals interned once into the Data section (offset math via `RESERVED_BASE`) and `len(s)` lowered to `i32.load` + `i64.extend`; a `struct` points to one 8-byte slot per field with positional construction (a `Call` named for the struct), `.field` read, and `p.field = v` mutation; a fixed `array` literal points to `[len i32][8-byte elem slots]` with construction, `a[i]` read, `a[i] = v` write (WASM traps on OOB memory access), and `len(a)`; field/element slots may be scalars or nested pointers, and `struct`/`array` construction calls `__alloc`. The module still exports `"memory"` (min 1 page), declares a mutable `i32` bump-pointer global (now initialized past the Data-seeded reserved region AND the whole string-literal pool), emits the internal `__alloc(size i32) -> i32` bump helper, and imports the host function `env.log_i64 (func (param i64))` exposed as the builtin `wasm_log(x i64) -> void` (a `wasm_log` call lowers to `call` of the import); imports occupy the low function indices, so internal function indices, call targets, and function-export indices are shifted by the import count. Functions using enums/`match`, `option`/`result`/`list`/`map`, runtime string construction, or other unsupported builtins are recorded as skipped with a reason and still run on the interpreters; no eligible function yields `WasmError` with code `L0338`. Structural + heap encoder tests (magic/version, section order and counts, import/export presence, shifted call targets, string-literal Data seeding, and struct/array construction eligibility) live in `src/wasm.rs`. `src/native_object.rs` also carries the extended native program emitter (the i64-scalar native mirror of the WASM subset): `emit_alpha1_native_program(&BytecodeModule) -> Result<NativeProgram, NativeProgramError>` lowers every function whose params and return are all `i64` (at most four Win64 register params) to x86-64 machine code with a stack-machine codegen over `rax` — integer literals, params/`let`/`for` locals in `[rbp - slot]` frame slots, `+ - * /` (signed `idiv`+`cqo`), comparisons, short-circuiting `and`/`or`/`not`, `if`/`elif`/`else`, `while`, `loop` with `break`/`continue`, range `for`, `return`, tail expressions, and calls to other compiled i64 functions. The subset now also lowers **stack-allocated scalar aggregates** as function locals: all-`i64` (optionally nested) structs and fixed-length `i64` arrays are laid out contiguously in the frame (each value a run of 8-byte words sized by a `NativeType` flattening), with positional/named struct construction, field read/mutation, array literal construction, constant- and dynamic-index array read/write (a dynamic index computes `rbp - (base + 8*const_words) - 8*elem_words*index` into `rcx`; native array bounds are not trapped for this increment), and a constant-folded `len` on a fixed native array. Aggregates as parameters/returns/call arguments remain deferred (locals only). It also takes the **first heap step**: `len("...")` over a string literal is lowered (not constant-folded) by interning the literal's bytes into a read-only `.rdata` section (`__str{i}`, deduped), copying them into a `.bss` bump-allocated heap region at runtime, and scanning the heap copy for its byte length — emitting two `.text` helper functions (`__lullaby_alloc` bump allocator; `__lullaby_strlen_copy` allocate+copy+scan) plus the `.bss` bump-pointer cell `__lullaby_heap_next` and 64 KiB region `__lullaby_heap_base`, with cross-section `IMAGE_REL_AMD64_REL32` relocations from the `lea`/`call` sites (only ASCII string literals, so the byte length equals the interpreter's char-count `len`; there is no native string value beyond the `len` argument). String-free programs keep the exact prior single-`.text` layout byte-for-byte. It writes a COFF object holding an entry stub (`_lullaby_start`, which calls `main`, moves the result into `ecx`, and calls the imported `ExitProcess`) plus every function, with `IMAGE_REL_AMD64_REL32` relocations for calls and long symbol names in the COFF string table; ineligible functions are skipped with a reason and no eligible function yields `NativeProgramError` with code `L0339`. The single-function literal prototype `emit_alpha1_coff_object` is retained for the checked-in object snapshots. Structural tests for the extended emitter (including struct locals, fixed-array const/dynamic indexing, non-i64-field struct rejection, aggregate layout sizing, and the `.rdata`/`.bss` string-constant layout with dedup) live in `src/native_object.rs`.
- `crates/lullaby_ir/tests/`: integration tests and checked-in snapshots for backend metadata. `tests/memory_snapshots.rs` compares bytecode memory-operation metadata for representative Alpha 1 fixtures against `tests/snapshots/*.memory.json`; `tests/native_contract_snapshots.rs` compares the Alpha 1 native backend contract against `tests/snapshots/alpha1_native_backend_contract.json`; `tests/native_object_snapshots.rs` compares literal-return, local-arithmetic, and assignment COFF object-emission output against `tests/snapshots/alpha1_return_42.coff.json`, `tests/snapshots/alpha1_locals_add.coff.json`, and `tests/snapshots/alpha1_assignments.coff.json`.

- `crates/lullaby_cli/src/loader.rs`: multi-file module loader. `load_program(entry)` lexes+parses the entry file and every transitively imported `NAME.lby` sibling, detects import cycles (`L0393`) and missing module files (`L0397`), enforces the flat-namespace no-shadowing rule (`L0391`) and cross-module `pub` visibility (`L0392`) before semantics, and merges all modules into one flat `Program`. `load_program_in_project(entry, search_dirs)` additionally resolves `import NAME` across the importing file's directory then every project/dependency `src` directory; `load_library_project(search_dirs)` loads and validates every module of an entry-less library project. Backends and the semantic analyzer are unchanged; a single-file program with no imports behaves as a direct lex+parse (empty `search_dirs`).

`test` validates the source as a library (no `main`) and runs every zero-parameter `test_*` function (returning `void`/`i64`/`bool`) through the AST interpreter via `run_named_function`, printing `PASS`/`FAIL` per test plus a `N passed, M failed` summary and exiting non-zero on any failure (a test fails when it produces a runtime error, e.g. an `assert(false)`); `compile`, `build`, `wasm`, `native`, and source `run` require zero-argument `main`; `wasm` validates and lowers to IR exactly as `compile` does, then runs `emit_wasm_module` over the scalar-subset functions and writes a binary `.wasm` (default `<stem>.wasm`), listing compiled/skipped functions under `--verbose` and reporting `L0338` when none are eligible;
