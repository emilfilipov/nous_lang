# Repository Map

This file maps the repository layout and explains where to find core information. Keep it up to date whenever source files, directories, commands, tests, docs, or responsibilities change.

## Root

- [CLAUDE.md](../CLAUDE.md): operating guide for agents and contributors. Defines workflow, documentation rules, MCP/ClickUp/GitHub usage, testing expectations, Git rules, and where to find core language information.
- [README.md](../README.md): user-facing project overview, portable package installation, checksum verification, local build commands, CLI summary, documentation links, and the license.
- [LICENSE](../LICENSE): the MIT License Lullaby ships under (`license = "MIT"` in `[workspace.package]`, inherited by every crate). Permissive — free to use, modify, and redistribute; only the copyright/permission notice must be retained.
- `Cargo.toml`: Rust workspace manifest for the Lullaby compiler/toolchain crates. The lockfile records the pinned dependency graph, including serde/serde_json for bytecode artifact serialization.
- `.gitignore`: ignores local build outputs, caches, editor state, and generated artifacts once implementation begins.
- `.gitattributes`: normalizes repository text files to LF line endings.
- `crates/`: Rust implementation crates.
- `assets/`: brand and visual-identity assets. `assets/brand/` holds the canonical logo (`lullaby-mark.svg` one-ink lavender; `lullaby-icon.svg` filled tile), a raster wordmark (`lullaby-wordmark.png`, real Nunito ExtraBold on transparent, for GitHub/READMEs), the multi-size app/installer/favicon icon (`lullaby.ico` + `lullaby-icon-256/512.png`), the 1200×630 social-preview card (`lullaby-social-card.png`, Open Graph / Twitter / GitHub repo preview), the bundled Nunito typeface (`nunito.woff2`, OFL), and the generators `render_icons.py` (raster icons from the mark geometry), `render_wordmark.py` (raster wordmark), and `render_social_card.py` (social card).
- `examples/`: user-facing `.lby` examples packaged with the toolchain.
- `scripts/`: release packaging and verification scripts, including `build_windows_installer.py` (stages the payload and builds the branded WiX `.msi`).
- `installer/`: Windows installer authoring — `lullaby.wxs` (WiX v7 package: Program Files layout, PATH entry, Start Menu shortcuts, branded ARP icon + wizard), `notice.rtf` (the MIT license the wizard shows on its license page), `banner.bmp`/`dialog.bmp` (branded wizard bitmaps), and `render_installer_art.py` (regenerates the bitmaps from the mark geometry).
- `web/`: one-line web installers served from the site — `install.sh` (POSIX) and `install.ps1` (Windows), pure-ASCII bootstrappers that detect OS/arch, resolve the newest GitHub Release (stable, falling back to the newest prerelease), download the matching portable archive, verify its published `.sha256` (abort on mismatch), unpack to a per-user prefix, and wire PATH via the package's bundled helper; support `LULLABY_VERSION`/`LULLABY_PREFIX`/`LULLABY_REPO` and an `uninstall` mode. `web/README.md` documents usage and the checksum-verification security posture.
- `tests/`: shared `.lby` fixtures used by crate and CLI tests.
- `benchmarks/`: performance baseline / regression harness. `run_bench.ps1` imports MSVC `vcvars64` (so `cl` and `lullaby native` link), builds a pure-C reference (`fib.c`), runs the same workload (`fib.lby`) across every tier (C, native, ast, ir, bytecode), and normalizes to ns per operation for before/after diffs (`results_<label>.csv`, git-ignored). `run_loop.ps1` is the compute/loop-bound companion (`sum 0..N`, C ref `loopsum.c`). `run_gcd.ps1` exercises the native inline `gcd` Euclid `div`-loop (accumulates `gcd(i, 1071)`, C ref `gcdbench.c` compiled with `/DGCD_N=<N>`; every tier is checked against the C output) — native runs at C parity (~1.0×). Three aggregate-shape harnesses add array/string/transform coverage (each with a byte-identical `cl /O2` C reference; native correctness spot-checked at a reps that fits a 32-bit exit code, then timed at the large reps): `run_arraysum.ps1` (fat-pointer `array<i64>` reduction, C ref `arraysum.c`; native ~8.7× C), `run_strscan.ps1` (linear `for c in s` char-code checksum, C ref `strscan.c`; native ~49× C — Lullaby chars are UTF-8 code points, so each step decodes, unlike the C byte-walk; the accumulation *build* idiom is intentionally excluded as it is O(n²) and exhausts the no-free native bump heap past ~1000 chars), and `run_csvsum.ps1` (realistic CSV-integer parse+aggregate over `for c in s`, C ref `csvsum.c`; native ~29× C). The generated per-`N` `.lby`/`.exe`/`.csv` are git-ignored. `README.md` records the methodology and the current baseline; the optimization backlog is the ClickUp `18 Performance Optimization` list.
- `benchmarks/crosslang/`: the standing six-language token + performance benchmark (Lullaby, C, C++, Rust, Python, JavaScript). `corpus/<category>/{lullaby.lby,c.c,cpp.cpp,rust.rs,python.py,js.js}` holds real-world function families with identical function names across all six; every `lullaby.lby` passes `lullaby check`. `corpus_tokens.py` tokenizes the corpus (`o200k_base`) into `corpus_data.json`; `run_perf.ps1` times the `count_primes_below` workload into `perf_data.json`; `run_compile_speed.ps1` measures whole-program compile latency (Lullaby `build` vs `cl` vs `rustc`, optimized + debug) into `compile_data.json`; `assemble_report.py` inlines the Nunito font + all three JSON files into `report_template.html` to produce the self-contained `report.html` artifact. `gen_perf_driver.py` + `run_corpus_perf.ps1` are the whole-corpus compute harness: they turn every category's `main` into a hot N-iteration loop and time all three interpreter tiers (plus a native column for the purely-i64-scalar categories, folding each function's i64 result directly) across the full 26-category / 434-function set, per-iteration ns with the parse+launch floor removed. `run_benchmark.ps1` is the one-command runner (tokenize [+ `-Perf`] + assemble). `README.md` documents the pipeline and the rerun-on-change cadence; `SPEC.md` holds the original method rationale; `LULLABY_CHEATSHEET.md` guides corpus authors. Rerun and republish the artifact whenever the language, optimizer, or corpus changes.
- `documents/`: core language documents and planning material.

- `documents/concurrency_model_design.md`: the design for Lullaby's 1.0 safe-tier **actor** concurrency model (actors on a thread pool + intra-actor `async`/`await`), building on the raw substrate in `concurrency_design.md`. **Stage 1 is delivered** (2026-07-15): the `actor`/`state`/`init`/`on` block, `spawn NAME(args) -> Actor<Name>`, and fire-and-forget `tell handle.handler(args)`, fully type-checked (sendability `L0353`, private `state` `L0354`, `spawn`/`tell`/declaration diagnostics `L0349`/`L0352`/`L0348`) and run on the **AST interpreter** via a single-threaded cooperative deterministic mailbox scheduler (drain-before-exit); the IR/bytecode backends reject an actor program (`L0355`) and native/WASM cleanly skip it (`L0339`/`L0338`). **Stage 2 is delivered** (2026-07-16): request-reply `ask handle.handler(args) -> Future<R>`, `await` on an actor future, and reply handlers (a reply is the handler block's final value). `await` drives the mailbox until the one-shot reply slot fills under a **non-reentrant run-to-completion** turn model (an actor is busy for its whole turn incl. nested `await`s; only the first *deliverable* message runs); a request cycle that would re-enter a busy actor is a deterministic deadlock (`L0356`), never a hang. `ask` reuses the `tell` AST node (an `is_ask` flag), so the IR/bytecode `L0355`, native/WASM `L0339`/`L0338`, and `no-runtime` `L0441` gates all cover it unchanged; sendability (`L0353`) is enforced on both `ask` arguments and the reply type, and the send-form mismatch (`ask` a fire-and-forget handler / `tell` a reply handler) is `L0352`. **Stages 3–4 are delivered** (message ownership: move-by-default + `copy`/`shared` + use-after-send `L0357` and the transitive-sendability closure; supervision: `spawn ... supervise restart|stop|escalate` with result-based failure, `L0358`/`L0359`/`L0362`/`L0363`). **Stage 5's future combinators are delivered** (2026-07-18): `join_all EXPR` (wait for all, results in input order, kind-preserving) and `select EXPR` (wait for the first, yielding the built-in `Selected<T>` `{ index i64, value T }`, lowest input index winning a tie) — new keyword prefix operators over a `list`/`array` of `ask` futures, on the AST interpreter only, with the operand-shape diagnostic `L0364` and the same `L0355`/`L0339`/`L0338`/`L0441` tier gates; stage-5 **back-pressure** (bounded mailbox / `try_tell`) and stage 6 (native/WASM actor codegen) remain design-only. See the doc's "Stage N delivery" sections for the live slice.

Dotted paths, qualified access, per-field visibility, and remote/registry dependency fetching deferred.
- `documents/standard_library.md`: the prelude — a single catalog of all compiler-provided built-in types and functions (always in scope, no import needed): conversions, collections (`list`/`map`), strings, math, streams/file/system I/O, memory/references, and error handling.
- `documents/generics_design.md`: user-defined generics design — generic functions `fn name<T> ...` with call-site type inference, erased at runtime (backends unchanged), inference mirrored in the IR lowerer, and `L0394`/`L0395`/`L0396`; explicit type arguments (turbofish) deferred (trait bounds are now implemented — see [traits_design.md](traits_design.md)). Stage 1 of generic user *types* has shipped: a single-parameter generic `struct Box<T>` with a scalar `T` — parser `type_params` on `StructDecl`, semantics scoping/substitution/inference-directed construction (`L0454`/`L0455`), type erasure on the three interpreters, and a clean native/WASM skip (`L0339`). Stage A1 adds single-parameter generic **enums** `enum Opt<T>` — `type_params` on `EnumDecl`, generic variant construction + `match` with the payload `T` substituted (head-aware `match_variants`/`variant_binding_types`), the recursion-through-indirection rule (`L0456`: a directly recursive generic enum must route through `rc`/`ref`/`ptr`/`list`/`map`/`array`), `decompose_generic` extended to recurse through user generics, again with erasure on the interpreters and a clean native/WASM skip. Stage 4 adds methods on generic types (inherent `impl Box<T>` blocks). Stage 5 (the final generics-frontend increment) adds **multiple type parameters** (`struct Pair<K, V>`, `enum Either<L, R>`, inferred independently) and **trait bounds on a type parameter** (`struct Sorted<T: Ord>` / `impl Sorted<T: Ord>`): the declaration's bound propagates into the type's inherent-impl methods so a `T` value may call the bound trait's methods, and every instantiation is bound-checked (`generic_type_bounds` unions decl + impl bounds per parameter position; unsatisfied → `L0400`), still erased on the interpreters with a clean native/WASM skip. **Native monomorphization for scalar type arguments has shipped** (A1 stage-1 native): a generic `struct`/`enum` instantiated with scalar arguments (`Box<i64>`, `Pair<i64, bool>`, `Opt<i64>`, `Either<i64, bool>`) is monomorphized in `resolve_native_type` — the `IrStructDef`/`IrEnumDef` `type_params` are substituted with the instantiation's concrete arguments (reusing `lullaby_semantics::substitute_type`) and the resulting scalar-only concrete layout compiles byte-identically to the interpreters' erased value (value-neutral, `is_scalar_only_layout` default-deny gate). **Native monomorphization for one-level heap-`string` type arguments has also shipped**: `Box<string>`, `Pair<string, i64>`, `Opt<string>`, `Either<i64, string>` now monomorphize to a layout byte-identical to a hand-written string-field struct / string-payload enum, so the recursive `rc_dec` drop-glue reclaims each instantiation's `string` field per iteration (composing with the RC/free-list path and the arena rewind), and the arena escape analysis (`heap_carrying_generic_instantiations`) recognizes a heap-`T` generic value as heap-carrying so a per-iteration sub-region never reclaims a live record. The layout gate widened from `is_scalar_only_layout` to `is_native_supported_generic_layout` (scalar + one-level `string`), matching the non-generic heap-field aggregate boundary. A **deeper** heap shape (a mutable heap field `Stack<i64>` → `list<i64>`, a recursive `Tree<i64>`, a nested-generic-aggregate field, a two-level `string` nesting) and inherent-method calls still skip cleanly to the interpreters. WASM monomorphization is the next stage.
- `documents/traits_design.md`: traits/interfaces design — `trait Name` method signatures with a `self` receiver, `impl Trait for Type`, receiver-type method dispatch via a shared impl table on all backends, and trait bounds on generics (`<T: Ord>`); records `L0398`/`L0399`/`L0400` and defers default bodies, trait objects, associated types, and supertraits.
- `documents/wasm_backend_design.md`: WebAssembly backend design — compile the typed IR's scalar subset (`i64`/`f32`/`f64`/`bool`/`char`/`byte` plus the fixed-width integer kinds, arithmetic/comparison/logic including single-precision `f32` ops and the `to_f32`/`to_f64` conversions, `if`/`while`/`loop`/`for`, calls, locals) to a binary `.wasm` module with std-only encoding, a `lullaby wasm` CLI command, and node/wasmtime-gated execution parity against the interpreter. The first linear-memory step is landed: the module exports a `"memory"` (min 1 page), declares a mutable `i32` bump-pointer global, seeds a Data section, emits an internal `__alloc(size i32) -> i32` bump helper, and imports three host functions from module `env` surfaced to Lullaby as builtins — `env.log_i64` as `wasm_log(x i64) -> void`, `env.console_log(ptr i32, len i32)` as `console_log(s string) -> void`, and `env.dom_set_text(id_ptr i32, id_len i32, text_ptr i32, text_len i32)` as `dom_set_text(id string, text string) -> void` — the JS/DOM browser-host interop layer (imports occupy the low function indices, so internal function indices/call targets/exports are shifted by the import count, driven by `IMPORT_FUNC_COUNT`). **Heap types are now lowered into linear memory:** `string` values are `i32` pointers to `[char_len i32][byte_len i32][utf8 bytes]` with literals interned once into the Data section (and `len(s)`); `struct` values are pointers to one 8-byte slot per field with positional construction/field read/field mutation; fixed `array` literals are pointers to `[len i32][8-byte elem slots]` with construction, indexed read/write, and `len(a)`; field/element slots may be scalars or nested pointers, and `__alloc` builds structs/arrays at runtime. The JS/DOM string imports pass each string's `(ptr, len)` decoded out of `memory`; on the interpreters `console_log` prints the string and `dom_set_text` prints `id=text` so parity holds. Enum/tagged-union + `match` lowering, `option`/`result` (incl. string and one-level mutable-heap payloads), `list`/`map` (scalar/string/one-level-mutable elements, scalar and content-equality string keys), runtime string construction (`+` concat, `to_string`, `substring`/`find`/predicates), the overflow-aware arithmetic builtins, the scalar math builtins (`sqrt`/`abs` on `f64`, `abs` on `i64`, and `min`/`max`/`gcd`/`sign`/`clamp` on `i64` — the `f64` cases of `min`/`max`/`sign`/`clamp` are deferred like native), and value-producing `if`/`else` are all landed; a free-list allocator (bump-only, no reclamation), `to_string(float)`, the transcendental math builtins (`sin`/`cos`/`floor`/…, `pow`), the remaining string builders (`replace`/`upper`/`lower`/`split`/`join`), deeper-nested/`array`/`map`-element collections, and a richer DOM interop surface remain deferred.

## Source Layout

The implementation is a Rust workspace. Unless changed by an explicit architecture decision, keep this layout:

- `crates/lullaby_lexer/`: source extension validation, tokenization, indentation scanning, string- and char-literal lexing (single-quoted `'c'` char literals, with malformed-literal diagnostic `L0105`), numeric literals with `_` digit separators (scanned into the number token; the parser validates placement), forbidden brace/semicolon diagnostics, current and planned keyword recognition (including the `import` and `pub` module keywords and the `const` compile-time-constant keyword), and lexical tests.
- `crates/lullaby_diagnostics/`: shared diagnostic data structures, serializable span/traceback metadata, registry metadata (the `DIAGNOSTIC_CATALOG` verbose-explanation table, which intentionally covers only the subset of codes that benefit from a long-form explanation), concise/verbose renderers, and deterministic JSON rendering.
- `crates/lullaby_diagnostics/tests/registry_sync.rs`: drift guard that keeps [diagnostic_registry.md](diagnostic_registry.md) and the codes actually emitted by the compiler in sync. It parses the registry's `## Codes` table rows and scans every non-test `crates/*/src/**/*.rs` for `L####` *string literals* (quoting is what separates an emission site from a prose mention), then fails if any code is declared twice (one number, two meanings), is emitted but undocumented, or is documented but no longer emitted. Both sides are read from the real sources, so the guard cannot go stale like a hand-maintained list.
- `crates/lullaby_lsp/`: additive Language Server Protocol (LSP) server for `.lby` files, standard-library + `serde_json` only (no third-party LSP/async framework). `src/transport.rs` implements JSON-RPC 2.0 over stdio with `Content-Length: N\r\n\r\n<json>` framing and the read/write loop (`serve`/`run_stdio`); `src/lib.rs` holds the pure `handle_message(state, method, id, params) -> Vec<Message>` request core (so the protocol is testable without stdio), the `ServerState` in-memory document map, `initialize` (advertising `textDocumentSync=1` full, `documentFormattingProvider=true`, `hoverProvider=true`, `definitionProvider=true`, `completionProvider={resolveProvider:false}`), `initialized`/`shutdown`/`exit`, and full-text `didOpen`/`didChange`/`didClose`; `src/diagnostics.rs` runs the existing lex -> parse -> `validate` pipeline over a document and converts each Lullaby diagnostic (code/message/1-based span) into an LSP `Diagnostic` with a 0-based `Range` (widened to the token at the span, else one char) for `textDocument/publishDiagnostics`, and formats via `format_program` for `textDocument/formatting` (one full-document `TextEdit`, or no edits when the document does not parse); `src/analysis.rs` resolves the identifier under the cursor for `textDocument/hover` (function signature, local/parameter type from the checker's `expression_types`, struct/enum declaration, or builtin description) and `textDocument/definition` (a `Location` for the declaration or `let`/parameter binding), reusing the parser plus `validate` metadata with no new semantics accessor. `src/completion.rs` handles `textDocument/completion`: it returns a `CompletionItem[]` from the union of the Lullaby keyword set (mirrored from the lexer's keyword table as `KEYWORDS` and pinned to it by a test that asserts every entry still lexes to a single `Keyword` token), the current buffer's top-level declarations (functions/structs/enums/aliases/traits/constants) plus the enclosing function's locals/parameters — each with the correct integer `CompletionItemKind` and a signature/type `detail` — and, for a module-aware file, the `pub` declarations of every imported module (added by `project::completion` via the loaded program's per-module views); keywords are produced unconditionally so an unparseable mid-edit buffer still yields keyword completions without panicking, and member/`.`-completion, signature help, and completion ranking are deferred. **`src/project.rs` makes the server module/project-aware** (closing the top B3 LSP gap): when the open document uses `import` or lives inside a `lullaby.json` project (found by walking up for a manifest), it runs the shared `lullaby_loader` over the file's project — with every open buffer supplied as a `SourceOverlay` so unsaved edits (not stale disk bytes) are analyzed — so **diagnostics** reflect the merged program (a `pub` symbol from an imported module no longer reads as undefined, and the loader's cross-module diagnostics `L0391`/`L0392`/`L0393`/`L0397` for this file surface), while the open file's own lex/parse errors (reported directly from the buffer) and semantic errors (attributed back to the open file by enclosing-function name — names are globally unique under `L0391` — falling back to span) stay at their real positions; and **hover/go-to-definition cross module boundaries** (a reference to an imported `pub` function/type/enum/alias resolves to its declaration in the other file via `LoadedProgram.modules`, returning a `Location` with the other file's `file://` URI). A lone file with no imports and no project falls through to the unchanged single-document pipeline, so single-file behavior is unregressed; every fallible step (non-`file://` URI, malformed manifest, missing import, mid-edit buffer) degrades to single-document analysis rather than panicking. Unit tests drive `handle_message` directly for the initialize/diagnostics/formatting/hover/definition/completion cases and the framed stdio loop end to end, plus filesystem-backed two-file-project tests in `project.rs` (imported symbol not flagged undefined, a genuine entry error still reported at the right range, cross-module hover/definition, completion including an imported `pub` symbol with its signature detail, an unsaved-buffer overlay overriding disk, and single-file parity with the single-document pipeline); `completion.rs` additionally unit-tests keyword/declaration/local kinds, the unparseable-buffer keyword fallback, label de-duplication, and the keyword-list-matches-lexer guard. Run over stdio with `lullaby lsp`. Design: [lsp_design.md](lsp_design.md).
- `crates/lullaby_wasm/`: WebAssembly entry point for the browser playground (excluded from the native workspace; built standalone with `wasm-pack build crates/lullaby_wasm --target web`). Depends on the lexer/parser/semantics/runtime and exposes a single `#[wasm_bindgen] run(source) -> String` that lexes → parses → semantic-checks → runs the AST interpreter entirely in wasm and returns JSON `{ok, output, error}`. Program `print`/`println` output is captured via a `#[cfg(target_arch = "wasm32")]` thread-local in the runtime (`clear_wasm_output`/`take_wasm_output`) since wasm has no stdout; randomness uses getrandom's `wasm_js` backend (`.cargo/config.toml` sets `--cfg getrandom_backend="wasm_js"`). The generated `pkg/*.js` + `*.wasm` are copied into the `lullaby-web` site; no code executes off the user's machine.
- `crates/lullaby_parser/`: AST model and parser for function declarations, typed parameters, return types, indentation blocks, explicit and inferred `let`, assignment, `return`, `break`, `continue`, if/elif/else, while/loop/range-for, calls, integer and `f64` float literals (with optional `_` digit separators validated and stripped via `normalize_number_literal`), `'c'` char literals, array literals/indexing, struct declarations/construction/field access, enum (tagged-union) declarations with positional payload types, `match` expressions over enums with variant/wildcard patterns and payload bindings, variables, arithmetic, comparison, logical expressions, named compile-time constant declarations `const NAME type = <expr>` (`ConstDecl`; the type annotation is mandatory), leading `import NAME` module lines and an optional `pub` export modifier on top-level `fn`/`struct`/`enum`/`alias`/`const` declarations, an optional `<T, U>` or bounded `<T: Trait + Other>` generic type-parameter list after a function name with duplicate/built-in-shadowing rejection (`L0394`), `trait NAME` declarations (indented `fn method self ... -> Ret` signatures) and `impl Trait for Type` blocks (indented `fn method self ... -> Ret` bodies, with the untyped `self` receiver injected as the implementing type), reported as `L0216` when malformed (`imports`/`is_public`/`type_params`/`traits`/`impls` are serde-defaulted, and a `TypeParam` serializes as the bare name string when unbounded, so single-file artifacts and AST snapshots stay valid), and `L0211` rejection for planned syntax keywords that are not yet supported. The parser also accepts an `async` modifier on `fn` declarations, the `Future<T>` type, and `await EXPR` expressions. Also hosts `format.rs`, the canonical source formatter (`format_program`) that renders an AST back to idempotent, source-ordered `.lby` source (including `trait`/`impl` blocks and `<T: Bound>` lists) and backs `lullaby fmt`, with per-construct idempotency/round-trip regression tests over representative valid fixtures.
- `crates/lullaby_parser/tests/`: parser integration tests, including checked-in AST golden snapshots under `crates/lullaby_parser/tests/snapshots/`. Refresh snapshots intentionally with `$env:LULLABY_UPDATE_PARSER_SNAPSHOTS='1'; cargo test -p lullaby_parser --test ast_snapshots; Remove-Item Env:LULLABY_UPDATE_PARSER_SNAPSHOTS`.
- `crates/lullaby_semantics/`: static validation for duplicate declarations, explicit and inferred local binding types, assignment targets/types, function call arity/types, return behavior, bool conditions, loop-control placement, arithmetic/comparison/logical expression operand types, homogeneous non-empty arrays, array indexing, nominal struct types with positional construction and field access, nominal enum (tagged-union) types with globally unique variant names and unit/payload construction, interim pointer-style memory builtins, reference-counted `rc<T>`/`ref<T>`/`ptr<T>` types with `rc_new`/`rc_clone`/`rc_release`/`rc_get`/`rc_borrow`/`ref_get` builtins and `unsafe`-gated raw `ptr_read`/`ptr_write`, the actor-model `shared<T>` atomic-rc immutable share with `share`/`shared_get` builtins (concurrency stage 3), text file I/O builtins, standard stream builtins (`print`/`println`/`warn`/`flush`), the `wasm_log(x i64) -> void` host-log builtin (prints the value on the interpreters; lowers to the imported `env.log_i64` on the WASM backend), the JS/DOM host-interop builtins `console_log(s string) -> void` and `dom_set_text(id string, text string) -> void` (print the string / `id=text` on the interpreters; lower to the imported `env.console_log`/`env.dom_set_text` on the WASM backend), `i64`/`f64` numeric arithmetic, `to_string` conversion and `+` string concatenation, safe system command builtins, user-defined generic functions with call-site type inference (structural unification of arguments against type-variable-containing parameter types, substitution into the return type, conflicting inference `L0395`, uninferable-return-only type variable `L0396`; a bare `T` supports only universal operations, so arithmetic/ordering on it is rejected), traits and impls (a trait table plus an impl table built from `trait`/`impl` declarations, `impl` conformance checking with `Self` substitution `L0398`, duplicate-impl `L0399`, trait-method resolution that dispatches a UFCS call on the receiver type, and trait-bound checking at bounded-generic call sites and on bounded receivers `L0400`, with free-function and trait-method namespaces kept disjoint), `async fn`/`await` typing (an `async fn ... -> T` produces `Future<T>`; `await e` requires `e: Future<T>` and yields `T`; misuse is `L0344`), `export fn` C-callable-export validation (the first increment requires an all-`i64`, non-generic signature; anything else is `L0424`), and executable zero-argument `main` validation for compile/run. Successful validation returns `CheckedProgram` with function signatures (including generic `type_params` and per-parameter trait bounds) and inferred expression-type metadata; the generic unifier (`unify_param`/`substitute_type`/`infer_generic_return`) is public so the IR lowerer reuses it.
- `crates/lullaby_ir/`: typed semantic IR schema, lowering from `CheckedProgram`, memory-operation analysis with sequence and safety metadata for allocation/load/store/deallocation, bounds-checked indexing, reference-counted `rc`/`ptr` operations, and region creation, deterministic stack-frame layout analysis (`frame_layout`) assigning parameters/locals/loop-variables stable word-aligned slots with per-scope reverse-order cleanup plans, reserved safety semantics for future region/copy/cleanup operations, serializable native backend contract data for target/layout/calling/cleanup policy, first `x86_64-pc-windows-msvc` COFF object-emission prototype for literal-return, stack-backed `i64` local arithmetic, and straight-line `i64` assignment entry functions, deterministic optimization pass configuration with opt-in constant folding, conservative block-local CSE, conservative loop-invariant motion, conservative block-local copy propagation, block-local dead-code elimination, executable IR interpreter with `throw`/`try`/`catch` structured error handling lowered and executed on the IR and bytecode backends at parity with the AST runtime, `async fn`/`await`/`Future` lowering and execution on the IR and bytecode backends at parity with the AST runtime (`L0344` on misuse), trait-method dispatch via a shared impl table (`impls`/`trait_methods` on `IrModule`/`BytecodeModule`, serde-defaulted and threaded through lowering, optimization, and the bytecode round-trip; a trait-method call selects the impl function by the receiver's runtime type), inline and fixture-driven AST/IR/bytecode parity tests including optimized backend variants, explicit instruction-bytecode lowering, versioned `.lbc` artifact encoding/decoding with metadata/function-table/memory-operation/instruction-contract compatibility checks, checked-in backend memory metadata snapshots, and bytecode VM entry point for the current implemented subset. `src/wasm.rs` is the WebAssembly backend (scalar subset, linear-memory step, and heap types in linear memory): `emit_wasm_module(&IrModule) -> Result<WasmArtifact, WasmError>` lowers every eligible top-level function to a binary `.wasm` module, implementing the WASM binary encoder from scratch with the Rust standard library only (module header, Type/Import/Function/Memory/Global/Export/Code/Data sections in canonical order, LEB128, and the stack-machine opcodes for literals, locals, arithmetic/comparison/logic, structured `block`/`loop`/`if`/`br`/`br_if`, calls, `return`, and typed `i32`/`i64`/`f32`/`f64` `load`/`store`). Scalars map as before (`i64`/`f32`/`f64`/`bool`/`char`/`byte`; `f32` is WASM's native single-precision `f32` value type and local). It now compiles **32-bit floats (`f32`) to parity with `f64`**: single-precision arithmetic `+ - * /` (`f32.add/sub/mul/div`, so f32 stays single precision and is bit-identical to the interpreter's real `f32`), IEEE-754 NaN-aware comparisons `< <= > >= == !=` (`f32.lt/le/gt/ge/eq/ne`, false on NaN for relational + `==`, true for `!=`), `f32.const` literals rounded to single precision, and the float conversions `to_f32(x f64) -> f32` (`f32.demote_f64`, round) and `to_f64(x f32) -> f64` (`f64.promote_f32`, widen) recognized and inlined as builtin calls (not real calls) — matching the interpreters and the native backend bit-for-bit (node execution parity for `run_f32.lby` → 3 and `native_floats.lby` → 9). Because the IR annotates a float *arithmetic* node `i64`, the operand float width for arithmetic/comparison is derived structurally from the reliably-typed leaves (float literals/locals and `to_f32`/`to_f64`) via `float_val_type_of`, mirroring the native backend's `float_width_of_expr`; f32/f64 struct fields and array elements round-trip through single/double-precision `f32.store`/`f32.load` (4-byte) slots. Within eligible functions it now also compiles the **fixed-width integer types** (`i8`/`i16`/`i32`/`u8`/`u16`/`u32`/`u64`/`isize`/`usize`), each held in an `i64` local as the interpreter's normalized cell: wrapping `+ - *` and `/` re-normalized to the width (sign-extend via `i64.extend8_s`/`extend16_s`/`extend32_s` for signed 8/16/32 kinds, an `i64.and` width mask for unsigned kinds, no-op for the 64-bit kinds), signedness-correct comparison (`lt_u`/`lt_s` etc.) and division (`div_u`/`div_s`, trapping on zero like the `i64` path), bitwise `& | ^` and `~` (as `xor -1`) re-normalized, shifts with the count masked to `width-1` and right shift logical-for-unsigned (`shr_u`) / arithmetic-for-signed (`shr_s`), and the `to_<T>`/`to_i64` conversions inlined as a width-normalize (not real calls) — matching the interpreters and the native backend bit-for-bit (node execution parity). Heap values beyond the pointer types below, the option-returning `checked_*`/`saturating_*`/`wrapping_*`/`to_string` builtins, and the float math builtins (`sqrt`/`sin`/…) remain deferred and skip gracefully. **Heap types are `i32` pointers into linear memory**: a `string` points to `[char_len i32][byte_len i32][utf8 bytes]` with literals interned once into the Data section (offset math via `RESERVED_BASE`) and `len(s)` lowered to `i32.load` + `i64.extend`; a `struct` points to one 8-byte slot per field with positional construction (a `Call` named for the struct), `.field` read, and `p.field = v` mutation; a fixed `array` literal points to `[len i32][8-byte elem slots]` with construction, `a[i]` read, `a[i] = v` write (WASM traps on OOB memory access), and `len(a)`; field/element slots may be scalars or nested pointers, and `struct`/`array` construction calls `__alloc`. The module still exports `"memory"` (min 1 page), declares a mutable `i32` bump-pointer global (now initialized past the Data-seeded reserved region AND the whole string-literal pool), emits the internal `__alloc(size i32) -> i32` bump helper, and imports three host functions from module `env` — `env.log_i64 (func (param i64))` exposed as `wasm_log(x i64) -> void`, `env.console_log (func (param i32 i32))` exposed as `console_log(s string) -> void`, and `env.dom_set_text (func (param i32 i32 i32 i32))` exposed as `dom_set_text(id string, text string) -> void` (the JS/DOM browser-host interop layer; a call to any of them lowers to `call` of its import, and the string imports push each string's `(ptr, len)` — the pointer plus the `[len i32]` header — decoded out of `memory` by the host); imports occupy the low function indices, so internal function indices, call targets, and function-export indices are shifted by the import count (`IMPORT_FUNC_COUNT`, now 3). It also compiles **enum values and `match`** for enums with scalar payloads: an enum value is an `i32` pointer to a `[tag i32 (padded to 8)][payload slots…]` linear-memory record — the tag is the variant's index in declaration order (matching the interpreters, which dispatch `match` by variant name against that same ordered layout) and there is one 8-byte payload slot per position, sized for the widest variant. Construction (`some(x)`/`none`/`ok(x)`/`err(e)` and user `Variant(payload…)`, all emitted by the IR lowerer as positional `Call`s) `__alloc`s the record, stores the tag at offset 0, and stores each payload; `match` loads the tag (`i32.load` at offset 0) and dispatches with a chain of `i32.eq` + typed `if`/`else` blocks (a `Wildcard` arm becomes the final `else`; with exhaustiveness guaranteed by semantics the last variant arm is emitted unconditionally so a value match always leaves a value), binding each arm's payload slots into locals before its body. This covers the built-in `option<T>`/`result<T, E>` when `T`/`E` are scalar and user enums whose every variant payload is scalar, including matching a call that returns `option<scalar>`/`result<scalar,scalar>`. It also **passes and returns scalar-field aggregates across function boundaries**: a `struct` (of scalars, nested scalar structs, or fixed scalar arrays), a fixed scalar `array`, or a scalar-payload `enum` may be a parameter, a return value, or a call argument — the `i32` pointer is passed/returned directly, but Lullaby value semantics are preserved by **deep-copying each mutable aggregate argument at the call site** (`is_mutable_aggregate` + `emit_deep_copy`): a fresh record is `__alloc`'d and every word copied, recursing into nested mutable aggregate fields/elements (a struct copies each field, an array copies its `[len]` header + elements in a runtime loop, an enum copies its `[tag][payload]` record word-for-word since payloads are scalar), exactly mirroring the interpreters' recursive `Value::clone`; an immutable `string` argument is shared, not copied, and a returned aggregate is the callee's own fresh record. Functions using an enum with a **heap** payload (`string`/`list`/`array`/`map` — notably `result<i64, string>`), aggregates containing `list`/`map`, runtime string construction, or other unsupported builtins are recorded as skipped with a reason and still run on the interpreters; no eligible function yields `WasmError` with code `L0338`. Structural + heap + enum/match encoder tests (magic/version, section order and counts, import/export presence, shifted call targets, string-literal Data seeding, struct/array construction eligibility, struct/array param+return compilation, the copy-on-pass `__alloc` for a forwarded struct/array argument and the no-copy of an immutable string argument, enum-layout discriminant ordering, the tag-load + typed-`if` dispatch, tag-store construction, and heap-payload-enum skip) live in `src/wasm.rs`. `src/native_object.rs` also carries the extended native program emitter (the i64-scalar native mirror of the WASM subset): `emit_native_program(&BytecodeModule) -> Result<NativeProgram, NativeProgramError>` lowers every function whose params and return are all `i64` (at most four Win64 register params) to x86-64 machine code with a stack-machine codegen over `rax` — integer literals, params/`let`/`for` locals in `[rbp - slot]` frame slots, `+ - * / %` (signed `idiv`+`cqo`), scalar bitwise `& | ^ << >>` (`and`/`or`/`xor rax`, `shl`/`sar` with the count masked to 63), comparisons, short-circuiting `and`/`or`/`not`, `if`/`elif`/`else`, `while`, `loop` with `break`/`continue`, range `for`, `return`, tail expressions, and calls to other compiled i64 functions. It also **auto-vectorizes** three array-loop shapes over contiguous `array<i64>` into SSE2 packed loops (two lanes/iter, scalar tail, bit-for-bit identical to the scalar loop): sum reductions (`paddq`), element-wise maps `c[i]=a[i]±b[i]` (`paddq`/`psubq`), and bitwise reductions/maps for `& | ^` (`pand`/`por`/`pxor`, seeded with the operator identity); the exact matcher falls back to the scalar loop for any other shape. Within these otherwise-`i64` functions it now also compiles the **fixed-width integer types** (`i8`/`i16`/`i32`/`u8`/`u16`/`u32`/`u64`/`isize`/`usize`), each held in a register as the interpreter's normalized `i64` cell: wrapping `+ - * /` re-normalized to the width, signedness-correct comparison (unsigned/signed condition codes) and division (`div`/`idiv`), bitwise `& | ^ ~`, shifts with the count masked to the width and right shift logical-for-unsigned / arithmetic-for-signed, and the `to_<T>`/`to_i64` conversions emitted inline as a width-normalize — matching the interpreters bit-for-bit (native link-and-run parity). Within these otherwise-`i64` functions it now also compiles **floating-point values** (`f64` and `f32`) using SSE scalar (XMM) registers: `f64` held as a `double`, `f32` kept rounded to single precision after each op (matching the interpreter's real `f32` storage), spilled to 8-byte frame slots when a temporary must survive another evaluation; IEEE-754 arithmetic `+ - * /` (`addsd/subsd/mulsd/divsd` for `f64`, `addss/subss/mulss/divss` for `f32`, so division by zero yields inf/NaN and never traps); ordered comparisons `< <= > >= == !=` (`ucomisd`/`ucomiss` with unordered-aware set conditions, so a NaN operand makes every relational compare and `==` false and `!=` true, producing an `i64`/`bool`); and the `to_f32(x f64) -> f32` (`cvtsd2ss`, round) / `to_f64(x f32) -> f64` (`cvtss2sd`, widen) conversions recognized and inlined; float literals are materialized via a GPR→XMM `movq`/`movd` (no `.rdata` constant). The signature constraint is unchanged (all params + return `i64`), so functions with a float parameter or return type, float heap values, and the transcendental/math builtins (`sqrt`/`sin`/`floor`/…) remain deferred and skip gracefully. Heap values, the option-returning `checked_*` builtins, and the `saturating_*`/`wrapping_*` builtins remain deferred and skip gracefully. The subset now also lowers **stack-allocated scalar aggregates** as function locals: all-`i64` (optionally nested) structs and fixed-length `i64` arrays are laid out contiguously in the frame (each value a run of 8-byte words sized by a `NativeType` flattening), with positional/named struct construction, field read/mutation, array literal construction, constant- and dynamic-index array read/write (a dynamic index computes `rbp - (base + 8*const_words) - 8*elem_words*index` into `rcx`; native array bounds are not trapped for this increment), and a constant-folded `len` on a fixed native array. Scalar-field aggregates (structs of scalars, fixed scalar arrays, scalar-payload enums) also cross function boundaries as parameters, returns, and call arguments via a hidden-return-pointer + by-pointer-argument ABI with copy-in value semantics (a callee mutating an aggregate parameter never affects the caller's copy), matching the interpreters; heap-containing aggregates and a top-level float scalar parameter/return remain deferred. It also lowers **stack-allocated enums with scalar payloads and `match`**: the built-in generics `option<T>`/`result<T, E>` (for scalar `T`/`E`) and user enums are laid out as a tag word (the variant's discriminant index in the interpreters' variant order) plus a shared payload region sized to the widest variant; construction writes the tag and payload words, and `match` reloads the tag and dispatches per arm with `cmp`/`jne` (binding scalar payloads into arm-scoped locals, honoring the `_` wildcard, and emitting `ud2` on the impossible fallthrough of a wildcard-free exhaustive match). The scrutinee may be an enum local or a freshly-constructed enum; heap-payload enums (`result<i64, string>`), enums as parameters/returns, and a `match` on an enum-returning call are deferred and skip gracefully. It also takes the **first heap step**: `len("...")` over a string literal is lowered (not constant-folded) by interning the literal's bytes into a read-only `.rdata` section (`__str{i}`, deduped), copying them into a `.bss` bump-allocated heap region at runtime, and scanning the heap copy for its byte length — emitting two `.text` helper functions (`__lullaby_alloc` bump allocator; `__lullaby_strlen_copy` allocate+copy+scan) plus the `.bss` bump-pointer cell `__lullaby_heap_next` and 64 KiB region `__lullaby_heap_base`, with cross-section `IMAGE_REL_AMD64_REL32` relocations from the `lea`/`call` sites (only ASCII string literals, so the byte length equals the interpreter's char-count `len`; there is no native string value beyond the `len` argument). String-free programs keep the exact prior single-`.text` layout byte-for-byte. It writes a COFF object holding an entry stub (`_lullaby_start`, which calls `main`, moves the result into `ecx`, and calls the imported `ExitProcess`) plus every function, with `IMAGE_REL_AMD64_REL32` relocations for calls and long symbol names in the COFF string table; ineligible functions are skipped with a reason and no eligible function yields `NativeProgramError` with code `L0339`. It also exposes **C-callable exports**: an `export fn` (tracked via `export_functions` on `IrModule`/`BytecodeModule`) is emitted like any i64-scalar function, and because every compiled function symbol is already storage-class `EXTERNAL` defined in `.text` under its plain name, a C caller declaring `extern <ret> NAME(<args>);` links against it. An export-only program (no `main`) emits a **library object**: the entry stub and its `ExitProcess` dependency are omitted (`NativeProgram.entry_symbol` empty) so a C `main` can link against it without a `main` collision; a program with both `main` and exports keeps the stub. It also **calls C via `extern fn`** (C-ABI FFI): a body-less `extern fn` (tracked via `extern_functions`, with each C-ABI signature carried in `extern_signatures` on `IrModule`/`BytecodeModule`, both serde-defaulted) lowers to a `call` of an undefined external symbol (`IMAGE_REL_AMD64_REL32`, section 0), linked against the C runtime import library `ucrt.lib`. Marshalling covers the integer-class C scalars — every fixed-width integer (`i8`…`u64`, `isize`/`usize`) plus `bool`/`char`/`byte`: arguments pass in the low bits of `rcx`/`rdx`/`r8`/`r9` (already width-normalized in the interpreter's cell model), and a narrow C return is re-normalized in `rax` (sign-extend for signed kinds, zero-extend for unsigned). `f32`/`f64` externs need XMM argument routing and are deferred, so a caller of a float extern demotes to the interpreters (which reject the extern call with `L0423`). The single-function literal prototype `emit_coff_object` is retained for the checked-in object snapshots. It also lowers **inline assembly**: an `unsafe` `asm` statement (an `IrStmt::Asm`/`BytecodeInstruction::Asm` carrying raw `Vec<u8>` bytes, native-only) is emitted verbatim into the current function's `.text`; a tail `asm` is followed by the normal Win64 epilogue so `rax` is returned intact (the programmer must not emit their own `ret`). It also emits optional **CodeView source-line debug info**: `emit_native_program_with_debug(module, Some(DebugOptions { source_file }))` adds a `.debug$S` section (C13 signature; `DEBUG_S_SYMBOLS` `S_COMPILE3`, one `DEBUG_S_LINES` per function with `SECREL32`+`SECTION` relocations to its `.text` symbol mapping the entry offset to `BytecodeFunction.span.line`, `DEBUG_S_FILECHKSMS`, `DEBUG_S_STRINGTABLE`) so a linker-built PDB lets a debugger break at a function and show its `.lby` line; it is opt-in (`lullaby native --debug`/`-g`) and leaves the default object bytes byte-for-byte unchanged (`emit_native_program` is the no-debug wrapper). Structural tests for the extended emitter (including struct locals, fixed-array const/dynamic indexing, non-i64-field struct rejection, aggregate layout sizing, the `.rdata`/`.bss` string-constant layout with dedup, verbatim `asm` byte emission, and the CodeView `.debug$S` per-function line table with byte-for-byte default parity) live in `src/native_object.rs`.

- `crates/lullaby_cli/src/`: the `lullaby` binary, split per concern (largest file ~500 lines, a third of the ~1500 cap). `main.rs` is the dispatcher only — it parses `argv` into an `Invocation` and routes it to a handler. `args.rs` holds the `Invocation`, the flag/mode enums, the parsers, and the per-command usage strings; `compile.rs` resolves a path argument to a single `.lby` file or a `lullaby.json` project and drives the loader + semantic validator; `diagnostics.rs` builds each phase's `DiagnosticReport` and renders a batch in the selected `OutputMode`; `help.rs` carries the `--help` text and the `--version` string. `commands/` has one module per subcommand (`build` = `check`/`build`/`compile`, `fmt`, `inspect`, `native` + `native_link` (linker discovery), `project` = `new`/`examples`/`docs`/`lsp`, `run`, `test` + `test_isolate`, `wasm`). `commands/test_isolate.rs` is the test runner's isolation mechanism: `lullaby test` re-invokes the binary through a hidden `__run-test-batch` subcommand so the suite runs in a CHILD process, contained against the two shapes that are not runtime errors (a stack overflow faults the process; a non-terminating test never returns). It is **batch-with-resume**, not process-per-test — the child runs the whole remaining suite sequentially (preserving the in-process runner's exact semantics) and the parent respawns only when a test actually kills it, resuming at the next index, so an all-passing suite costs one extra spawn + compile regardless of test count (+12ms on 4 tests, +22ms on 100) instead of ~16ms per test. The child reports each result over a **private pipe** the parent hands it in the **stdin slot** — deliberately NOT stdout/stderr, which stay the test's own and are never parsed. `take_protocol_channel` then does the load-bearing part: before any test code runs, the child takes the descriptor OUT of the stdin slot, duplicates it onto one that cannot be inherited (no-inherit `DuplicateHandle` on Windows; `fcntl(F_DUPFD_CLOEXEC, 3)` on POSIX — close-on-exec, and the minimum of 3 keeps it out of a std slot), and **puts the null device in the stdin slot**, disposing of the original with it: `SetStdHandle(STD_INPUT_HANDLE, NUL)` on Windows, `dup2(nul, 0)` on POSIX (`dup2` specifically, because `File::open` always sets `O_CLOEXEC` and `Command` does no `dup2` for an inherited stdin, so a close-then-reopen would hand the grandchild a *closed* fd 0 — safe but a footgun). Two earlier designs were defeated here, both through the OS rather than the language: a stderr protocol behind a per-run nonce in `argv` (a process can read its own command line, so a test forged `done` -> a green `0 passed, 0 failed` + exit 0), and then a reclaimed-but-not-reopened stdin slot (`proc_spawn` inherits stdin, so a grandchild got a writable handle and `cmd /c echo pass 1 >&0` forged a line — no builtin names a descriptor; inheritance needs none). After the reopen the channel has no name a program can reach and no slot a child can inherit, which holds against spawn routes not yet imagined instead of depending on every spawning builtin to null its stdin. `done` is additionally validated against what was reported — an integrity check, NOT a security control (a forger reports every index first and satisfies it). Reopening also restores a clean EOF for a test's `read_line`. Results are printed as they stream in (they arrive in index order), preserving live progress and ordering. On timeout it kills the process **TREE**, not just the child (`taskkill /T` on Windows, a process-group signal on POSIX): `sys_status`/`sys_output`/`proc_spawn` let a test spawn real processes, and a grandchild outlives a killed child *and* inherits its stderr pipe handle — killing only the child left `--timeout 3` running 14s, bounded by the grandchild rather than the deadline. The reader thread is deliberately detached rather than joined, so the deadline holds even if a kill fails; the child signals completion with an explicit `done` line rather than the parent inferring it from stderr EOF, which a grandchild can defer indefinitely (that would stall even a wholly passing suite). Batch completion is never inferred from EOF alone: on an EOF with no test in flight the parent resumes after the last test it actually saw reported, so a child that died having reported nothing can never be mistaken for one that ran everything (which would silently print `0 passed, 0 failed` and exit 0).
