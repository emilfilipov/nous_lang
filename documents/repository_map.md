# Repository Map

This file maps the repository layout and explains where to find core information. Keep it up to date whenever source files, directories, commands, tests, docs, or responsibilities change.

## Root

- [CLAUDE.md](../CLAUDE.md): operating guide for agents and contributors. Defines workflow, documentation rules, MCP/ClickUp/GitHub usage, testing expectations, Git rules, and where to find core language information.
- [README.md](../README.md): user-facing project overview, portable package installation, checksum verification, local build commands, CLI summary, documentation links, and the license.
- [LICENSE](../LICENSE): the MIT License Lullaby ships under (`license = "MIT"` in `[workspace.package]`, inherited by every crate). Permissive — free to use, modify, and redistribute; only the copyright/permission notice must be retained.
- `Cargo.toml`: Rust workspace manifest for the Lullaby compiler/toolchain crates. The lockfile records the pinned dependency graph, including serde/serde_json for bytecode artifact serialization.
- `.gitignore`: ignores local build outputs, caches, editor state, and generated artifacts once implementation begins.
- `.gitattributes`: normalizes repository text files to LF line endings.
- `crates/`: Rust implementation crates.
- `assets/`: brand and visual-identity assets. `assets/brand/` holds the canonical logo (`lullaby-mark.svg` one-ink lavender; `lullaby-icon.svg` filled tile), a raster wordmark (`lullaby-wordmark.png`, real Nunito ExtraBold on transparent, for GitHub/READMEs), the multi-size app/installer/favicon icon (`lullaby.ico` + `lullaby-icon-256/512.png`), the 1200×630 social-preview card (`lullaby-social-card.png`, Open Graph / Twitter / GitHub repo preview), the bundled Nunito typeface (`nunito.woff2`, OFL), and the generators `render_icons.py` (raster icons from the mark geometry), `render_wordmark.py` (raster wordmark), and `render_social_card.py` (social card).
- `examples/`: user-facing `.lby` examples packaged with the toolchain.
- `scripts/`: release packaging and verification scripts, including `build_windows_installer.py` (stages the payload and builds the branded WiX `.msi`).
- `installer/`: Windows installer authoring — `lullaby.wxs` (WiX v7 package: Program Files layout, PATH entry, Start Menu shortcuts, branded ARP icon + wizard), `notice.rtf` (the MIT license the wizard shows on its license page), `banner.bmp`/`dialog.bmp` (branded wizard bitmaps), and `render_installer_art.py` (regenerates the bitmaps from the mark geometry).
- `web/`: one-line web installers served from the site — `install.sh` (POSIX) and `install.ps1` (Windows), pure-ASCII bootstrappers that detect OS/arch, resolve the newest GitHub Release (stable, falling back to the newest prerelease), download the matching portable archive, verify its published `.sha256` (abort on mismatch), unpack to a per-user prefix, and wire PATH via the package's bundled helper; support `LULLABY_VERSION`/`LULLABY_PREFIX`/`LULLABY_REPO` and an `uninstall` mode. `web/README.md` documents usage and the checksum-verification security posture.
- `tests/`: shared `.lby` fixtures used by crate and CLI tests.
- `benchmarks/`: performance baseline / regression harness. `run_bench.ps1` imports MSVC `vcvars64` (so `cl` and `lullaby native` link), builds a pure-C reference (`fib.c`), runs the same workload (`fib.lby`) across every tier (C, native, ast, ir, bytecode), and normalizes to ns per operation for before/after diffs (`results_<label>.csv`, git-ignored). `run_loop.ps1` is the compute/loop-bound companion (`sum 0..N`, C ref `loopsum.c`). `run_gcd.ps1` exercises the native inline `gcd` Euclid `div`-loop (accumulates `gcd(i, 1071)`, C ref `gcdbench.c` compiled with `/DGCD_N=<N>`; every tier is checked against the C output) — native runs at C parity (~1.0×). Three aggregate-shape harnesses add array/string/transform coverage (each with a byte-identical `cl /O2` C reference; native correctness spot-checked at a reps that fits a 32-bit exit code, then timed at the large reps): `run_arraysum.ps1` (fat-pointer `array<i64>` reduction, C ref `arraysum.c`; native ~8.7× C), `run_strscan.ps1` (linear `for c in s` char-code checksum, C ref `strscan.c`; native ~49× C — Lullaby chars are UTF-8 code points, so each step decodes, unlike the C byte-walk; the accumulation *build* idiom is intentionally excluded as it is O(n²) and exhausts the no-free native bump heap past ~1000 chars), and `run_csvsum.ps1` (realistic CSV-integer parse+aggregate over `for c in s`, C ref `csvsum.c`; native ~29× C). The generated per-`N` `.lby`/`.exe`/`.csv` are git-ignored. `README.md` records the methodology and the current baseline; the optimization backlog is the ClickUp `18 Performance Optimization` list.
- `benchmarks/crosslang/`: the standing six-language token + performance benchmark (Lullaby, C, C++, Rust, Python, JavaScript). `corpus/<category>/{lullaby.lby,c.c,cpp.cpp,rust.rs,python.py,js.js}` holds real-world function families with identical function names across all six; every `lullaby.lby` passes `lullaby check`. `corpus_tokens.py` tokenizes the corpus (`o200k_base`) into `corpus_data.json`; `run_perf.ps1` times the `count_primes_below` workload into `perf_data.json`; `run_compile_speed.ps1` measures whole-program compile latency (Lullaby `build` vs `cl` vs `rustc`, optimized + debug) into `compile_data.json`; `assemble_report.py` inlines the Nunito font + all three JSON files into `report_template.html` to produce the self-contained `report.html` artifact. `gen_perf_driver.py` + `run_corpus_perf.ps1` are the whole-corpus compute harness: they turn every category's `main` into a hot N-iteration loop and time all three interpreter tiers (plus a native column for the purely-i64-scalar categories, folding each function's i64 result directly) across the full 26-category / 434-function set, per-iteration ns with the parse+launch floor removed. `run_benchmark.ps1` is the one-command runner (tokenize [+ `-Perf`] + assemble). `README.md` documents the pipeline and the rerun-on-change cadence; `SPEC.md` holds the original method rationale; `LULLABY_CHEATSHEET.md` guides corpus authors. Rerun and republish the artifact whenever the language, optimizer, or corpus changes.
- `documents/`: core language documents and planning material.

- `documents/versioning.md`: the release versioning scheme — display/tag form `MAJOR.PATCH-STATUS` (e.g. `1.0-preview`) with statuses `stable`/`preview`/`experimental`, its mapping to Cargo semver (`MAJOR.PATCH.0[-STATUS]`; `stable` = no prerelease suffix), release-tag format, and how the version is centralized in `[workspace.package]` and surfaced by `lullaby --version`.
- `documents/numeric_types_design.md`: implementation-grade design for the full scalar numeric lattice (`i8`–`i64`, `u8`–`u64`, `usize`/`isize`, `f32`/`f64`, `byte` = `u8`): erased-width runtime representation with per-operation normalization, typed literal suffixes + range checking, total conversion/cast rules, and wrapping-by-default arithmetic with checked/saturating/wrapping variants, all at backend parity.
- `documents/ffi_design.md`: implementation-grade FFI / C-ABI design — `extern`/`export fn` surface, Win64/System V/AArch64 calling conventions, full type marshalling (scalars/pointers/`repr(C)` structs/`string`↔`cstr`), dynamic/static library linking, callbacks, C-header generation, the native-only parity model (interpreters reject `extern` calls), and proposed diagnostics.
- `documents/atomics_design.md`: implementation-grade atomics & memory-ordering design — per-width atomic handle types, load/store/swap/CAS/fetch-op operations, the five memory orderings with checker rules, fences, thread-local storage, `Arc<std::sync::atomic>`-backed interpreter semantics as the reference spec, native atomic-instruction lowering, and proposed diagnostics.
- `documents/wasm_heap_design.md`: implementation-grade design for the WASM linear-memory (heap) phase — byte layouts for every heap type, a production allocator (`__alloc`/`__free`/`__realloc`), heap-type + `match`/option/result lowering, host imports (console + minimal DOM) with a dependency-free JS loader, extended `lullaby wasm` CLI, and the node/wasmtime parity plan.
- `documents/distribution_design.md`: implementation-grade "ease of access" design — Lullaby branding and the install command surface, the toolchain bundle + on-disk layout, Windows MSI/winget/NSIS, Linux `.deb`/`.rpm`/AppImage, macOS tarball + Homebrew (with the Apple signing/notarization caveat), the one-line web installer, the release-automation CI matrix, and `lullaby new` first-run UX.
- `documents/platform_io_design.md`: implementation-grade design for the 1.0 platform I/O primitives — monotonic + wall clocks and `sleep`, an OS CSPRNG (`os_random`), process spawn/exec/pipe (`Value::Process`), and a minimal cross-platform non-blocking poll primitive (`Value::Poller` over epoll/kqueue/IOCP), with the structural-parity testing strategy for non-deterministic results and proposed diagnostics `L0430`–`L0435`.
- `documents/closures_capture_design.md`: implementation-grade design for environment-capturing closures (the follow-up to the delivered non-capturing first-class functions) — a `fn PARAMS -> EXPR` lambda typed as an existing `fn(...) -> R`, eager by-value capture (snapshotting value-semantic collections, sharing reference-semantic handles), a `Value::Closure` handle threaded through all interpreters at parity, IR lowering (interpreter-first, native/WASM deferred), the `spawn`/`parallel_map` capture unlock, and proposed diagnostics `L0440`–`L0442`.
- `documents/error_propagation_design.md`: implementation-grade design for a postfix `?` error-propagation operator on `option`/`result` — semantics + enclosing-return compatibility, postfix parsing, and (the crux) a statement-level A-normal-form desugaring that hoists each `?` into a `let` + typed success temporary + early-return statement-`match`, so no expression ever hides a non-local return and no new IR node or backend change is needed; proposed diagnostics `L0427`–`L0429`.
- `documents/stdlib_modules_design.md`: implementation-grade design for the 1.0 stdlib "honesty pass" — the primitive-vs-module boundary (a 3-part test), a full row-by-row reclassification of the current prelude (HTTP → `http`, string conveniences → `strings`, transcendental math → `math`), stdlib module mechanics over the existing `import`/`pub` system, the non-breaking HTTP migration plan, on-disk/bundle layout, and proposed diagnostics `L0345`/`L0346`.
- `documents/portable_package_ci_workflow.yml`: GitHub Actions workflow template for explicit target-triple portable-package verification on Ubuntu, Windows, macOS ARM64, and macOS x64 runners. Copy it to `.github/workflows/portable-package.yml` from an authenticated session with GitHub `workflow` scope to activate it.

Canonical decision: `execution_tiers_and_1_0_scope.md`.

- `documents/struct_design.md`: token-minimalist struct/record design — indentation-only `struct` declaration with `name type` fields, positional and named-field construction, `.` field access, field mutation, nominal typing, and the first-increment scope versus deferred features.
- `documents/enum_and_match_design.md`: token-minimalist enum/tagged-union and `match` design — indentation-only `enum` declaration with positional payload types, bare-variant construction resolved semantically, exhaustive `match` with payload binding and wildcard, nominal typing, backend-parity representation, and sequencing toward `option`/`result`.
- `documents/option_result_design.md`: `option<T>`/`result<T, E>` design as built-in generic enums — variants `some`/`none`/`ok`/`err`, context-directed type inference at `let`/return/`match`/call-argument sites (a nested constructor in a call argument infers from the callee's parameter type, mirrored in the IR lowerer), the multi-argument generic-type foundation (`generic_args`, comma-separated generic spelling), backend-parity representation via `Value::Enum`, and `L0386`.
- `documents/closures_design.md`: first-class function values design — `fn(T) -> R` function types, referring to a top-level function by name as a value, calling function-typed locals, `Value::Func` at backend parity, and `L0390`; environment-capturing closures deferred to a follow-up increment.
- `documents/modules_design.md`: modules/imports/visibility design — file-as-module, `pub` to export, flat unqualified `import NAME`, a frontend-only loader that merges all files into one `Program` (backends unchanged), no-shadowing rule, and `L0391`/`L0392`/`L0393`; plus the `lullaby.json` project manifest (multiple `src` directories and local-path dependencies, resolved across the project and its dependencies, `L0343`). Dotted paths, qualified access, per-field visibility, and remote/registry dependency fetching deferred.
- `documents/standard_library.md`: the prelude — a single catalog of all compiler-provided built-in types and functions (always in scope, no import needed): conversions, collections (`list`/`map`), strings, math, streams/file/system I/O, memory/references, and error handling.
- `documents/generics_design.md`: user-defined generics design — generic functions `fn name<T> ...` with call-site type inference, erased at runtime (backends unchanged), inference mirrored in the IR lowerer, and `L0394`/`L0395`/`L0396`; explicit type arguments (turbofish) deferred (trait bounds are now implemented — see [traits_design.md](traits_design.md)). Stage 1 of generic user *types* has shipped: a single-parameter generic `struct Box<T>` with a scalar `T` — parser `type_params` on `StructDecl`, semantics scoping/substitution/inference-directed construction (`L0454`/`L0455`), type erasure on the three interpreters, and a clean native/WASM skip (`L0339`); generic enums, methods on generic types, heap-`T`, multi-param, and bounds are the next stages.
- `documents/traits_design.md`: traits/interfaces design — `trait Name` method signatures with a `self` receiver, `impl Trait for Type`, receiver-type method dispatch via a shared impl table on all backends, and trait bounds on generics (`<T: Ord>`); records `L0398`/`L0399`/`L0400` and defers default bodies, trait objects, associated types, and supertraits.
- `documents/wasm_backend_design.md`: WebAssembly backend design — compile the typed IR's scalar subset (`i64`/`f32`/`f64`/`bool`/`char`/`byte` plus the fixed-width integer kinds, arithmetic/comparison/logic including single-precision `f32` ops and the `to_f32`/`to_f64` conversions, `if`/`while`/`loop`/`for`, calls, locals) to a binary `.wasm` module with std-only encoding, a `lullaby wasm` CLI command, and node/wasmtime-gated execution parity against the interpreter. The first linear-memory step is landed: the module exports a `"memory"` (min 1 page), declares a mutable `i32` bump-pointer global, seeds a Data section, emits an internal `__alloc(size i32) -> i32` bump helper, and imports three host functions from module `env` surfaced to Lullaby as builtins — `env.log_i64` as `wasm_log(x i64) -> void`, `env.console_log(ptr i32, len i32)` as `console_log(s string) -> void`, and `env.dom_set_text(id_ptr i32, id_len i32, text_ptr i32, text_len i32)` as `dom_set_text(id string, text string) -> void` — the JS/DOM browser-host interop layer (imports occupy the low function indices, so internal function indices/call targets/exports are shifted by the import count, driven by `IMPORT_FUNC_COUNT`). **Heap types are now lowered into linear memory:** `string` values are `i32` pointers to `[char_len i32][byte_len i32][utf8 bytes]` with literals interned once into the Data section (and `len(s)`); `struct` values are pointers to one 8-byte slot per field with positional construction/field read/field mutation; fixed `array` literals are pointers to `[len i32][8-byte elem slots]` with construction, indexed read/write, and `len(a)`; field/element slots may be scalars or nested pointers, and `__alloc` builds structs/arrays at runtime. The JS/DOM string imports pass each string's `(ptr, len)` decoded out of `memory`; on the interpreters `console_log` prints the string and `dom_set_text` prints `id=text` so parity holds. Enum/tagged-union + `match` lowering, `option`/`result` (incl. string and one-level mutable-heap payloads), `list`/`map` (scalar/string/one-level-mutable elements, scalar and content-equality string keys), runtime string construction (`+` concat, `to_string`, `substring`/`find`/predicates), the overflow-aware arithmetic builtins, the scalar math builtins (`sqrt`/`abs` on `f64`, `abs` on `i64`, and `min`/`max`/`gcd`/`sign`/`clamp` on `i64` — the `f64` cases of `min`/`max`/`sign`/`clamp` are deferred like native), and value-producing `if`/`else` are all landed; a free-list allocator (bump-only, no reclamation), `to_string(float)`, the transcendental math builtins (`sin`/`cos`/`floor`/…, `pow`), the remaining string builders (`replace`/`upper`/`lower`/`split`/`join`), deeper-nested/`array`/`map`-element collections, and a richer DOM interop surface remain deferred.
- `documents/lsp_design.md`: Language Server Protocol server design — the `lullaby_lsp` crate and `lullaby lsp` command, JSON-RPC-over-stdio `Content-Length` framing, the pure `handle_message` request core, advertised capabilities (`textDocumentSync=1` full, `documentFormattingProvider`, `hoverProvider`, `definitionProvider`), lifecycle (`initialize`/`initialized`/`shutdown`/`exit`), full-text document sync, diagnostics from the reused lex/parse/semantic pipeline mapped to 0-based LSP ranges, canonical-formatter formatting edits, `analysis.rs` hover (function signature / local type / struct-enum declaration / builtin description from the checked metadata) and go-to-definition (declaration or `let`/parameter binding `Location`), testing strategy, and deferred features (completion/references, incremental sync, project-aware analysis).
- `documents/lullaby_control_structures.md`: control flow and operators, including the current if/while/loop rules plus planned switches, try/catch, coroutines, arithmetic/logical/bitwise operators, collections, conversions, and utility operations.
- `documents/lullaby_input_output.md`: I/O and concurrency model, including files, streams, memory-mapped files, threads, processes, async, multiplexing, IPC, sockets, and performance strategies.
- `documents/lullaby_error_handling.md`: error model, compact error tokens, compile-time and runtime categories, throw/catch/recovery behavior, diagnostics, and compiler integration.
- `documents/lullaby_compilation_architecture.md`: compiler architecture from tokenization through semantic analysis, IR, optimization, code generation, linking, and binary verification.
- `documents/brand_guidelines.md`: Lullaby visual identity — the plain lavender `lullaby` wordmark (with an `l`-monogram filled tile for small-size app/installer/favicon), the soft-pastel palette with hex tokens and roles, Nunito typography, voice and tagline, and the `assets/brand/` asset inventory.
- `documents/packaging_roadmap.md`: distribution-channel roadmap — the shipped Windows `.msi` and portable archives, plus planned release automation, one-line web installer, winget, Homebrew, Linux `.deb`/`.rpm`, macOS tarball, and code signing, each with its build machinery and the owner-provided prerequisites.

## Examples

- `examples/README.md`: user-facing instructions for valid and invalid example programs.
- `examples/valid/`: executable `.lby` examples for calculator, arrays/control flow, file I/O, Windows system command status, `rotation.lby` (2D vector rotation exercising structs and the trigonometric math builtins), `primes.lby` (count primes below 50 with a trial-division helper), `collatz.lby` (Collatz stopping times), `traffic_light.lby` (an enum + `match` state machine), `word_count.lby` (string `split`/`len` word statistics), and `inventory.lby` (a `map<string, i64>` ledger read via `match` on `map_get`'s `option`). Every top-level `.lby` here is auto-run by the CLI test `runs_user_facing_valid_examples`.
- `examples/valid/tests_demo/tests_demo.lby`: a runnable language-level test suite (no `main`) with four `test_*` functions (`test_arith`, `test_strings`, `test_option`, `test_bool_return`) that all pass via the `assert` builtin. Run it with `lullaby test examples/valid/tests_demo/tests_demo.lby` (prints `4 passed, 0 failed`). It lives in a subdirectory so the single-file example runner and parity harness ignore it.
- `examples/valid/project_demo/` and `examples/valid/mathx/`: a sample multi-package build. `project_demo/lullaby.json` names `src/main.lby` as its `entry`, `src` as its source dir (holding `geometry.lby`), and a local path dependency `mathx` (`../mathx`). `main.lby` imports both `mathx` (from the dependency) and `geometry` (from its own `src`) and computes `square(5) + cube(2) + triple(4) = 45`. `mathx/` is an entry-less library project (`lullaby.json` + `src/mathx.lby`). Run with `lullaby run examples/valid/project_demo` (or the `lullaby.json` path). Both live in subdirectories so the single-file example/parity runners ignore them.
- `examples/valid/fullstack/`: an end-to-end full-stack web demo proving one Lullaby domain module compiles into two targets. `shared.lby` is a `pub` module (`classify(title) -> string`, `priority_score(title) -> i64`, `is_valid_title`, `detail_threshold`) kept inside the WASM-eligible surface (scalars + strings only) so it lowers to `.wasm` for the browser AND runs unchanged in the interpreter-based HTTP backend. `frontend.lby` imports `shared`, runs the shared classification/scoring on sample titles, and renders results via the `console_log`/`dom_set_text` host builtins (compiles fully with `lullaby wasm frontend.lby`; interpreter `main` = 4). `index.html` is a self-contained host page (no CDN/remote assets, relative `frontend.wasm` load) that supplies the `env.console_log`/`env.dom_set_text` imports and calls the exported `main`. `backend.lby` is a bounded HTTP/1.1 server (mirroring `http_server/server.lby`) that imports `http` and `shared` and answers `/classify` with the shared label/score for a sample title. `http.lby` is the `http_server` framework module (copied so the flat `import http` resolves in this directory). `examples/valid/fullstack/README.md` documents building the frontend, opening the page locally, and running/curling the backend. Scalars map as before (`i64`/`f32`/`f64`/`bool`/`char`/`byte`; `f32` is WASM's native single-precision `f32` value type and local). It now compiles **32-bit floats (`f32`) to parity with `f64`**: single-precision arithmetic `+ - * /` (`f32.add/sub/mul/div`, so f32 stays single precision and is bit-identical to the interpreter's real `f32`), IEEE-754 NaN-aware comparisons `< <= > >= == !=` (`f32.lt/le/gt/ge/eq/ne`, false on NaN for relational + `==`, true for `!=`), `f32.const` literals rounded to single precision, and the float conversions `to_f32(x f64) -> f32` (`f32.demote_f64`, round) and `to_f64(x f32) -> f64` (`f64.promote_f32`, widen) recognized and inlined as builtin calls (not real calls) — matching the interpreters and the native backend bit-for-bit (node execution parity for `run_f32.lby` → 3 and `native_floats.lby` → 9). Because the IR annotates a float *arithmetic* node `i64`, the operand float width for arithmetic/comparison is derived structurally from the reliably-typed leaves (float literals/locals and `to_f32`/`to_f64`) via `float_val_type_of`, mirroring the native backend's `float_width_of_expr`; f32/f64 struct fields and array elements round-trip through single/double-precision `f32.store`/`f32.load` (4-byte) slots. Within eligible functions it now also compiles the **fixed-width integer types** (`i8`/`i16`/`i32`/`u8`/`u16`/`u32`/`u64`/`isize`/`usize`), each held in an `i64` local as the interpreter's normalized cell: wrapping `+ - *` and `/` re-normalized to the width (sign-extend via `i64.extend8_s`/`extend16_s`/`extend32_s` for signed 8/16/32 kinds, an `i64.and` width mask for unsigned kinds, no-op for the 64-bit kinds), signedness-correct comparison (`lt_u`/`lt_s` etc.) and division (`div_u`/`div_s`, trapping on zero like the `i64` path), bitwise `& | ^` and `~` (as `xor -1`) re-normalized, shifts with the count masked to `width-1` and right shift logical-for-unsigned (`shr_u`) / arithmetic-for-signed (`shr_s`), and the `to_<T>`/`to_i64` conversions inlined as a width-normalize (not real calls) — matching the interpreters and the native backend bit-for-bit (node execution parity). Heap values beyond the pointer types below, the option-returning `checked_*`/`saturating_*`/`wrapping_*`/`to_string` builtins, and the float math builtins (`sqrt`/`sin`/…) remain deferred and skip gracefully. **Heap types are `i32` pointers into linear memory**: a `string` points to `[char_len i32][byte_len i32][utf8 bytes]` with literals interned once into the Data section (offset math via `RESERVED_BASE`) and `len(s)` lowered to `i32.load` + `i64.extend`; a `struct` points to one 8-byte slot per field with positional construction (a `Call` named for the struct), `.field` read, and `p.field = v` mutation; a fixed `array` literal points to `[len i32][8-byte elem slots]` with construction, `a[i]` read, `a[i] = v` write (WASM traps on OOB memory access), and `len(a)`; field/element slots may be scalars or nested pointers, and `struct`/`array` construction calls `__alloc`. The module still exports `"memory"` (min 1 page), declares a mutable `i32` bump-pointer global (now initialized past the Data-seeded reserved region AND the whole string-literal pool), emits the internal `__alloc(size i32) -> i32` bump helper, and imports three host functions from module `env` — `env.log_i64 (func (param i64))` exposed as `wasm_log(x i64) -> void`, `env.console_log (func (param i32 i32))` exposed as `console_log(s string) -> void`, and `env.dom_set_text (func (param i32 i32 i32 i32))` exposed as `dom_set_text(id string, text string) -> void` (the JS/DOM browser-host interop layer; a call to any of them lowers to `call` of its import, and the string imports push each string's `(ptr, len)` — the pointer plus the `[len i32]` header — decoded out of `memory` by the host); imports occupy the low function indices, so internal function indices, call targets, and function-export indices are shifted by the import count (`IMPORT_FUNC_COUNT`, now 3). It also compiles **enum values and `match`** for enums with scalar payloads: an enum value is an `i32` pointer to a `[tag i32 (padded to 8)][payload slots…]` linear-memory record — the tag is the variant's index in declaration order (matching the interpreters, which dispatch `match` by variant name against that same ordered layout) and there is one 8-byte payload slot per position, sized for the widest variant. Construction (`some(x)`/`none`/`ok(x)`/`err(e)` and user `Variant(payload…)`, all emitted by the IR lowerer as positional `Call`s) `__alloc`s the record, stores the tag at offset 0, and stores each payload; `match` loads the tag (`i32.load` at offset 0) and dispatches with a chain of `i32.eq` + typed `if`/`else` blocks (a `Wildcard` arm becomes the final `else`; with exhaustiveness guaranteed by semantics the last variant arm is emitted unconditionally so a value match always leaves a value), binding each arm's payload slots into locals before its body. This covers the built-in `option<T>`/`result<T, E>` when `T`/`E` are scalar and user enums whose every variant payload is scalar, including matching a call that returns `option<scalar>`/`result<scalar,scalar>`. It also **passes and returns scalar-field aggregates across function boundaries**: a `struct` (of scalars, nested scalar structs, or fixed scalar arrays), a fixed scalar `array`, or a scalar-payload `enum` may be a parameter, a return value, or a call argument — the `i32` pointer is passed/returned directly, but Lullaby value semantics are preserved by **deep-copying each mutable aggregate argument at the call site** (`is_mutable_aggregate` + `emit_deep_copy`): a fresh record is `__alloc`'d and every word copied, recursing into nested mutable aggregate fields/elements (a struct copies each field, an array copies its `[len]` header + elements in a runtime loop, an enum copies its `[tag][payload]` record word-for-word since payloads are scalar), exactly mirroring the interpreters' recursive `Value::clone`; an immutable `string` argument is shared, not copied, and a returned aggregate is the callee's own fresh record. Functions using an enum with a **heap** payload (`string`/`list`/`array`/`map` — notably `result<i64, string>`), aggregates containing `list`/`map`, runtime string construction, or other unsupported builtins are recorded as skipped with a reason and still run on the interpreters; no eligible function yields `WasmError` with code `L0338`. Structural + heap + enum/match encoder tests (magic/version, section order and counts, import/export presence, shifted call targets, string-literal Data seeding, struct/array construction eligibility, struct/array param+return compilation, the copy-on-pass `__alloc` for a forwarded struct/array argument and the no-copy of an immutable string argument, enum-layout discriminant ordering, the tag-load + typed-`if` dispatch, tag-store construction, and heap-payload-enum skip) live in `src/wasm.rs`. `src/native_object.rs` also carries the extended native program emitter (the i64-scalar native mirror of the WASM subset): `emit_native_program(&BytecodeModule) -> Result<NativeProgram, NativeProgramError>` lowers every function whose params and return are all `i64` (at most four Win64 register params) to x86-64 machine code with a stack-machine codegen over `rax` — integer literals, params/`let`/`for` locals in `[rbp - slot]` frame slots, `+ - * / %` (signed `idiv`+`cqo`), scalar bitwise `& | ^ << >>` (`and`/`or`/`xor rax`, `shl`/`sar` with the count masked to 63), comparisons, short-circuiting `and`/`or`/`not`, `if`/`elif`/`else`, `while`, `loop` with `break`/`continue`, range `for`, `return`, tail expressions, and calls to other compiled i64 functions. It also **auto-vectorizes** three array-loop shapes over contiguous `array<i64>` into SSE2 packed loops (two lanes/iter, scalar tail, bit-for-bit identical to the scalar loop): sum reductions (`paddq`), element-wise maps `c[i]=a[i]±b[i]` (`paddq`/`psubq`), and bitwise reductions/maps for `& | ^` (`pand`/`por`/`pxor`, seeded with the operator identity); the exact matcher falls back to the scalar loop for any other shape. Within these otherwise-`i64` functions it now also compiles the **fixed-width integer types** (`i8`/`i16`/`i32`/`u8`/`u16`/`u32`/`u64`/`isize`/`usize`), each held in a register as the interpreter's normalized `i64` cell: wrapping `+ - * /` re-normalized to the width, signedness-correct comparison (unsigned/signed condition codes) and division (`div`/`idiv`), bitwise `& | ^ ~`, shifts with the count masked to the width and right shift logical-for-unsigned / arithmetic-for-signed, and the `to_<T>`/`to_i64` conversions emitted inline as a width-normalize — matching the interpreters bit-for-bit (native link-and-run parity). Within these otherwise-`i64` functions it now also compiles **floating-point values** (`f64` and `f32`) using SSE scalar (XMM) registers: `f64` held as a `double`, `f32` kept rounded to single precision after each op (matching the interpreter's real `f32` storage), spilled to 8-byte frame slots when a temporary must survive another evaluation; IEEE-754 arithmetic `+ - * /` (`addsd/subsd/mulsd/divsd` for `f64`, `addss/subss/mulss/divss` for `f32`, so division by zero yields inf/NaN and never traps); ordered comparisons `< <= > >= == !=` (`ucomisd`/`ucomiss` with unordered-aware set conditions, so a NaN operand makes every relational compare and `==` false and `!=` true, producing an `i64`/`bool`); and the `to_f32(x f64) -> f32` (`cvtsd2ss`, round) / `to_f64(x f32) -> f64` (`cvtss2sd`, widen) conversions recognized and inlined; float literals are materialized via a GPR→XMM `movq`/`movd` (no `.rdata` constant). The signature constraint is unchanged (all params + return `i64`), so functions with a float parameter or return type, float heap values, and the transcendental/math builtins (`sqrt`/`sin`/`floor`/…) remain deferred and skip gracefully. Heap values, the option-returning `checked_*` builtins, and the `saturating_*`/`wrapping_*` builtins remain deferred and skip gracefully. The subset now also lowers **stack-allocated scalar aggregates** as function locals: all-`i64` (optionally nested) structs and fixed-length `i64` arrays are laid out contiguously in the frame (each value a run of 8-byte words sized by a `NativeType` flattening), with positional/named struct construction, field read/mutation, array literal construction, constant- and dynamic-index array read/write (a dynamic index computes `rbp - (base + 8*const_words) - 8*elem_words*index` into `rcx`; native array bounds are not trapped for this increment), and a constant-folded `len` on a fixed native array. Scalar-field aggregates (structs of scalars, fixed scalar arrays, scalar-payload enums) also cross function boundaries as parameters, returns, and call arguments via a hidden-return-pointer + by-pointer-argument ABI with copy-in value semantics (a callee mutating an aggregate parameter never affects the caller's copy), matching the interpreters; heap-containing aggregates and a top-level float scalar parameter/return remain deferred. It also lowers **stack-allocated enums with scalar payloads and `match`**: the built-in generics `option<T>`/`result<T, E>` (for scalar `T`/`E`) and user enums are laid out as a tag word (the variant's discriminant index in the interpreters' variant order) plus a shared payload region sized to the widest variant; construction writes the tag and payload words, and `match` reloads the tag and dispatches per arm with `cmp`/`jne` (binding scalar payloads into arm-scoped locals, honoring the `_` wildcard, and emitting `ud2` on the impossible fallthrough of a wildcard-free exhaustive match). The scrutinee may be an enum local or a freshly-constructed enum; heap-payload enums (`result<i64, string>`), enums as parameters/returns, and a `match` on an enum-returning call are deferred and skip gracefully. It also takes the **first heap step**: `len("...")` over a string literal is lowered (not constant-folded) by interning the literal's bytes into a read-only `.rdata` section (`__str{i}`, deduped), copying them into a `.bss` bump-allocated heap region at runtime, and scanning the heap copy for its byte length — emitting two `.text` helper functions (`__lullaby_alloc` bump allocator; `__lullaby_strlen_copy` allocate+copy+scan) plus the `.bss` bump-pointer cell `__lullaby_heap_next` and 64 KiB region `__lullaby_heap_base`, with cross-section `IMAGE_REL_AMD64_REL32` relocations from the `lea`/`call` sites (only ASCII string literals, so the byte length equals the interpreter's char-count `len`; there is no native string value beyond the `len` argument). String-free programs keep the exact prior single-`.text` layout byte-for-byte. It writes a COFF object holding an entry stub (`_lullaby_start`, which calls `main`, moves the result into `ecx`, and calls the imported `ExitProcess`) plus every function, with `IMAGE_REL_AMD64_REL32` relocations for calls and long symbol names in the COFF string table; ineligible functions are skipped with a reason and no eligible function yields `NativeProgramError` with code `L0339`. It also exposes **C-callable exports**: an `export fn` (tracked via `export_functions` on `IrModule`/`BytecodeModule`) is emitted like any i64-scalar function, and because every compiled function symbol is already storage-class `EXTERNAL` defined in `.text` under its plain name, a C caller declaring `extern <ret> NAME(<args>);` links against it. An export-only program (no `main`) emits a **library object**: the entry stub and its `ExitProcess` dependency are omitted (`NativeProgram.entry_symbol` empty) so a C `main` can link against it without a `main` collision; a program with both `main` and exports keeps the stub. It also **calls C via `extern fn`** (C-ABI FFI): a body-less `extern fn` (tracked via `extern_functions`, with each C-ABI signature carried in `extern_signatures` on `IrModule`/`BytecodeModule`, both serde-defaulted) lowers to a `call` of an undefined external symbol (`IMAGE_REL_AMD64_REL32`, section 0), linked against the C runtime import library `ucrt.lib`. Marshalling covers the integer-class C scalars — every fixed-width integer (`i8`…`u64`, `isize`/`usize`) plus `bool`/`char`/`byte`: arguments pass in the low bits of `rcx`/`rdx`/`r8`/`r9` (already width-normalized in the interpreter's cell model), and a narrow C return is re-normalized in `rax` (sign-extend for signed kinds, zero-extend for unsigned). `f32`/`f64` externs need XMM argument routing and are deferred, so a caller of a float extern demotes to the interpreters (which reject the extern call with `L0423`). The single-function literal prototype `emit_coff_object` is retained for the checked-in object snapshots. It also lowers **inline assembly**: an `unsafe` `asm` statement (an `IrStmt::Asm`/`BytecodeInstruction::Asm` carrying raw `Vec<u8>` bytes, native-only) is emitted verbatim into the current function's `.text`; a tail `asm` is followed by the normal Win64 epilogue so `rax` is returned intact (the programmer must not emit their own `ret`). It also emits optional **CodeView source-line debug info**: `emit_native_program_with_debug(module, Some(DebugOptions { source_file }))` adds a `.debug$S` section (C13 signature; `DEBUG_S_SYMBOLS` `S_COMPILE3`, one `DEBUG_S_LINES` per function with `SECREL32`+`SECTION` relocations to its `.text` symbol mapping the entry offset to `BytecodeFunction.span.line`, `DEBUG_S_FILECHKSMS`, `DEBUG_S_STRINGTABLE`) so a linker-built PDB lets a debugger break at a function and show its `.lby` line; it is opt-in (`lullaby native --debug`/`-g`) and leaves the default object bytes byte-for-byte unchanged (`emit_native_program` is the no-debug wrapper). Structural tests for the extended emitter (including struct locals, fixed-array const/dynamic indexing, non-i64-field struct rejection, aggregate layout sizing, the `.rdata`/`.bss` string-constant layout with dedup, verbatim `asm` byte emission, and the CodeView `.debug$S` per-function line table with byte-for-byte default parity) live in `src/native_object.rs`.
