# Repository Map

This file maps the repository layout and explains where to find core information. Keep it up to date whenever source files, directories, commands, tests, docs, or responsibilities change.

## Root

- [CLAUDE.md](../CLAUDE.md): operating guide for agents and contributors. Defines workflow, documentation rules, MCP/ClickUp/GitHub usage, testing expectations, Git rules, and where to find core language information.
- [README.md](../README.md): user-facing project overview, portable package installation, checksum verification, local build commands, CLI summary, and documentation links.
- `Cargo.toml`: Rust workspace manifest for the Lullaby compiler/toolchain crates. The lockfile records the pinned dependency graph, including serde/serde_json for bytecode artifact serialization.
- `.gitignore`: ignores local build outputs, caches, editor state, and generated artifacts once implementation begins.
- `.gitattributes`: normalizes repository text files to LF line endings.
- `crates/`: Rust implementation crates.
- `examples/`: user-facing `.lby` examples packaged with the toolchain.
- `scripts/`: release packaging and verification scripts.
- `tests/`: shared `.lby` fixtures used by crate and CLI tests.
- `documents/`: core language documents and planning material.
- `offline_docs/`: self-contained browser documentation bundle that can be opened directly from disk.

Dotted paths, qualified access, per-field visibility, and remote/registry dependency fetching deferred.
- `documents/standard_library.md`: the prelude — a single catalog of all compiler-provided built-in types and functions (always in scope, no import needed): conversions, collections (`list`/`map`), strings, math, streams/file/system I/O, memory/references, and error handling.
- `documents/generics_design.md`: user-defined generics design — generic functions `fn name<T> ...` with call-site type inference, erased at runtime (backends unchanged), inference mirrored in the IR lowerer, and `L0394`/`L0395`/`L0396`; generic user types and explicit type arguments deferred (trait bounds are now implemented — see [traits_design.md](traits_design.md)).
- `documents/traits_design.md`: traits/interfaces design — `trait Name` method signatures with a `self` receiver, `impl Trait for Type`, receiver-type method dispatch via a shared impl table on all backends, and trait bounds on generics (`<T: Ord>`); records `L0398`/`L0399`/`L0400` and defers default bodies, trait objects, associated types, and supertraits.
- `documents/wasm_backend_design.md`: WebAssembly backend design — compile the typed IR's scalar subset (`i64`/`f32`/`f64`/`bool`/`char`/`byte` plus the fixed-width integer kinds, arithmetic/comparison/logic including single-precision `f32` ops and the `to_f32`/`to_f64` conversions, `if`/`while`/`loop`/`for`, calls, locals) to a binary `.wasm` module with std-only encoding, a `lullaby wasm` CLI command, and node/wasmtime-gated execution parity against the interpreter. The first linear-memory step is landed: the module exports a `"memory"` (min 1 page), declares a mutable `i32` bump-pointer global, seeds a Data section, emits an internal `__alloc(size i32) -> i32` bump helper, and imports three host functions from module `env` surfaced to Lullaby as builtins — `env.log_i64` as `wasm_log(x i64) -> void`, `env.console_log(ptr i32, len i32)` as `console_log(s string) -> void`, and `env.dom_set_text(id_ptr i32, id_len i32, text_ptr i32, text_len i32)` as `dom_set_text(id string, text string) -> void` — the JS/DOM browser-host interop layer (imports occupy the low function indices, so internal function indices/call targets/exports are shifted by the import count, driven by `IMPORT_FUNC_COUNT`). **Heap types are now lowered into linear memory:** `string` values are `i32` pointers to `[char_len i32][byte_len i32][utf8 bytes]` with literals interned once into the Data section (and `len(s)`); `struct` values are pointers to one 8-byte slot per field with positional construction/field read/field mutation; fixed `array` literals are pointers to `[len i32][8-byte elem slots]` with construction, indexed read/write, and `len(a)`; field/element slots may be scalars or nested pointers, and `__alloc` builds structs/arrays at runtime. The JS/DOM string imports pass each string's `(ptr, len)` decoded out of `memory`; on the interpreters `console_log` prints the string and `dom_set_text` prints `id=text` so parity holds. Enum/tagged-union + `match` lowering, `option`/`result`, `list`/`map`, runtime string construction, a free-list allocator, and a richer DOM interop surface remain deferred.
- `documents/lsp_design.md`: Language Server Protocol server design — the `lullaby_lsp` crate and `lullaby lsp` command, JSON-RPC-over-stdio `Content-Length` framing, the pure `handle_message` request core, advertised capabilities (`textDocumentSync=1` full, `documentFormattingProvider`, `hoverProvider`, `definitionProvider`), lifecycle (`initialize`/`initialized`/`shutdown`/`exit`), full-text document sync, diagnostics from the reused lex/parse/semantic pipeline mapped to 0-based LSP ranges, canonical-formatter formatting edits, `analysis.rs` hover (function signature / local type / struct-enum declaration / builtin description from the checked metadata) and go-to-definition (declaration or `let`/parameter binding `Location`), testing strategy, and deferred features (completion/references, incremental sync, project-aware analysis).
- `documents/lullaby_control_structures.md`: control flow and operators, including the current alpha if/while/loop rules plus planned switches, try/catch, coroutines, arithmetic/logical/bitwise operators, collections, conversions, and utility operations.
- `documents/lullaby_input_output.md`: I/O and concurrency model, including files, streams, memory-mapped files, threads, processes, async, multiplexing, IPC, sockets, and performance strategies.
- `documents/lullaby_error_handling.md`: error model, compact error tokens, compile-time and runtime categories, throw/catch/recovery behavior, diagnostics, and compiler integration.
- `documents/lullaby_compilation_architecture.md`: compiler architecture from tokenization through semantic analysis, IR, optimization, code generation, linking, and binary verification.
- `documents/repository_map.md`: this file. Use it as the first navigation aid and update it with repository changes.

- `examples/valid/tests_demo/tests_demo.lby`: a runnable language-level test suite (no `main`) with four `test_*` functions (`test_arith`, `test_strings`, `test_option`, `test_bool_return`) that all pass via the `assert` builtin. Run it with `lullaby test examples/valid/tests_demo/tests_demo.lby` (prints `4 passed, 0 failed`). It lives in a subdirectory so the single-file example runner and parity harness ignore it.
- `examples/valid/project_demo/` and `examples/valid/mathx/`: a sample multi-package build. `project_demo/lullaby.json` names `src/main.lby` as its `entry`, `src` as its source dir (holding `geometry.lby`), and a local path dependency `mathx` (`../mathx`). `main.lby` imports both `mathx` (from the dependency) and `geometry` (from its own `src`) and computes `square(5) + cube(2) + triple(4) = 45`. `mathx/` is an entry-less library project (`lullaby.json` + `src/mathx.lby`). Run with `lullaby run examples/valid/project_demo` (or the `lullaby.json` path). Both live in subdirectories so the single-file example/parity runners ignore them.
- `examples/valid/fullstack/`: an end-to-end full-stack web demo proving one Lullaby domain module compiles into two targets. `shared.lby` is a `pub` module (`classify(title) -> string`, `priority_score(title) -> i64`, `is_valid_title`, `detail_threshold`) kept inside the WASM-eligible surface (scalars + strings only) so it lowers to `.wasm` for the browser AND runs unchanged in the interpreter-based HTTP backend. `frontend.lby` imports `shared`, runs the shared classification/scoring on sample titles, and renders results via the `console_log`/`dom_set_text` host builtins (compiles fully with `lullaby wasm frontend.lby`; interpreter `main` = 4). `index.html` is a self-contained host page (no CDN/remote assets, relative `frontend.wasm` load) that supplies the `env.console_log`/`env.dom_set_text` imports and calls the exported `main`. `backend.lby` is a bounded HTTP/1.1 server (mirroring `http_server/server.lby`) that imports `http` and `shared` and answers `/classify` with the shared label/score for a sample title. `http.lby` is the `http_server` framework module (copied so the flat `import http` resolves in this directory). `examples/valid/fullstack/README.md` documents building the frontend, opening the page locally, and running/curling the backend. - `examples/valid/http_server/`: a working HTTP/1.1 server written in pure Lullaby on the TCP socket builtins — `server.lby` (the bounded accept loop with the `tcp_read`/`tcp_write`/`tcp_shutdown`/`tcp_close` teardown, port and request count from `args()`), `http.lby` (the reusable `pub` "framework" module: request-line parsing, path routing, HTTP/1.1 response building, and a char-by-char `parse_i64`), and `examples/valid/http_server/README.md` (how to run and curl it). It lives in a subdirectory so the single-file example runner ignores it; the CLI round-trip test drives it as a real HTTP client.
- `examples/invalid/`: intentionally invalid `.lby` examples for inspecting diagnostics, including `int_float_mismatch.lby` (`L0307`), `unknown_function.lby` (`L0309`), `wrong_argument_count.lby` (`L0312`), `non_exhaustive_match.lby` (`L0384`), `break_outside_loop.lby` (`L0317`), and `index_non_array.lby` (`L0325`). ## Source Layout

The implementation is a Rust workspace. Unless changed by an explicit architecture decision, keep this layout:

- `crates/lullaby_lexer/`: source extension validation, tokenization, indentation scanning, string- and char-literal lexing (single-quoted `'c'` char literals, with malformed-literal diagnostic `L0105`), numeric literals with `_` digit separators (scanned into the number token; the parser validates placement), forbidden brace/semicolon diagnostics, current and planned keyword recognition (including the `import` and `pub` module keywords), and lexical tests.
- `crates/lullaby_diagnostics/`: shared diagnostic data structures, serializable span/traceback metadata, registry metadata, concise/verbose renderers, and deterministic JSON rendering.
- `crates/lullaby_lsp/`: additive Language Server Protocol (LSP) server for `.lby` files, standard-library + `serde_json` only (no third-party LSP/async framework). `src/transport.rs` implements JSON-RPC 2.0 over stdio with `Content-Length: N\r\n\r\n<json>` framing and the read/write loop (`serve`/`run_stdio`); `src/lib.rs` holds the pure `handle_message(state, method, id, params) -> Vec<Message>` request core (so the protocol is testable without stdio), the `ServerState` in-memory document map, `initialize` (advertising `textDocumentSync=1` full, `documentFormattingProvider=true`, `hoverProvider=true`, `definitionProvider=true`), `initialized`/`shutdown`/`exit`, and full-text `didOpen`/`didChange`/`didClose`; `src/diagnostics.rs` runs the existing lex -> parse -> `validate` pipeline over a document and converts each Lullaby diagnostic (code/message/1-based span) into an LSP `Diagnostic` with a 0-based `Range` (widened to the token at the span, else one char) for `textDocument/publishDiagnostics`, and formats via `format_program` for `textDocument/formatting` (one full-document `TextEdit`, or no edits when the document does not parse); `src/analysis.rs` resolves the identifier under the cursor for `textDocument/hover` (function signature, local/parameter type from the checker's `expression_types`, struct/enum declaration, or builtin description) and `textDocument/definition` (a `Location` for the declaration or `let`/parameter binding), reusing the parser plus `validate` metadata with no new semantics accessor. Unit tests drive `handle_message` directly for the initialize/diagnostics/formatting/hover/definition cases and the framed stdio loop end to end. Run over stdio with `lullaby lsp`. Design: [lsp_design.md](lsp_design.md).
- `crates/lullaby_parser/`: AST model and parser for function declarations, typed parameters, return types, indentation blocks, explicit and inferred `let`, assignment, `return`, `break`, `continue`, if/elif/else, while/loop/range-for, calls, integer and `f64` float literals (with optional `_` digit separators validated and stripped via `normalize_number_literal`), `'c'` char literals, array literals/indexing, struct declarations/construction/field access, enum (tagged-union) declarations with positional payload types, `match` expressions over enums with variant/wildcard patterns and payload bindings, variables, arithmetic, comparison, logical expressions, leading `import NAME` module lines and an optional `pub` export modifier on top-level `fn`/`struct`/`enum`/`alias` declarations, an optional `<T, U>` or bounded `<T: Trait + Other>` generic type-parameter list after a function name with duplicate/built-in-shadowing rejection (`L0394`), `trait NAME` declarations (indented `fn method self ... -> Ret` signatures) and `impl Trait for Type` blocks (indented `fn method self ... -> Ret` bodies, with the untyped `self` receiver injected as the implementing type), reported as `L0216` when malformed (`imports`/`is_public`/`type_params`/`traits`/`impls` are serde-defaulted, and a `TypeParam` serializes as the bare name string when unbounded, so single-file artifacts and AST snapshots stay valid), and `L0211` rejection for planned syntax keywords that are not in Alpha 1. The parser also accepts an `async` modifier on `fn` declarations, the `Future<T>` type, and `await EXPR` expressions. Also hosts `format.rs`, the canonical source formatter (`format_program`) that renders an AST back to idempotent, source-ordered `.lby` source (including `trait`/`impl` blocks and `<T: Bound>` lists) and backs `lullaby fmt`, with per-construct idempotency/round-trip regression tests over representative valid fixtures.
- `crates/lullaby_parser/tests/`: parser integration tests, including checked-in AST golden snapshots under `crates/lullaby_parser/tests/snapshots/`. Refresh snapshots intentionally with `$env:LULLABY_UPDATE_PARSER_SNAPSHOTS='1'; cargo test -p lullaby_parser --test ast_snapshots; Remove-Item Env:LULLABY_UPDATE_PARSER_SNAPSHOTS`.
- `crates/lullaby_semantics/`: static validation for duplicate declarations, explicit and inferred local binding types, assignment targets/types, function call arity/types, return behavior, bool conditions, loop-control placement, arithmetic/comparison/logical expression operand types, homogeneous non-empty arrays, array indexing, nominal struct types with positional construction and field access, nominal enum (tagged-union) types with globally unique variant names and unit/payload construction, interim pointer-style memory builtins, reference-counted `rc<T>`/`ref<T>`/`ptr<T>` types with `rc_new`/`rc_clone`/`rc_release`/`rc_get`/`rc_borrow`/`ref_get` builtins and `unsafe`-gated raw `ptr_read`/`ptr_write`, text file I/O builtins, standard stream builtins (`print`/`println`/`warn`/`flush`), the `wasm_log(x i64) -> void` host-log builtin (prints the value on the interpreters; lowers to the imported `env.log_i64` on the WASM backend), the JS/DOM host-interop builtins `console_log(s string) -> void` and `dom_set_text(id string, text string) -> void` (print the string / `id=text` on the interpreters; lower to the imported `env.console_log`/`env.dom_set_text` on the WASM backend), `i64`/`f64` numeric arithmetic, `to_string` conversion and `+` string concatenation, safe system command builtins, user-defined generic functions with call-site type inference (structural unification of arguments against type-variable-containing parameter types, substitution into the return type, conflicting inference `L0395`, uninferable-return-only type variable `L0396`; a bare `T` supports only universal operations, so arithmetic/ordering on it is rejected), traits and impls (a trait table plus an impl table built from `trait`/`impl` declarations, `impl` conformance checking with `Self` substitution `L0398`, duplicate-impl `L0399`, trait-method resolution that dispatches a UFCS call on the receiver type, and trait-bound checking at bounded-generic call sites and on bounded receivers `L0400`, with free-function and trait-method namespaces kept disjoint), `async fn`/`await` typing (an `async fn ... -> T` produces `Future<T>`; `await e` requires `e: Future<T>` and yields `T`; misuse is `L0344`), `export fn` C-callable-export validation (the first increment requires an all-`i64`, non-generic signature; anything else is `L0424`), and executable zero-argument `main` validation for compile/run. Successful validation returns `CheckedProgram` with function signatures (including generic `type_params` and per-parameter trait bounds) and inferred expression-type metadata; the generic unifier (`unify_param`/`substitute_type`/`infer_generic_return`) is public so the IR lowerer reuses it.

Construction (`some(x)`/`none`/`ok(x)`/`err(e)` and user `Variant(payload…)`, all emitted by the IR lowerer as positional `Call`s) `__alloc`s the record, stores the tag at offset 0, and stores each payload; `match` loads the tag (`i32.load` at offset 0) and dispatches with a chain of `i32.eq` + typed `if`/`else` blocks (a `Wildcard` arm becomes the final `else`; with exhaustiveness guaranteed by semantics the last variant arm is emitted unconditionally so a value match always leaves a value), binding each arm's payload slots into locals before its body.
