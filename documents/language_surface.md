# Earliest Installable Language Surface

Canonical language rules: see [core_language_rules.md](core_language_rules.md).

This document records the earliest installable language surface, preserved for reference; the current language is described in [language_specification.md](language_specification.md). The implemented parser grammar is drafted in [formal_grammar.md](formal_grammar.md). A feature listed here was part of that earliest installable surface.

## Source And Blocks

- Source files use the `.lby` extension. It is the only accepted source extension; the original `.lullaby` extension has been retired.
- Scope is indentation-only.
- Curly braces are rejected as block delimiters.
- Semicolons are rejected as statement terminators.
- Comments begin with `#` and continue to the end of the line.

## Declarations

- Functions use `fn name param Type -> ReturnType`.
- Function parameters require explicit types.
- Non-void functions return the last reachable expression unless `return expression` exits earlier.
- Void functions use `-> void` and may use bare `return`.
- Executable source passed to `lullaby compile` or source `lullaby run` must define `fn main -> Type` with zero parameters. `lullaby check` can still validate helper/library-style functions that do not define `main`.
- Local bindings use `let name Type = expression` for explicit annotations or `let name = expression` when the initializer type is unambiguous.
- Existing local bindings can be updated with `=`, `+=`, `-=`, `*=`, and `/=` when the types are valid.

## Types

- Implemented scalar types: `i64`, `f64`, `bool`, `string`, `char`, `byte`, and `void`.
- Float literals contain a decimal point (e.g. `3.14`, `2.0`) and have type `f64`. `i64` and `f64` do not mix implicitly; combining them is a type error.
- Numeric literals may use `_` as a digit separator between digits for readability (`1_000_000`, `3.141_592`); separators are purely cosmetic and do not change the value. A leading, trailing, doubled, or `.`-adjacent underscore is a malformed-literal parse error. The canonical formatter (`lullaby fmt`) renders literals without separators.
- Integer literals may also be written in hexadecimal (`0x`/`0X`, e.g. `0xFF`, `0XdeadBEEF`), binary (`0b`/`0B`, e.g. `0b1010`), or octal (`0o`/`0O`, e.g. `0o17`) and always have type `i64`. `_` digit separators are allowed strictly between two valid radix digits (`0xFF_FF`, `0b1010_0101`); a leading, trailing, doubled, or prefix-adjacent underscore, empty digits after the prefix (`0x`), an out-of-radix digit (`0xG`, `0b2`, `0o8`), a `.` (base-prefixed literals are integer-only), or an `i64` overflow is a malformed-literal parse error. Negative values are written with a leading `-` (`-0xFF`).
- `char` is a single Unicode scalar, written with single quotes (e.g. `'a'`, `'Z'`). A char literal must hold exactly one character; an empty (`''`), multi-character (`'ab'`), or unterminated literal is a lexer error (`L0105`). Chars compare with `==`/`!=` and order with `<`/`<=`/`>`/`>=` by code point. `char_code(c)` returns the Unicode scalar value as `i64`, and `char_from(i)` returns the `char` for a scalar value (a runtime error if `i` is not a valid Unicode scalar).
- `byte` is an 8-bit unsigned integer (0–255). There is no byte literal; construct one with `byte(i)` from an `i64` (a runtime error outside 0–255) and read it back with `byte_val(b)` as `i64`. Bytes compare with `==`/`!=` and order with `<`/`<=`/`>`/`>=` numerically. No byte arithmetic is implemented in this surface — conversions and comparisons only. Wrong argument types or arities to `char_code`/`char_from`/`byte`/`byte_val` report `L0389`. Other sized integer types (`i32`, `u32`, etc.) remain planned.
- Implemented array spelling: `array<T>`. Arrays are created either as a literal (`[1, 2, 3]`, at least one element) or, for a runtime length, with the builtin `array_fill(n i64, value T) -> array<T>` — a new array of length `n` with every element `value`. `array_fill(n, 0)` is the natural runtime-sized zeroed buffer for DP tables, matrices, and BFS/visited sets, replacing the old fixed-capacity-literal workaround; a negative length is the runtime error `L0433`. The array is indexable and mutable like any other (`a[i]`, `a[i] = v`, `len(a)`). Because a fixed array's length is inferred at compile time for the native/WASM backends, a function that uses `array_fill` (runtime length) is **deferred to the interpreters** and skips native/WASM codegen, like other heap-shaped features.
- Structs: `struct NAME` followed by indented `field type` lines declares a nominal record type (top level only). Construct positionally with call spelling — `Point(3, 4)` — read fields with `.` — `p.x` — and mutate fields with assignment — `p.x = 5`, `p.y += 1`, including nested `a.b.c = e`. Invalid declarations report `L0370`, bad field access `L0371`, and construction mismatches `L0372`; a field assignment with a wrong-type value reports `L0314`. Structs work across the AST, IR, and bytecode backends. Fields can also be set by name in any order — `Point(y: 4, x: 3)` — where every declared field must be supplied exactly once; duplicate, missing, or unknown named fields report `L0372`.
- Enums (tagged unions): `enum NAME` followed by indented `Variant type...` lines declares a nominal tagged-union type (top level only). Each variant is a name plus zero or more positional, unnamed payload types (e.g. `Circle f64`, `Rect f64 f64`, `Empty`). Construct with the existing spelling — a payload variant reads like a call (`Circle(2.0)`, `Rect(3.0, 4.0)`) and a unit variant reads like a bare name (`Empty`) — resolved semantically with no new syntax. Variant names are globally unique across all enums. Invalid declarations (duplicate enum, duplicate variant, or empty enum) report `L0380`, construction arity/payload-type mismatches report `L0381`, and a variant name colliding across enums reports `L0382`. Enums construct and pass identically across the AST, IR, and bytecode backends (including under optimization). Enum methods and user-defined generics over enums remain planned.
- `option<T>` and `result<T, E>`: two built-in generic enums for absence and fallible results. `option<T>` has variants `some(value)` and `none`; `result<T, E>` has variants `ok(value)` and `err(error)`. The variant names `some`/`none`/`ok`/`err` are reserved globally (a user enum declaring one reports `L0382`), and `option`/`result` may not be redeclared as user enums (`L0380`). Type spelling reuses the generic form with the canonical `", "`-joined argument list — `option<i64>`, `result<i64, string>`, `option<array<i64>>`. Construction is context-directed: `some(v)` synthesizes `option<typeof v>`, while `none`, `ok(v)`, and `err(v)` require an expected `option`/`result` type from a `let` annotation or a function return type; missing that context reports `L0386`, and a payload whose type disagrees reports `L0303`. `match` over an `option<U>` binds `some(x): x=U` and covers `some`+`none` (or `_`); over a `result<T, E>` it binds `ok(x): x=T` and `err(x): x=E` and covers `ok`+`err` (or `_`), reusing `L0384`/`L0385`. `option`/`result` construct and match identically across the AST, IR, and bytecode backends (including under optimization). Construction context now also includes call-argument position — a nested `none`/`ok`/`err` (or `list_new`/`map_new`) infers from the callee's parameter type (`describe(ok(5))`, `push(list_new(), byte(65))`); `?`-style propagation remains planned.
- Array literals must be non-empty and homogeneous, such as `[1, 2, 3]`.
- Array indexing is bounds-checked at runtime and requires an `i64` index.
- Array elements are mutable through assignment — `xs[i] = e`, `xs[i] += 1`, and mixed struct/array targets such as `cells[0].value = e`. The root variable is what optimizers track, and out-of-bounds writes report `L0413`.
- `list<T>`: a growable, value-semantic list spelled `list<T>` (e.g. `list<i64>`), backed by the same runtime array representation. Its builtins are functional — a "mutating" call returns a new list, so the idiom is `l = push(l, x)`. `list_new()` builds an empty list and takes its element type from the expected `list<...>` type (a `let` annotation or function return type); with no such context it reports `L0387`. `push(l, x)` appends, `get(l, i)` reads (bounds-checked, `L0413`), `set(l, i, x)` replaces at an index (bounds-checked), `pop(l)` drops the last element (empty pop errors), and `len(l)` returns the `i64` length. A wrong argument type or arity to these builtins reports `L0387`. `list<T>` behaves identically across the AST, IR, and bytecode backends (including under optimization).
- `map<K, V>`: an insertion-ordered, value-semantic hash map spelled `map<K, V>` (e.g. `map<string, i64>`). Keys are restricted to `i64` or `string`. Its builtins are functional — a "mutating" call returns a new map, so the idiom is `m = map_set(m, k, v)`. `map_new()` builds an empty map and takes its key/value types from the expected `map<...>` type (a `let` annotation or function return type); with no such context, an unsupported key type, or a wrong argument/arity it reports `L0388`. `map_get(m, k)` returns `option<V>` — `some(v)` if present, else `none` — so reads are unwrapped with `match`. `map<K, V>` behaves identically across the AST, IR, and bytecode backends (including under optimization).
- First-class functions: a function type is spelled `fn(T1, T2) -> R` (zero or more parameter types, an arrow, a return type — e.g. `fn(i64) -> i64`, `fn() -> void`, `fn(array<i64>) -> i64`), stored in the canonical `fn(...) -> R` form. A bare top-level function name used in value position is a function value of that type, so a function can be stored in a `let`, passed as an argument, and returned. Calling a function-typed local (`f(v)`) dispatches through the held value. Calling a non-function local, or passing a function value whose signature does not match the expected function type, reports `L0390` (a plain signature mismatch may also surface as an argument type error `L0313`). First-class functions run identically across the AST, IR, and bytecode backends (including under optimization). Capturing closures / lambda literals remain planned.
- Interim pointer type names use concrete spellings such as `ptr_i64`.
- Omitted local binding annotations are inferred from the initializer expression. Empty arrays and `void` initializers cannot supply an inferred local type.

## Expressions

- Implemented expressions include literals, variables, function calls with parentheses, grouped expressions, array literals, array indexing, arithmetic, bitwise, equality, ordering, and logical operators.
- Function values: a bare top-level function name is a value of type `fn(params) -> ret` (name-as-value), and a call whose name is a function-typed local calls through that value (call-through-local). No new syntax — both reuse the existing name and call spellings.
- Method-call sugar: `receiver.name(args)` desugars to `name(receiver, args)` (UFCS) at parse time, so any function whose first parameter matches the receiver can be called in method position — e.g. `p.norm2()` calls `fn norm2 p Point -> i64`. Plain `receiver.name` (no parentheses) remains struct field access.
- Arithmetic `+`, `-`, `*`, and `/` operate on two `i64`s or two `f64`s (operands must match); `+` also concatenates two `string`s. Integer division by zero is a runtime error; `f64` division follows IEEE 754 (division by zero yields infinity/NaN).
- Unary minus `-x`: arithmetic negation on any numeric operand (`i64`, `f64`, `f32`, or a fixed-width integer), preserving the operand's type. It is a real negation, not the `0 - x` desugar, so it negates floats correctly (an IEEE-754 sign-bit flip) and needs no matching-zero literal. Integer negation wraps (`-i64::MIN == i64::MIN`). All five backends agree; a non-numeric operand is `L0307`.
- Integer remainder `%`: `a % b` on two operands of the same integer type (`i64` or a fixed-width kind), yielding that type. It is the truncated remainder — the sign follows the dividend, so `-17 % 5 == -2`, exactly matching C and Rust and the `a - (a/b)*b` identity. `%` is **not** defined on floats (a float operand reuses the arithmetic operand-type diagnostic `L0307`). Remainder by zero is a runtime error (`L0404`), like integer division. `%` binds at the multiplicative precedence level (same as `* /`). The compound form `%=` applies it in place on an `i64` local. Remainder evaluates identically across the AST, IR, bytecode, native, and WASM backends (verified by cross-backend parity tests).
- Bitwise operators: binary `&` (and), `|` (or), `^` (xor), `<<` (shift left), `>>` (arithmetic/sign-preserving shift right), and unary `~` (one's complement / bitwise not). They apply to `i64` and to the fixed-width integers (`i8`–`u64`, `isize`/`usize`); both operands of `& | ^ << >>` (and the operand of `~`) must share the same integer type, and the result is that type (shifts mask the count to the type width; `>>` is logical for unsigned kinds and arithmetic for signed). A non-integer or mixed-width operand reuses the arithmetic operand-type diagnostic (`L0307`). `&`/`|`/`^` are the bitwise operators — the logical connectives are the `and`/`or`/`not` keywords, not `&&`/`||`.
  - **Shift-amount masking:** the shift amount is masked to its low 6 bits (`amount & 63`, i.e. taken mod 64), matching x86/Java `long` semantics. This makes `<<`/`>>` total and deterministic on every backend — a large or negative shift amount never raises a runtime error or panics (e.g. `1 << 65` is `1 << 1 = 2`, and `1 << -63` is `1 << 1 = 2`). `>>` is an arithmetic shift, so it preserves the sign of a negative `i64` (`-8 >> 1 == -4`).
  - **Precedence** (tightest to loosest, C-like among the new operators): unary `~` binds like `not`; then multiplicative `* /`, then additive `+ -`, then the shifts `<< >>`, then `&`, then `^`, then `|`, then the comparison/equality operators, then `and`, then `or`. So shifts bind just below additive (`a + b << c` is `(a + b) << c`), and the three bitwise-logical operators bind tighter than comparison (`a & b == c` is `(a & b) == c`). Among themselves `<<`/`>>` bind tighter than `&`, which binds tighter than `^`, which binds tighter than `|` (`a | b ^ c & d << e` is `a | (b ^ (c & (d << e)))`). The bitwise operators evaluate identically across all five backends — the AST, IR, and bytecode interpreters (including under constant-folding optimization), the WASM backend (`i64.and`/`or`/`xor`/`shl`/`shr_s`), and the scalar-subset native backend (`and`/`or`/`xor rax`, `shl`/`sar` with the count masked to 63). The native backend additionally auto-vectorizes bitwise `& | ^` reductions and element-wise maps over `array<i64>` into SSE2 packed loops (`pand`/`por`/`pxor`).
- Equality requires matching operand types.
- Ordering comparisons `<`, `<=`, `>`, and `>=` require two `i64`s or two `f64`s.
- Logical operators are `and`, `or`, and `not`; `and` and `or` short-circuit.
- Postfix error-propagation operator `EXPR?`: on a `result<T, E>` operand it evaluates to `T` for `ok(t)` and otherwise immediately returns `err(e)` from the enclosing function; on an `option<T>` operand it evaluates to `T` for `some(t)` and otherwise returns `none`. It binds tighter than every binary operator (so `a + b?` is `a + (b?)`, `f()?` applies to the call), and chains/nests (`x??`, `f(g()?)?`). The enclosing function's return type must be a compatible propagation target: a `result<T, E>` operand requires the function to return `result<U, E>` — the SAME error type `E` (a wrong `E` is `L0429`); an `option<T>` operand requires an `option<U>` return. A return type that is not a matching `result`/`option` is `L0427`, and `?` on a non-`option`/`result` value is `L0428`. `?` lives only in the parser AST: the AST interpreter realizes it via a function-level early-return signal, and the IR lowerer desugars it into `let`/`match`/`return`, so no new IR node exists and it runs identically on the AST, IR, and bytecode backends (including under optimization). Functions that use `?` (like any `match` over `option`/`result`) demote to the interpreters under the native/WASM backends.

## Control Flow

- Implemented branches: `if`, `elif`, and `else`.
- Implemented loops: `while`, inclusive range `for name from start to end` (optional `by step`), element iteration `for name in collection`, and `loop`.
- Element iteration `for x in collection`: iterates `x` over each element of an `array<T>` or `list<T>` (binding `x: T`), or each character of a `string` (binding `x: char`). A non-iterable collection is `L0434`. It desugars to an index-based range `for` (over `0..len-1`) in the IR, with the collection evaluated once, so it runs identically on every backend — including native codegen when the collection is a bare local/parameter (a computed collection binds to a hidden local and, being heap-shaped, defers to the interpreters). This replaces the `for i from 0 to len(xs) - 1` + `xs[i]` idiom.
- Implemented loop controls: `break` and `continue`.
- Loop control outside a loop is a semantic error.
- Pattern matching: `match SCRUTINEE` selects on an enum value with an indented arm list. Each arm is a pattern (`Variant(bind1, bind2, ...)` binding positional payloads, a bare `Variant` for a unit variant, or `_` wildcard), then `->`, then an inline expression or an indented block whose last expression is the arm's value. Like `if`/`try`, a `match` is an expression: when every arm yields the same type it produces that value (so it can be a function's final expression); otherwise it is a void statement. Matches must be exhaustive — every variant covered or a `_` present (`L0384`); a non-enum scrutinee reports `L0383`, and an unknown/duplicate variant arm or wrong binding arity reports `L0385`. Pattern matching runs identically on the AST, IR, and bytecode backends (including under optimization).

## Builtins

- Memory builtins: `alloc(value)`, `load(ptr)`, `store(ptr, value)`, and `dealloc(ptr)`.
- Text file builtins: `read_file(path)`, `write_file(path, content)`, `append_file(path, content)`, and `file_exists(path)`.
- Directory and binary file builtins: `read_lines(path) -> list<string>` (file split into lines, no trailing newline per element), `read_bytes(path) -> list<byte>`, `write_bytes(path, data list<byte>) -> void` (truncating), `file_size(path) -> i64`, `is_file(path) -> bool`, `is_dir(path) -> bool`, `list_dir(path) -> list<string>` (entry names only), `make_dir(path) -> void` (creates parents), `remove_file(path) -> void`, and `remove_dir(path) -> void` (empty directory only). These are one-shot operations; stateful file handles, seeking, and buffered streams are deferred. Wrong argument types or arities report `L0333`. A failed read/metadata query reports `L0414`; a failed write/create/remove reports `L0415`.
- System command builtins: `sys_status(program, args)` and `sys_output(program, args)`, where `args` is `array<string>`.
- System command builtins execute a program with an argv array directly and do not invoke a shell.
- Process/environment builtins: `env(name string)` reads an environment variable and returns `option<string>` — `some(value)` when set, `none` otherwise — so reads are unwrapped with `match`. `args()` returns the running program's CLI arguments as a `list<string>` (an empty list when none were passed); `lullaby run <file.lby> [args...]` passes trailing tokens after the source path as those program arguments. Wrong argument types or arities to `env`/`args` report `L0332`.
- Standard stream builtins: `print(text)` and `println(text)` write a `string` to stdout, `warn(text)` writes a `string` line to stderr, and `flush()` flushes stdout. Each returns `void`.
- Testing builtin: `assert(cond bool) -> void` raises a catchable runtime error with the message `assertion failed` (the same error path a user `throw` produces, so `try`/`catch` recovers it) when `cond` is false, and returns `void` when true. A non-`bool` argument reports `L0342`. This is the assertion primitive the `lullaby test` runner uses to turn `test_*` functions into pass/fail tests.
- String operations: `to_string(x)` converts an `i64`, `f64`, `bool`, `string`, `char`, or `byte` to a `string`; `+` concatenates when both operands are `string` (and still adds when both are `i64`). Mixed `string`/`i64` operands to `+` are a type error (`L0307`). This makes computed values printable, e.g. `println("answer: " + to_string(40 + 2))`.
- String ergonomics: `s += piece` concatenates in place (string `+=`, desugaring to `s = s + piece`); `s[i]` reads the `i`-th character as a `char` (bounds-checked at runtime, `L0413`); and two strings compare with `<`/`<=`/`>`/`>=` **lexicographically** by Unicode code point. These evaluate correctly on the AST, IR, and bytecode interpreters; because they operate on heap `string` values, a function that uses string `+=`, string indexing, or string ordering is **deferred to the interpreters** and skips native/WASM codegen (like the other index-based string ops).
- Char/byte conversions: `char_code(c char)` returns an `i64` Unicode scalar value and `char_from(i i64)` returns a `char` (runtime error on an invalid scalar); `byte(i i64)` returns a `byte` (runtime error outside 0–255) and `byte_val(b byte)` returns its `i64` value. Wrong argument types or arities report `L0389`.
- Collection length: `len(x)` returns the `i64` element count of an `array<T>` or `list<T>`, or the character count of a `string`; other argument types report `L0373`. Combine it with indexing for iteration, e.g. `for i from 0 to len(xs) - 1`.
- List library (functional, value-semantic): `list_new()` builds an empty `list<T>` (element type from the expected `list<...>` type; `L0387` otherwise), `push(l, x)` appends, `get(l, i)` reads (bounds-checked, `L0413`), `set(l, i, x)` replaces at an index, and `pop(l)` drops the last element. Each mutating call returns a new list, so the idiom is `l = push(l, x)`. Wrong argument types or arities report `L0387`.
- Map library (functional, value-semantic): `map_new()` builds an empty `map<K, V>` with `i64`/`string` keys (key/value types from the expected `map<...>` type; `L0388` otherwise), `map_set(m, k, v)` inserts or replaces, `map_get(m, k)` returns `option<V>` (`some(v)` or `none`), `map_has(m, k)` returns a `bool`, `map_len(m)` returns the `i64` entry count, and `map_del(m, k)` removes a key (no error if absent). Each mutating call returns a new map, so the idiom is `m = map_set(m, k, v)`. Wrong argument types, arities, or unsupported key types report `L0388`.
- String library (character-indexed): `substring(s, start, end)` slices the half-open `[start, end)` char range (out-of-range indices error at runtime); `find(s, needle)` returns the first char index or `-1`; `contains(s, needle)` returns a `bool`; `split(s, sep)` returns `array<string>` (empty `sep` errors); `join(parts, sep)` joins an `array<string>`; `trim(s)` removes leading/trailing ASCII whitespace; `replace(s, from, to)` replaces every occurrence (empty `from` errors); `upper(s)`/`lower(s)` change case. Wrong argument types report `L0375`.
- Math library: type-directed numeric builtins over `i64` and `f64`. `abs(x)` returns the magnitude (`i64->i64`, `f64->f64`); `min(a, b)` and `max(a, b)` pick the smaller/larger of two matching operands; `pow(base, exp)` raises to a power (integer form requires `exp >= 0`, a negative integer exponent is a runtime `L0417`); `sqrt(x f64)`, `floor(x f64)`, `ceil(x f64)`, and `round(x f64)` return `f64`. Wrong or mismatched operand types report `L0374`.
- Concurrency (data parallelism): `parallel_map(f fn(i64) -> i64, args list<i64>) -> list<i64>` evaluates `f(arg)` for every element of `args` concurrently on separate OS threads and returns the results in the **same order as `args`** (deterministic regardless of scheduling). Each thread builds a fresh interpreter over the shared program, so heaps are per-thread with no shared mutable state and no locking; `f` must be an ordinary top-level `fn(i64) -> i64`. Wrong arity, a non-`fn(i64) -> i64` first argument, or a non-`list<i64>` second argument report `L0334`. It runs identically on the AST, IR, and bytecode backends (including under optimization).
- Concurrency (threads/channels/mutex): `chan_new() -> Chan` creates an unbounded `i64` message-passing channel (a shared handle: cloning shares the same queue); `send(ch Chan, v i64) -> void` enqueues (never blocks), `recv(ch Chan) -> i64` dequeues blocking, and `try_recv(ch Chan) -> option<i64>` dequeues non-blocking (`some(v)`/`none`). `spawn(f fn(Chan, i64) -> void, ch Chan, v i64) -> Task` runs `f(ch, v)` on a detached OS thread that owns a share of the program (an `Arc<Program>`/`Arc<IrModule>` clone) and returns a `Task`; `task_join(t Task) -> void` waits for it (a second `task_join` is a no-op). It is named `task_join`, not `join`, because `join` is the string-list joiner. `mutex_new(v i64) -> Mutex` creates a shared mutex over one `i64` (shared on clone); `mutex_get`/`mutex_set`/`mutex_add` read, write, and atomically add-and-return. Wrong arity or a wrong-typed argument (a non-`Chan`/`Task`/`Mutex` handle, a non-`i64` value, or a `spawn` first argument that is not a `fn(Chan, i64) -> void`) reports `L0337`. These run identically on the AST, IR, and bytecode backends.
- Concurrency (`async`/`await`): `async fn NAME ... -> T` declares an asynchronous function; calling it spawns the body on an OS thread and returns a `Future<T>` immediately (so multiple `async` calls run concurrently), and `await e` (where `e: Future<T>`) blocks for the result and yields `T`. Results are deterministic though scheduling is not; awaiting a non-`Future` or an unresolvable future type reports `L0344`. It runs identically on the AST, IR, and bytecode backends (the native and WASM scalar backends skip `Future`/`await`). Generic channels, `select`, a cooperative single-thread executor, passing a `Mutex`/`Socket` into a worker, and cross-thread socket sharing remain deferred (see [concurrency_design.md](concurrency_design.md)).
- Networking (TCP/UDP sockets): a `Socket` is an opaque handle to an OS network resource (a listener, stream, or UDP socket), surfaced as an integer handle backed by a per-runtime table (like a heap pointer). All fallible socket builtins return a `result<T, string>`, so network failures are ordinary runtime values matched with `match` (an `err(message string)` on the error path), never panics. TCP: `tcp_connect(host string, port i64) -> result<Socket, string>` opens a client stream; `tcp_listen(host string, port i64) -> result<Socket, string>` binds a listener; `tcp_accept(listener Socket) -> result<Socket, string>` blocks for a connection and yields a stream `Socket`; `tcp_read(conn Socket) -> result<string, string>` reads up to 4096 bytes as a UTF-8 string (empty string on clean EOF); `tcp_write(conn Socket, data string) -> result<i64, string>` writes the string's full byte buffer (retrying short writes), flushes, and returns the count; `tcp_shutdown(conn Socket) -> void` gracefully shuts down the write half of the connection (signalling EOF) so a buffered response is delivered before the socket is dropped; `tcp_close(conn Socket) -> void` frees the handle. The graceful server teardown order is `tcp_write` then `tcp_shutdown` then `tcp_close`. UDP: `udp_bind(host string, port i64) -> result<Socket, string>` binds a datagram socket; `udp_send_to(sock Socket, data string, host string, port i64) -> result<i64, string>` sends one datagram and returns the byte count; `udp_recv(sock Socket) -> result<string, string>` receives one datagram as a string (the sender address is dropped in this increment). Wrong argument types or arities report `L0335`. Sockets run identically on the AST, IR, and bytecode backends. A full HTTP/1.1 server can be written in pure Lullaby on these builtins — request parsing, routing, and response building as ordinary `pub` functions — as shown by `examples/valid/http_server/`.
- HTTP/1.1 client (built over `TcpStream`, std only, no TLS): `http_get(url string) -> result<string, string>` performs an HTTP/1.1 GET and returns the response body on a 2xx/3xx response; `http_post(url string, body string) -> result<string, string>` performs an HTTP/1.1 POST with a `text/plain` request body (correct `Content-Length`) and returns the response body. Only the `http` scheme is supported — an HTTPS URL returns `err("https not supported")`; chunked transfer decoding is not implemented (responses are read to EOF via `Connection: close`); a 4xx/5xx status returns `err("http {code}: {first-body-line}")`; a 10-second read timeout surfaces a hung server as `err`. Connection/parse/HTTP failures are ordinary `err(message)` results; wrong argument types or arities report `L0336`. The HTTP client runs identically on the AST, IR, and bytecode backends.
- Reference types: `rc<T>` is a reference-counted shared owner, `ref<T>` is a non-null borrowed reference, and `ptr<T>` (legacy spelling `ptr_T`) is a raw pointer.
- Reference builtins: `rc_new(value)` creates an `rc<T>`; `rc_clone(rc<T>)` shares ownership; `rc_release(rc<T>)` drops one owner and frees at zero; `rc_get(rc<T>)`/`ref_get(ref<T>)` read the referent; `rc_borrow(rc<T>)` yields a `ref<T>`.
- `unsafe` block: an indented block introduced by `unsafe` in which raw-pointer operations are permitted. `ptr_read(ptr<T>)` and `ptr_write(ptr<T>, value)` require an `unsafe` context (`L0330` otherwise); `unsafe` is a transparent scope, so bindings inside it remain visible afterward.
- `asm` inline assembly (native-only): `asm b0, b1, ...` emits a comma-separated list of raw x86-64 machine-code byte literals (each an `i64` in `0..=255`) verbatim into the current function's `.text`. It is inherently unsafe and must appear inside an `unsafe` block (`L0330` otherwise); each byte is validated in range and the statement must be non-empty (`L0425` at check time). A trailing `asm` is trusted to leave the return value in `rax` (the Win64 epilogue returns it), so it satisfies a non-void function's final-value requirement — e.g. `asm 72, 199, 192, 42, 0, 0, 0` is `mov rax, 42`, making the function return `42`. Only the native backend can emit it; the AST/IR/bytecode interpreters reject any `asm` at runtime with `L0425`. There is no register/clobber modeling, no operand substitution, and no assembly-text parsing (bytes only).
- Region declarations: `region NAME: size=N[, align=N][, kind=static|dynamic][, mutable=true|false]` declares a named memory region. Size must be positive, alignment (if present) must be a power of two, and kind must be `static` or `dynamic` (`L0340`); region names must be unique within a function (`L0341`). Regions are compile-time metadata in this surface (surfaced as `RegionCreate` in memory analysis) with no runtime allocation yet.
- Lifetime analysis: a conservative compile-time pass rejects straight-line use-after-free and double-free of resources freed by `dealloc`/`rc_release` (`L0350`), and rejects returning a borrowed `ref<T>` from a function (`L0351`). Freeing inside a branch is not tracked out of that branch; the runtime `L0406` guard remains as defense-in-depth. Deterministic per-block cleanup ordering is provided by `lullaby_ir::frame_layout`.
- Structured error handling: `throw EXPR` raises a `string` error value; `try` / `catch NAME` runs a protected block and, if it throws, binds the error message to `NAME` (a `string`) and runs the handler for recovery. Like `if`/`else`, a `try`/`catch` is an expression: when both arms yield the same type it produces that value (so it can be a function's final expression); otherwise it is a void statement. Uncaught throws surface as runtime `L0420`. Only user-thrown errors are caught; a `try` without `catch` reports `L0213`. Structured error handling runs identically on the AST, IR, and bytecode backends (including under optimization).
- Type aliases: `alias NAME = TYPE` declares a top-level type alias (e.g. `alias Count = i64`, `alias Numbers = array<i64>`). Aliases resolve structurally to their canonical target before type checking, so an alias and its target are interchangeable, and aliases carry no runtime representation. Aliases inside generic arguments (`array<Count>`) resolve too. Duplicate aliases report `L0360` and cyclic aliases report `L0361`. Struct/record and map generics remain planned until those types exist.
- Modules, imports, and visibility: each `.lby` file is a module. A top-level `fn`/`struct`/`enum`/`alias` is file-private unless prefixed with `pub`, which exports it. A file pulls in another module with one or more leading `import NAME` lines (before any declaration); `import NAME` loads `NAME.lby` from the **entry file's directory** and makes that module's `pub` items available **unqualified**. A frontend-only loader lexes, parses, resolves imports (recursively), and merges every module into one flat program before semantic analysis, so the type checker and all three backends run unchanged. The merged namespace is flat with **no shadowing**: a name declared in more than one module reports `L0391`; referencing a name that is private in another module (or a `pub` name from a module you did not `import`) reports `L0392`; an import cycle reports `L0393`; and a missing module file reports `L0397`. Builtins, primitive types, and enum variants are never subject to visibility. Dotted module paths, qualified access (`Module.item`), per-field visibility, and re-exports are deferred (see [modules_design.md](modules_design.md)).
- Generic functions: a `fn` may declare type parameters in angle brackets after its name (`fn identity<T> x T -> T`, `fn wrap<T> x T -> option<T>`, `fn choose<T> pick bool a T b T -> T`) and use each name as an ordinary type in its parameters and return type, including inside built-in generics (`list<T>`, `option<T>`). At a call site the type parameters are inferred from the argument types by structural unification and substituted into the return type — `identity(41)` yields `i64`, `identity("hi")` yields `string`, `wrap(1)` yields `option<i64>`. A duplicate or built-in-shadowing type-parameter list reports `L0394`; two arguments that force a type parameter to two different types report `L0395`; a type parameter that only appears in the return type and cannot be inferred reports `L0396`. Generics are **erased** at runtime (a `T` is just a value), so the AST, IR, and bytecode backends run unchanged. A bare `T` value supports only universal operations (binding, passing, returning, `==`/`!=` between two same-`T` values, and use as a payload to a built-in generic like `some(x)`); arithmetic and ordering on a bare `T` are rejected until trait bounds exist (see [generics_design.md](generics_design.md)).
- Traits: `trait NAME` followed by indented method signatures `fn method self [param Type ...] -> Ret` (no body) names a set of methods a type can implement. The receiver is `self`; `Self` in a later parameter/return type means the implementing type. `impl Trait for Type` provides a body for every method the trait requires (`fn method self ... -> Ret`, first parameter `self`). A trait method is called with the existing UFCS spelling — `p.show()` desugars to `show(p)` and dispatches on the receiver's concrete runtime type. A generic function may bound a type parameter with `<T: Trait>` (or `<T: A + B>`); inside the function a value of type `T` may call the bound trait's methods, and at a call site the inferred concrete type must implement every bound. An `impl` that does not match its trait (missing/extra/mismatched method, or a trait-method/free-function name clash) reports `L0398`, a duplicate `impl Trait for Type` reports `L0399`, and a type used where a bound requires a trait it does not implement reports `L0400`. Dispatch uses one shared impl table consulted identically by the AST, IR, and bytecode backends (including under optimization). Default method bodies, trait objects (`dyn`), associated types, and supertraits remain planned. See [traits_design.md](traits_design.md).

## CLI And Artifacts

- Development commands are available through `cargo run -p lullaby_cli -- ...`.
- The release package exposes `bin\lullaby.exe`.
- Supported commands:
  - `lullaby check [--verbose|--format json] <file.lby | project-dir | lullaby.json>`
  - `lullaby compile [--optimize none|constant-fold|dead-code|full] [-o output.lbc] [--verbose|--format json] <file.lby | project-dir | lullaby.json>`
  - `lullaby build [--optimize none|constant-fold|dead-code|full] [-o output.lbc] [--verbose|--format json] <file.lby | project-dir | lullaby.json>`
  - `lullaby inspect [--verbose|--format json] <file.lbc>`
  - `lullaby run [--backend ast|ir|bytecode] [--optimize none|constant-fold|dead-code|full] [--verbose|--format json] <file.lby | project-dir | lullaby.json> [args...]`
  - `lullaby run [--verbose|--format json] <file.lbc>`
  - `lullaby test [--verbose] <file.lby | project-dir | lullaby.json>`
  - `lullaby wasm [--verbose] [-o out.wasm] <file.lby | project-dir | lullaby.json>`
  - `lullaby native [--verbose] [-o out.exe] <file.lby | project-dir | lullaby.json>`
  - `lullaby fmt [--write|--check] <file.lby>`
  - `lullaby docs`
  - `lullaby examples`
  - `lullaby --version`
- `--diagnostic-format json` is accepted as a JSON diagnostics alias.
- **Projects (`lullaby.json`)**: `run`/`check`/`build`/`test`/`compile`/`wasm`/`native` accept either a single `.lby` file (unchanged) **or** a project — a directory containing a `lullaby.json` manifest, or the `lullaby.json` path itself. The manifest declares `name`, an optional `entry` (the executable `.lby`, relative to the manifest dir), `src` (source directories, default `["."]`), and `dependencies` (name → path to another local project root that contains its own `lullaby.json`; **local paths only**, remote fetching deferred). In a project build, `import NAME` resolves `NAME.lby` by searching the importing file's directory, then every `src` directory of the project, then every `src` directory of each transitively resolved dependency; all module rules (`L0391`/`L0392`/`L0393`) apply across the whole merged set. `run`/`build`/`compile`/`wasm`/`native` need an `entry`; `check`/`test` can validate a library project without one. A missing/malformed manifest, a missing `src` directory, a missing dependency or its missing `lullaby.json`, or a missing `entry` for a run/build target report `L0343`. See [modules_design.md](modules_design.md). Single-file behavior is unchanged.
- `lullaby test` runs a language-level test suite. It validates the source as a **library** (no `main` required), then discovers every top-level function whose name starts with `test_`, takes zero parameters, is non-generic, and returns `void`/`i64`/`bool`, and runs each through the AST interpreter. A test **passes** if it returns without a runtime error and **fails** if it produces one (an `assert(false)`, a `throw`, or any other runtime error). It prints one line per test (`PASS test_name` or `FAIL test_name: <message>`), a `N passed, M failed` summary, and a `skip` line for any `test_*` function it cannot run (parameters, generic, or an unsupported return type). It exits non-zero when any test fails, zero otherwise. `--verbose` adds a traceback under each failing test. `examples/valid/tests_demo/tests_demo.lby` is a runnable demo suite.
- `lullaby wasm` compiles the source's scalar functions to a binary `.wasm` module (default output `<stem>.wasm`). Only top-level functions whose parameter and return types are all scalars (`i64`/`f64`/`bool`/`char`/`byte`) are compiled and exported by name; any function that uses a non-scalar type, heap value, `match`, or a builtin is skipped and still runs on the interpreters. `--verbose` lists which functions compiled and which were skipped and why. When no function is eligible, it reports `L0338`. This is the first (scalar-subset) increment of the WebAssembly backend; string/struct/enum/collection support via linear memory is deferred.
- `lullaby native` emits a native x86-64 COFF object for the source's **i64-scalar** functions and, best-effort, links it into a runnable Windows console `.exe` (default output `<stem>.exe`, with the intermediate object at `<stem>.obj`). Only top-level functions whose parameters and return type are all `i64` (at most four parameters, passed Win64-style in `rcx`/`rdx`/`r8`/`r9`) are compiled; supported bodies are integer literals, params/`let` locals, `+ - * /` (signed `idiv`), comparisons, `and`/`or`/`not`, `if`/`elif`/`else`, `while`, `loop` with `break`/`continue`, range `for`, `return`, and calls to other compiled i64 functions. Any function that uses a non-i64 type, `f64`/`bool`/`char`/`byte`, a heap value, `match`, or a builtin is skipped and still runs on the interpreters. The emitted entry stub (`_lullaby_start`) calls `main`, moves its `i64` result into the exit-code register, and calls `ExitProcess` (imported from `kernel32`), so the process exit code is `main`'s result mod 256. `--verbose` lists compiled and skipped functions. When no i64-scalar function (including `main`) is eligible, it reports `L0339`. The COFF object is always written (the reliable floor); linking is attempted only when `rust-lld` and `kernel32.lib` are discoverable (the MSVC `LIB` environment variable, e.g. from a Developer Command Prompt). If the linker or import library is unavailable, the command reports the object it produced and explains that linking was unavailable rather than failing. This is the first (i64-scalar) increment of the native backend; `f64`/`bool`/`char`/`byte` scalars and heap/string/struct/enum/collection support are deferred. **C-ABI FFI:** a body-less `extern fn NAME params -> Ret` declaration (an `extern` modifier on a `fn` with no indented body, `i64` params/return in this increment) imports a C function; a call to it type-checks like any other call and, on the native backend, lowers to a `call` of the external C symbol (Win64 register args, result in `rax`) linked against the C runtime (`ucrt.lib`, discovered via `LIB` like `kernel32.lib`). The interpreters cannot execute C, so calling an `extern fn` on the AST/IR/bytecode backends reports `L0423` (`lullaby check` still validates the declaration and its call sites). The other FFI direction is also supported: an `export fn NAME params -> Ret` (an `export` modifier on a normal bodied `fn`, `i64` params/return in this increment or `L0424`) exposes a Lullaby function under its plain C name as an externally visible, defined `.text` symbol so C can call **into** Lullaby; `export` runs like an ordinary function on the interpreters and only affects native symbol visibility. An export-only program (no `main`) emits a C-callable library object with no entry stub. Non-scalar/pointer C types, callbacks, and non-Windows C ABIs are deferred. See [native_backend_contract.md](native_backend_contract.md).
- `lullaby compile`, `lullaby build`, and source `lullaby run` require a zero-argument `main` entry point before lowering or execution. Invalid executable entry points report `L0329`.
- `lullaby compile` and its artifact-generation alias `lullaby build` write a versioned `.lbc` instruction-bytecode artifact with a format marker, artifact version, metadata, entry point, function table, ordered memory operation metadata, compatibility checks, and bytecode module instructions. `lullaby inspect` prints artifact metadata, function signatures, and memory operation counts without executing the program.

## Diagnostics

- Diagnostics use stable `L####` codes documented in [diagnostic_registry.md](diagnostic_registry.md).
- Concise diagnostics are the default.
- `--verbose` adds source excerpts, caret markers, root-cause text, suggested fixes, related notes, and runtime traceback frames when available.
- `--format json` emits deterministic machine-readable diagnostics for tools, CI, editors, and LLM agents.

## Planned Beyond This Surface

The following are not part of this surface:

- Native code generation for the full language, linking, and machine-code binary output (a COFF object prototype exists for a small subset).
- Packages, unions, interfaces, and classes (single-directory file modules with flat unqualified `import` and `pub` visibility are implemented; dotted module paths, package roots, qualified access, per-field visibility, and re-exports remain planned — see [modules_design.md](modules_design.md)). Traits are implemented — `trait` declarations, `impl Trait for Type`, receiver-type method dispatch, and trait bounds on generics (`<T: Show>`); default method bodies, trait objects (`dyn`), associated types, and supertraits are deferred (see [traits_design.md](traits_design.md)). Structs and enums are implemented — structs including field mutation and positional/named-field construction, enums including unit/payload declaration and construction, `match` pattern matching over enums with payload binding, exhaustiveness, and `_` wildcard, and the built-in generic enums `option<T>`/`result<T, E>`; struct/enum methods (beyond trait impls) and user-defined generics over enums are deferred.
- User-defined generic *types* (`struct Box<T>`, `enum Tree<T>`) and **explicit type arguments** at a call site (`identity<i64>(3)`). User-defined generic *functions* with call-site type inference are implemented (see [generics_design.md](generics_design.md)), and **trait bounds** (`<T: Ord>`) are implemented (see [traits_design.md](traits_design.md)) — a bounded `T` may call its bound traits' methods. Argument-position inference for `option`/`result`/`list`/`map` construction is implemented — a nested constructor in a call argument infers from the callee's parameter type (see [option_result_design.md](option_result_design.md)).
- GC hooks and runtime region allocation (region *declarations* are analyzed as compile-time metadata; reference counting via `rc<T>` and conservative lifetime analysis are implemented).
- Binary I/O, memory mapping, async, sockets, IPC, and general syscall APIs (standard text streams and file/system builtins are implemented).
- Capturing closures and lambda literals (first-class *top-level* function values — name-as-value, storing in a `let`, passing, returning, and calling through a local — are implemented; nested function literals that capture surrounding locals remain planned).
- A full installer; this surface uses a Windows portable archive with optional user PATH helper scripts.

When planned syntax keywords such as `module`, `package`, `switch`, or a bare `catch` appear as source syntax, the parser reports `L0211` instead of accepting a partial or ambiguous construct. (`import` and `pub` are now accepted module syntax.)
